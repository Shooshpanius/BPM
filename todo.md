# Функциональные требования: BPM-система

**Стек**: Backend — C# ASP.NET Core, Frontend — React  
**Архитектура**: REST API + SignalR для real-time, модульная монолитная архитектура (возможен переход на микросервисы)

---

## 1. Аутентификация и авторизация

### FR-AUTH-01: Вход в систему

#### FR-AUTH-01.1: Вход по логину и паролю (JWT-токены, refresh token)
- [ ] Форма входа: поля «логин» (email или username) и «пароль», кнопка «Войти»
- [ ] Клиентская валидация полей (не пусто, минимальная длина, формат email)
- [ ] `POST /api/auth/login` — возвращает access token (JWT) + refresh token
- [ ] Access token: короткое время жизни (настраивается, по умолчанию 15–60 мин), хранится в памяти / sessionStorage на фронтенде
- [ ] Refresh token: длинное время жизни (настраивается, по умолчанию 7–30 дней), передаётся в HttpOnly-cookie
- [ ] `POST /api/auth/refresh` — обновление access token по действующему refresh token
- [ ] `POST /api/auth/logout` — инвалидация refresh token (занесение в чёрный список / удаление из БД)
- [ ] Политика паролей: минимальная длина, требования к сложности (буквы, цифры, спецсимволы), срок действия пароля — настраивается администратором
- [ ] Блокировка учётной записи после N неудачных попыток входа (N и время блокировки — настраиваются)
- [ ] Смена пароля пользователем: форма «старый пароль / новый пароль / подтверждение»
- [ ] Сброс пароля: отправка ссылки / одноразового кода на email, форма установки нового пароля, ограниченное время жизни ссылки
- [ ] Страница входа: отображение ошибок (неверный логин/пароль, аккаунт заблокирован, необходима смена пароля)
- [ ] Опция «Запомнить меня» (продление жизни refresh token до максимального значения)

#### FR-AUTH-01.2: Вход по сертификату (X.509)
- [ ] Административный интерфейс: загрузка и управление доверенными корневыми и промежуточными CA
- [ ] Настройка политики: обязательный или опциональный вход по сертификату для указанных ролей / всех пользователей
- [ ] Страница входа: кнопка «Войти по сертификату», выбор сертификата из хранилища браузера или загрузка PEM/PFX-файла
- [ ] Backend: проверка цепочки доверия, срока действия и статуса отзыва сертификата (OCSP / CRL)
- [ ] Сопоставление сертификата с учётной записью по Subject DN или Subject Alternative Name (email / UPN)
- [ ] Выдача JWT-пары (access + refresh) при успешной проверке сертификата
- [ ] Управление доверенными сертификатами в профиле пользователя: привязка нескольких сертификатов, просмотр, удаление

#### FR-AUTH-01.3: Вход по hardware-токену (OTP / FIDO2)
- [ ] Поддержка TOTP (RFC 6238 / Google Authenticator): генерация QR-кода для регистрации устройства, поле ввода одноразового кода на странице входа
- [ ] Поддержка FIDO2 / WebAuthn: регистрация аппаратного ключа (YubiKey и аналоги) через браузерный API, беспарольная аутентификация или в качестве второго фактора
- [ ] Двухфакторная аутентификация (2FA): вход в два шага — сначала логин/пароль, затем OTP или FIDO2-подтверждение
- [ ] Административные настройки: включение/отключение 2FA глобально или для отдельных ролей, обязательная 2FA
- [ ] Резервные коды (backup codes): генерация и выдача пользователю при регистрации устройства, однократное использование
- [ ] Управление токенами в профиле пользователя: список зарегистрированных устройств, добавление нового, удаление/отзыв

#### FR-AUTH-01.4: SSO / сквозная авторизация (SAML 2.0, OpenID Connect)
- [ ] Поддержка SAML 2.0: SP-Initiated и IdP-Initiated потоки, импорт метаданных IdP (URL или XML-файл), настройка сертификата подписи утверждений, маппинг атрибутов SAML → поля пользователя
- [ ] Поддержка OpenID Connect / OAuth 2.0: настройка клиента (client_id, client_secret, authorization/token/userinfo endpoints, scopes), маппинг claims → поля пользователя
- [ ] Поддержка нескольких провайдеров одновременно (мультитенантность / несколько IdP)
- [ ] Автоматическое создание учётной записи при первом SSO-входе (Just-In-Time provisioning) с применением дефолтной роли
- [ ] Синхронизация данных профиля при каждом SSO-входе (ФИО, email, роли из claims)
- [ ] Страница входа: отдельная кнопка / ссылка для каждого настроенного SSO-провайдера
- [ ] Административный интерфейс: CRUD-управление SSO-провайдерами, тестирование подключения
- [ ] Выход из системы с уведомлением IdP (Single Logout, SLO) — для SAML 2.0

#### FR-AUTH-01.5: Интеграция с LDAP / Active Directory
- [ ] Административный интерфейс: настройка подключения (хост, порт, протокол LDAP/LDAPS, Base DN, сервисная учётная запись и пароль), проверка соединения
- [ ] Режим аутентификации через LDAP: проверка логина и пароля напрямую в каталоге (bind-аутентификация)
- [ ] Режим импорта: синхронизация пользователей и групп/подразделений из каталога без аутентификации через LDAP
- [ ] Маппинг атрибутов LDAP → поля профиля системы (ФИО, email, телефон, должность, подразделение, фото)
- [ ] Маппинг групп LDAP → роли системы
- [ ] Ручной запуск синхронизации из административного интерфейса
- [ ] Автоматическая синхронизация по расписанию (настраиваемый cron-интервал)
- [ ] Журнал синхронизации: дата/время, количество обработанных/созданных/обновлённых/ошибочных записей
- [ ] Обработка удалённых / заблокированных пользователей в AD: автоматическая деактивация учётной записи в системе
- [ ] Поддержка вложенных групп (nested groups) Active Directory

### FR-AUTH-02: Управление сессиями

#### FR-AUTH-02.1: Активные сессии пользователя
- [ ] Страница «Мои активные сессии» в профиле пользователя: таблица со столбцами — устройство/браузер (User-Agent), IP-адрес, геолокация (город, страна по IP), дата и время последней активности, статус (текущая / другая)
- [ ] `GET /api/auth/sessions` — получение списка активных сессий текущего пользователя
- [ ] Выделение текущей сессии (пометка «эта сессия»)
- [ ] `DELETE /api/auth/sessions/{sessionId}` — завершение конкретной сессии: инвалидация соответствующего refresh token
- [ ] `DELETE /api/auth/sessions` — завершение всех остальных сессий, кроме текущей («Выйти со всех устройств»)
- [ ] Автоматическое завершение сессии по истечении времени неактивности (настраиваемый idle timeout в системных настройках)
- [ ] Административный интерфейс: просмотр активных сессий любого пользователя, принудительное завершение сессии администратором
- [ ] `GET /api/admin/users/{userId}/sessions` + `DELETE /api/admin/users/{userId}/sessions/{sessionId}` — административные эндпоинты
- [ ] Уведомление пользователя (in-app / email) при входе с нового устройства или неизвестного IP

#### FR-AUTH-02.2: Доверенные устройства (Remember Device)
- [ ] При входе в систему — опция «Запомнить это устройство» (чекбокс или отдельный экран после успешного 2FA)
- [ ] При отметке устройства как доверенного: выпуск уникального device token (UUID), сохранение его в БД (привязка к пользователю, User-Agent, fingerprint, дата выдачи, срок действия), передача в HttpOnly-cookie с длинным сроком жизни
- [ ] При следующем входе с того же устройства — пропуск шага 2FA (TOTP / FIDO2), если device token действителен
- [ ] `GET /api/auth/trusted-devices` — список доверенных устройств пользователя: название/UA, дата добавления, последнее использование
- [ ] `DELETE /api/auth/trusted-devices/{deviceId}` — отзыв (удаление) доверенного устройства
- [ ] `DELETE /api/auth/trusted-devices` — отзыв всех доверенных устройств сразу
- [ ] Административные настройки: максимальный срок действия device token (по умолчанию 30 дней), запрет на сохранение доверенных устройств (принудительный 2FA при каждом входе)
- [ ] Страница в профиле пользователя: список доверенных устройств с кнопкой удаления каждого

#### FR-AUTH-02.3: Доверенные сертификаты для подписания документов
- [ ] Страница «Мои сертификаты для подписания» в профиле пользователя
- [ ] Загрузка сертификата: форма с полем для PEM/DER/PFX-файла и необязательным паролем к закрытому ключу
- [ ] `POST /api/auth/signing-certificates` — загрузка и привязка сертификата к профилю пользователя
- [ ] Серверная проверка сертификата при загрузке: формат, срок действия, цепочка доверия до корневого CA (настроенного в системе)
- [ ] Хранение только публичной части сертификата на сервере; закрытый ключ хранится на стороне пользователя (смарт-карта, токен, локальное хранилище браузера)
- [ ] Список сохранённых сертификатов: субъект (CN), эмитент, серийный номер, срок действия, дата добавления
- [ ] `DELETE /api/auth/signing-certificates/{certId}` — отвязка и удаление сертификата
- [ ] Отображение статуса сертификата: действителен, истёк, отозван (проверка OCSP при просмотре списка)
- [ ] Использование сертификата при подписании документа в рамках процесса: выбор сертификата из списка доверенных, формирование электронной подписи (CMS/CAdES) на стороне клиента, передача подписи на сервер для верификации и сохранения
- [ ] Административные настройки: список разрешённых корневых CA для подписания, политика обязательного наличия сертификата для определённых ролей

### FR-AUTH-03: Управление доступом (RBAC + ABAC)

#### FR-AUTH-03.1: Роли и права на уровне модулей (RBAC)
- [ ] Административный интерфейс: CRUD-управление ролями (название, описание, цвет/иконка, статус активна/неактивна)
- [ ] Иерархия ролей: наследование прав от родительской роли с возможностью расширения или ограничения
- [ ] Предустановленные системные роли (Администратор, Пользователь, Гость) — защищены от удаления, частично редактируемы
- [ ] Матрица прав: список модулей системы (Процессы, Задачи, Документы, Справочники, Отчёты, Настройки) × типы действий (просмотр, создание, редактирование, удаление, публикация, администрирование)
- [ ] `GET /api/admin/roles` — список ролей с пагинацией и фильтрацией
- [ ] `POST /api/admin/roles` — создание роли
- [ ] `PUT /api/admin/roles/{roleId}` — обновление роли
- [ ] `DELETE /api/admin/roles/{roleId}` — удаление роли (с проверкой: роль не назначена ни одному пользователю)
- [ ] `GET /api/admin/roles/{roleId}/permissions` — матрица прав роли
- [ ] `PUT /api/admin/roles/{roleId}/permissions` — массовое обновление матрицы прав
- [ ] Назначение ролей пользователям: несколько ролей на одного пользователя; назначение через карточку пользователя или страницу роли
- [ ] `POST /api/admin/users/{userId}/roles` / `DELETE /api/admin/users/{userId}/roles/{roleId}` — добавление и снятие роли
- [ ] `GET /api/admin/roles/{roleId}/users` — список пользователей с данной ролью
- [ ] Назначение роли группе пользователей из оргструктуры или LDAP-группы
- [ ] Страница «Роли» в административном интерфейсе: таблица с поиском, фильтрами, счётчиком пользователей на роль
- [ ] Копирование роли (клонирование с именем «Копия <название>») для быстрого создания похожих ролей
- [ ] Экспорт/импорт матрицы прав в JSON/CSV для переноса между окружениями

#### FR-AUTH-03.2: Права на уровне объектов (ABAC — ACL)
- [ ] Политики доступа к объектам (документ, папка, справочник, экземпляр процесса): набор ACE-записей «субъект → операция → разрешено / запрещено»
- [ ] Субъекты политики: конкретный пользователь, роль, подразделение, все авторизованные пользователи
- [ ] Операции над объектами: чтение, запись, удаление, согласование, архивирование, управление правами (share)
- [ ] Явный запрет (Deny) с приоритетом над разрешением (Deny overrides Allow)
- [ ] Наследование прав: дочерние папки/объекты наследуют ACL родительской папки с возможностью переопределения на уровне дочернего
- [ ] Режим «только владелец»: автор объекта всегда сохраняет полный доступ и не может быть лишён его через ACL
- [ ] Интерфейс управления правами (контекстное меню объекта → «Управление доступом»): таблица ACE с добавлением, редактированием и удалением записей
- [ ] `GET /api/permissions/{objectType}/{objectId}` — получение текущего ACL объекта
- [ ] `POST /api/permissions/{objectType}/{objectId}` — добавление новой ACE
- [ ] `PUT /api/permissions/{objectType}/{objectId}/{aceId}` — обновление ACE
- [ ] `DELETE /api/permissions/{objectType}/{objectId}/{aceId}` — удаление ACE
- [ ] Проверка прав на уровне middleware / ASP.NET Core authorization handler перед каждой операцией с объектом
- [ ] Делегирование прав: пользователь с правом «управление правами» может передать часть своих прав на объект другому пользователю
- [ ] Инструмент «Проверить доступ»: администратор вводит пользователя и объект — система отображает итоговый набор прав с объяснением, какое правило сработало

#### FR-AUTH-03.3: Права на уровне полей/свойств объектов
- [ ] Матрица «поле × роль» для каждой формы: состояния — скрыто, только для чтения, редактируемо
- [ ] Настройка матрицы в дизайнере форм (для каждого поля — кнопка/панель прав) и в административном интерфейсе (отдельная вкладка формы)
- [ ] `GET /api/admin/forms/{formId}/field-permissions` — получение матрицы прав полей формы
- [ ] `PUT /api/admin/forms/{formId}/field-permissions` — обновление матрицы
- [ ] Динамические ABAC-правила видимости и редактируемости на основе атрибутов контекста: выражения вида «поле "Сумма" → только чтение, если роль = "Исполнитель" И статус ≠ "Утверждён"»
- [ ] Применение ограничений на уровне API: сервер не включает скрытые поля в ответ, игнорирует значения readonly-полей из тела запроса
- [ ] Маскирование значений по роли (например, номер счёта — только последние 4 цифры для всех ролей, кроме «Финансист»)
- [ ] Отображение в UI: скрытые поля не рендерятся, readonly-поля блокируются средствами фреймворка (React controlled components + серверная валидация)

#### FR-AUTH-03.4: Глобальные настройки доступа
- [ ] Страница глобальных настроек безопасности в административном интерфейсе
- [ ] Режим «закрытая система»: доступ к объекту запрещён по умолчанию, если явно не разрешён ACL
- [ ] Режим «открытая система»: доступ разрешён по умолчанию для всех авторизованных пользователей, если не указан явный запрет
- [ ] Настройка прав по умолчанию для новых объектов: автор — владелец с полным доступом; дополнительные роли и их права (например, роль «Пользователь» — только чтение)
- [ ] `GET /api/admin/access-settings` / `PUT /api/admin/access-settings` — чтение и сохранение глобальных настроек
- [ ] Матрица совместимости ролей (Separation of Duties): настройка пар несовместимых ролей (система запрещает одновременное назначение обеих ролей одному пользователю)
- [ ] Настройка максимального числа ролей у одного пользователя (опционально)
- [ ] Политика доступа для API-ключей и сервисных аккаунтов: отдельный набор разрешений, ограниченная область действия (scopes)
- [ ] Аудит операций: запись в журнал событий всех READ/WRITE/DELETE по защищённым объектам (включается в глобальных настройках, уровень детализации настраивается)
- [ ] Журнал изменений прав: кто, когда, что изменил в ролях, ACL и политиках — с возможностью просмотра diff и отката на предыдущее состояние

#### FR-AUTH-03.5: Замещение пользователей
- [ ] Страница «Замещения» в профиле пользователя (мои замещения) и в административном интерфейсе (все замещения)
- [ ] Создание замещения: выбор замещаемого пользователя, заместителя, периода (дата/время начала и окончания), области действия (все права / только задачи / выбранные роли)
- [ ] `POST /api/substitutions` — создание замещения
- [ ] `GET /api/substitutions` — список активных и запланированных замещений текущего пользователя
- [ ] `GET /api/admin/substitutions` — административный список всех замещений с фильтрами (по пользователям, статусу, периоду)
- [ ] `PUT /api/substitutions/{subId}` — изменение периода или области действия замещения
- [ ] `DELETE /api/substitutions/{subId}` — досрочное завершение замещения
- [ ] Автоматическая активация и деактивация замещения по расписанию: фоновый сервис (Hangfire / .NET hosted service) отслеживает плановые даты
- [ ] Переключатель контекста в интерфейсе: заместитель видит индикатор «Работаю за <ФИО>» и может переключаться между своим рабочим пространством и пространством замещаемого без повторной аутентификации
- [ ] Задачи и процессы замещаемого пользователя отображаются в рабочем пространстве заместителя в режиме замещения
- [ ] Уведомления: заместителю — о начале замещения (email / in-app); замещаемому — о передаче замещения и о его завершении
- [ ] Аудит действий заместителя: в журнале отображается «Иванов И.И. (замещает Петрова П.П.)» для полной прозрачности
- [ ] Ограничения: заместитель не может делегировать полученные полномочия третьим лицам; цепочки замещений (A → B → C) запрещены
- [ ] Административное принудительное замещение: администратор может создать замещение от имени любого пользователя (например, при экстренной ситуации)

## 2. Профиль и оргструктура

### FR-ORG-01: Организационная структура

#### FR-ORG-01.1: Управление деревом подразделений
- [ ] Административный интерфейс: интерактивное дерево подразделений (collapse/expand узлов, drag-and-drop для перемещения узла вместе с дочерними)

### FR-ORG-02: Профиль пользователя

#### FR-ORG-02.1: Основные данные профиля
- [ ] Страница профиля: фото (загрузка, кадрирование, удаление), ФИО (фамилия, имя, отчество), отображаемое имя
- [ ] Контактные данные: рабочий телефон, мобильный телефон, внутренний (добавочный) номер, рабочий email, личный email (опционально)
- [ ] Должность, подразделение, руководитель — отображаются из оргструктуры (FR-ORG-01), редактируются только HR/администратором
- [ ] Дополнительные поля: дата рождения (видимость регулируется настройками приватности), биография/о себе, ссылки на корпоративные мессенджеры
- [ ] `GET /api/users/{userId}/profile` — получение профиля (публичная часть — для всех авторизованных, полная — для владельца и HR/администратора)
- [ ] `PUT /api/users/{userId}/profile` — обновление профиля (с проверкой прав: пользователь редактирует только разрешённые поля, администратор — все)
- [ ] Загрузка фото: `POST /api/users/{userId}/avatar` (multipart/form-data), сервер сохраняет оригинал и генерирует миниатюры (40×40, 80×80, 200×200)
- [ ] Удаление фото: `DELETE /api/users/{userId}/avatar` — сброс на дефолтный аватар
- [ ] Настройки приватности: пользователь выбирает, кто видит телефон/email/дату рождения (все / только подразделение / только HR+администратор)
- [ ] Верификация email при смене: отправка кода подтверждения на новый адрес, применение после подтверждения

#### FR-ORG-02.2: Настройки уведомлений
- [ ] Страница «Уведомления» в профиле: таблица типов событий × каналы (email, SMS, push, in-app)
- [ ] Типы событий (примеры): назначена задача, задача просрочена, согласование требует действия, документ изменён, новое сообщение в чате задачи, системное оповещение, замещение началось/завершилось
- [ ] Для каждого типа события и каждого канала — чекбокс «включено/выключено»
- [ ] `GET /api/users/{userId}/notification-settings` — получение настроек уведомлений
- [ ] `PUT /api/users/{userId}/notification-settings` — сохранение настроек (patch-семантика: передаются только изменённые позиции)
- [ ] Настройка расписания «тихого режима» (Do Not Disturb): интервал времени (например, 22:00–08:00) и дни недели, в которые push/SMS не отправляются
- [ ] Email-дайджест: опция получения сводки за день/неделю вместо немедленных уведомлений (настройка периодичности: сразу / раз в час / раз в день / раз в неделю)
- [ ] Административные настройки: администратор может принудительно включить обязательные уведомления (например, о безопасности), которые пользователь не может отключить
- [ ] Тестовое уведомление: кнопка «Отправить тестовое» для каждого канала (проверка работоспособности настроенного канала)

#### FR-ORG-02.3: Личные настройки интерфейса
- [ ] Язык интерфейса: выбор из списка доступных локалей; применяется немедленно без перезагрузки страницы
- [ ] Часовой пояс: выбор из полного списка IANA TZ; даты и время во всём интерфейсе отображаются в выбранном поясе
- [ ] Тема оформления: светлая / тёмная / системная (следовать настройке ОС)
- [ ] Формат дат и чисел: выбор локали для форматирования (DD.MM.YYYY vs MM/DD/YYYY, разделитель тысяч и дробной части)
- [ ] `GET /api/users/{userId}/preferences` — получение личных настроек
- [ ] `PUT /api/users/{userId}/preferences` — сохранение личных настроек
- [ ] Настройка главной страницы / портлетов: состав, порядок и размер виджетов (сохраняется в профиле, восстанавливается при следующем входе и на других устройствах)
- [ ] Настройка списков по умолчанию: количество записей на странице (10 / 25 / 50 / 100), дефолтная сортировка в таблицах задач и документов
- [ ] Горячие клавиши: просмотр и кастомизация назначений клавиш для основных действий (создать задачу, открыть поиск и т.д.)
- [ ] Сброс настроек: кнопка «Вернуть настройки по умолчанию» с подтверждением

#### FR-ORG-02.4: Безопасность и аккаунт
- [ ] Смена пароля: форма «текущий пароль / новый пароль / подтверждение» с применением политики паролей из FR-AUTH-01.1
- [ ] Управление двухфакторной аутентификацией (2FA): отображение текущего статуса, кнопки регистрации/удаления TOTP и FIDO2-устройств (делегирует в FR-AUTH-01.3)

- [ ] Управление сертификатами для подписания (делегирует в FR-AUTH-02.3): список сертификатов, добавление, удаление
- [ ] Активные сессии (делегирует в FR-AUTH-02.1): список сессий, кнопки завершения конкретной и всех остальных
- [ ] Журнал действий пользователя (личный): последние N событий (вход, смена пароля, изменение профиля, действия с документами и задачами) с датой, IP и типом операции
- [ ] Удаление / деактивация аккаунта: запрос на деактивацию самим пользователем (требует подтверждения пароля); финальное удаление — только администратором

## 3. Главная страница и портал

### FR-PORTAL-01: Настраиваемая главная страница

#### FR-PORTAL-01.1: Архитектура портлетов и их управление
- [ ] Сетка-контейнер (grid layout): адаптивная система колонок (1–4), фиксированная высота строки; на мобильных устройствах — одноколоночный стек
- [ ] Drag-and-drop перемещение и изменение размера портлетов (resize-handles по правому и нижнему краям); сохранение позиций в профиле пользователя
- [ ] `GET /api/portal/dashboard` — получение текущей конфигурации рабочего стола (список портлетов с позициями, размерами, параметрами)
- [ ] `PUT /api/portal/dashboard` — сохранение конфигурации рабочего стола (полная замена)
- [ ] `POST /api/portal/dashboard/widgets` — добавление нового портлета на рабочий стол
- [ ] `DELETE /api/portal/dashboard/widgets/{widgetInstanceId}` — удаление экземпляра портлета
- [ ] `PUT /api/portal/dashboard/widgets/{widgetInstanceId}` — обновление настроек отдельного портлета (заголовок, параметры, размер)
- [ ] Режим редактирования (Edit mode): кнопка «Настроить» переводит рабочий стол в режим, где разрешены перемещение/удаление; выход из режима — «Сохранить» / «Отмена»
- [ ] Кнопка «Добавить виджет»: боковая панель со списком доступных портлетов, поиск по названию, группировка по категории (Задачи / Процессы / Документы / Отчёты / Информация)
- [ ] Сброс рабочего стола к ролевому шаблону по умолчанию (с подтверждением)
- [ ] Синхронизация конфигурации рабочего стола между устройствами (сохранение в профиле, применяется при следующем входе)
- [ ] API портлетов: каждый виджет запрашивает свои данные самостоятельно через отдельный endpoint; индикатор загрузки и обработка ошибок внутри каждого портлета
- [ ] Автообновление данных виджетов по настраиваемому интервалу (30 с / 1 мин / 5 мин / вручную); индикатор «последнее обновление»
- [ ] Коллапс портлета: кнопка сворачивания до заголовка (сохраняется в конфигурации)

#### FR-PORTAL-01.2: Библиотека встроенных виджетов
- [ ] **Мои задачи**: список активных задач пользователя с приоритетом, сроком, статусом; фильтр «просроченные / сегодня / на этой неделе»; быстрый переход к задаче
- [ ] **Мои процессы**: список запущенных экземпляров процессов с именем, стадией, датой старта; ссылка на карточку экземпляра
- [ ] **Ожидает действия**: сводный список всех объектов, требующих реакции пользователя (задачи на согласование, входящие документы, уведомления); счётчик на заголовке
- [ ] **Уведомления**: последние N непрочитанных уведомлений с типом, текстом и временем; кнопка «Пометить все как прочитанные»; ссылка на полный журнал уведомлений
- [ ] **Мои коллеги** (из FR-ORG-01.4): список прямых коллег и непосредственного руководителя с фото, должностью, статусом онлайн; быстрый поиск по подразделению
- [ ] **Новости / Объявления**: последние записи из раздела новостей портала с превью и датой; настройка источника (категория новостей или тег)
- [ ] **Последние документы**: последние открытые / изменённые документы пользователя; столбцы — имя, тип, дата изменения
- [ ] **Избранное**: ярлыки на объекты, добавленные пользователем в избранное (задачи, документы, процессы, страницы)
- [ ] **Мой календарь**: мини-календарь с отображением задач и событий текущего месяца; клик на дату — список объектов
- [ ] **Сводный отчёт** (аналитика): настраиваемый виджет-диаграмма (круговая / столбчатая / линейная) на основе предопределённых отчётов; выбор отчёта и периода в настройках виджета
- [ ] **Быстрые действия**: набор кнопок-ярлыков для частых действий (Создать задачу / Запустить процесс / Загрузить документ); состав настраивается пользователем
- [ ] **HTML-виджет** (для администратора): произвольный HTML/markdown-контент (объявления, инструкции, ссылки); редактируется в визуальном редакторе
- [ ] **Внешний RSS-виджет**: отображение заголовков из RSS-канала; настройка URL источника, количества записей, автообновление

#### FR-PORTAL-01.3: Ролевые страницы и персонализация
- [ ] Ролевые шаблоны рабочего стола: администратор задаёт шаблонную конфигурацию для каждой системной роли (набор виджетов, позиции, размеры)
- [ ] `GET /api/portal/templates` — список ролевых шаблонов
- [ ] `POST /api/portal/templates` — создание нового шаблона
- [ ] `PUT /api/portal/templates/{templateId}` — обновление шаблона
- [ ] `DELETE /api/portal/templates/{templateId}` — удаление шаблона
- [ ] `POST /api/portal/templates/{templateId}/publish` — применение шаблона ко всем пользователям с соответствующей ролью (с опцией «не перезаписывать пользовательские конфигурации»)
- [ ] Приоритет шаблонов при нескольких ролях: администратор задаёт порядок приоритетности ролей
- [ ] Принудительные виджеты: администратор помечает отдельные виджеты в шаблоне как «обязательные» — пользователь не может их удалить или переместить за пределы зафиксированной области
- [ ] Копирование конфигурации рабочего стола от одного пользователя другому (инструмент администратора)
- [ ] Текстовые страницы портала: отдельные страницы с rich-text (TipTap / CKEditor), доступные всем авторизованным; создание/редактирование только для роли «редактор портала»
- [ ] Страницы с встроенными портлетами: смешанная страница — верхняя часть rich-text, нижняя — колонки с виджетами; полезна для разделов «О компании», «Новости» с боковой панелью

#### FR-PORTAL-01.4: Навигация и оформление портала
- [ ] Боковое (или верхнее) меню навигации: иерархические пункты (до 3 уровней), иконка, ярлык и URL / встроенная страница / внешняя ссылка
- [ ] `GET /api/portal/menu` — получение структуры меню для текущего пользователя (с учётом ролей)
- [ ] `PUT /api/portal/menu` — сохранение структуры меню (только для администратора); drag-and-drop в административном интерфейсе
- [ ] Видимость пунктов меню: настройка по ролям (показывать / скрывать конкретный пункт для выбранных ролей)

### FR-PORTAL-02: Поиск

#### FR-PORTAL-02.1: Полнотекстовый поиск
- [ ] Единое поле глобального поиска в хедере портала: поиск по всем проиндексированным типам объектов одновременно
- [ ] `GET /api/search?q={query}&types={}&filters={}` — унифицированный поисковый endpoint; возвращает результаты, сгруппированные по типу объекта
- [ ] Поиск по следующим типам объектов: задачи, экземпляры процессов, документы (метаданные + содержимое), пользователи, подразделения, должности, страницы портала, новости
- [ ] Ранжирование результатов: релевантность (TF-IDF / BM25), дата изменения, тип объекта; настройка весов в конфигурации
- [ ] Подсветка вхождений поискового запроса в заголовке и тексте анонса результата (highlight)
- [ ] Поиск с учётом морфологии русского и английского языков (стемминг / лемматизация через встроенные анализаторы поискового движка)
- [ ] Поиск с опечатками (fuzzy search): допуск до 1–2 замен символов; порог нечёткости настраивается
- [ ] Автодополнение (autocomplete): при вводе от 2 символов в поле поиска — выпадающий список подсказок (название задачи, имя пользователя, номер документа); `GET /api/search/suggest?q=`
- [ ] Результаты живого поиска (instant search): предпросмотр топ-5 результатов прямо в выпадающем меню хедера (без перехода на страницу поиска)
- [ ] Страница полных результатов: открывается при нажатии Enter или «Показать все результаты»; пагинация (page / page_size)
- [ ] Права доступа в поиске: результаты фильтруются по правам текущего пользователя (не возвращаются объекты, к которым у него нет доступа)
- [ ] Индексирование в реальном времени: при создании/изменении/удалении объекта событие публикуется во внутреннюю очередь, индекс обновляется асинхронно с задержкой не более 5–10 с

#### FR-PORTAL-02.2: Расширенный поиск и EQL
- [ ] Страница «Расширенный поиск»: форма с отдельными полями-фильтрами для каждого типа объекта; переключатель «Простой / Расширенный»
- [ ] Фильтры для задач: статус, приоритет, исполнитель, назначивший, процесс, дата создания (диапазон), срок (диапазон), теги
- [ ] Фильтры для документов: тип документа, автор, папка, дата изменения (диапазон), статус, версия
- [ ] Фильтры для процессов: определение процесса, стадия, исполнитель, дата запуска (диапазон), статус
- [ ] Фильтры для пользователей: подразделение, должность, роль, статус (активный/архивный)
- [ ] EQL (Enterprise Query Language): текстовый синтаксис запросов вида `status:active AND assignee:me AND due:<2026-05-01 AND tag:"важно"`; поддержка операторов AND, OR, NOT, скобок, wildcards (`*`, `?`), диапазонов (`due:[2026-01-01 TO 2026-06-01]`), специальных значений (`me`, `today`, `this_week`)
- [ ] Поле EQL в расширенном поиске: подсветка синтаксиса, автодополнение полей и значений, сообщения об ошибках парсинга
- [ ] `GET /api/search/eql?q={eql_query}` — поиск через EQL; возвращает единый список с type-дискриминатором
- [ ] Сохранённые поисковые запросы: пользователь сохраняет поиск под именем; `POST /api/search/saved`, `GET /api/search/saved`, `DELETE /api/search/saved/{id}`
- [ ] Запуск сохранённого поиска одним кликом из меню «Сохранённые поиски»
- [ ] Поиск-подписка: пользователь подписывается на сохранённый поиск — получает уведомление при появлении новых результатов; `POST /api/search/saved/{id}/subscribe`
- [ ] Экспорт результатов поиска в Excel / CSV (до 10 000 записей): `GET /api/search/export?q=...&format=xlsx`

#### FR-PORTAL-02.3: Поисковые результаты и UX
- [ ] Страница результатов: вкладки по типам объектов с счётчиками («Все (142) / Задачи (47) / Документы (31) / ...»)
- [ ] Карточка результата: иконка типа, название (ссылка), анонс с подсветкой, метаданные (автор, дата, статус)
- [ ] Сортировка результатов: по релевантности / по дате изменения (↑/↓) / по названию (↑/↓)
- [ ] Панель фильтров слева на странице результатов: динамические фасеты (тип объекта, статус, автор, дата) — обновляются при каждом запросе с актуальными счётчиками
- [ ] История поиска: последние 20 запросов текущего пользователя в выпадающем меню поля поиска; очистка истории
- [ ] «Нет результатов»: страница с подсказками (проверьте орфографию, попробуйте более короткий запрос, похожие запросы)
- [ ] Быстрые действия из карточки результата: открыть, добавить в избранное, скопировать ссылку — без перехода на страницу объекта
- [ ] Клавиатурная навигация по результатам: стрелки вверх/вниз, Enter для перехода, Escape для закрытия
- [ ] Кешированные запросы: повторные одинаковые запросы в течение 30 с возвращаются из кеша без обращения к поисковому движку

#### FR-PORTAL-02.4: Поисковая индексация и администрирование
- [ ] Административная страница поиска: статистика индекса (количество документов по типам, размер, дата последнего обновления)
- [ ] `POST /api/admin/search/reindex` — запуск полной переиндексации (асинхронно); `GET /api/admin/search/reindex/status` — статус задачи переиндексации
- [ ] Настройка типов объектов для индексации: включение/отключение индексации конкретных типов; `PUT /api/admin/search/settings`
- [ ] Настройка весов полей: настройка boost-коэффициентов для полей (название > описание > содержимое файла)
- [ ] Стоп-слова: список слов, исключаемых из индекса; редактирование через административный интерфейс
- [ ] Синонимы: таблица синонимов (например, «договор» = «контракт»); применяются при индексации и поиске
- [ ] Индексация содержимого файлов: извлечение текста из PDF (pdftotext / iTextSharp), DOCX (Open XML SDK), XLSX, TXT, HTML; файлы > 50 МБ индексируются частично (первые 50 МБ)
- [ ] Очередь индексации: просмотр очереди ожидающих индексации событий; ручной сброс застрявших задач
- [ ] Метрики поиска: топ-20 запросов за период, запросы с нулевыми результатами, среднее время ответа; экспорт в CSV

## 4. Бизнес-процессы

### FR-BPM-01: Моделирование процессов (Дизайнер)

#### FR-BPM-01.1: Графический редактор BPMN 2.0
- [ ] Холст (canvas) с бесконечной прокруткой, масштабирование (Ctrl+scroll / кнопки ±), режим «По размеру» (Fit to screen)
- [ ] Палитра элементов BPMN 2.0 слева: события, задачи, шлюзы, соединители, подпроцессы, swimlane, аннотации, артефакты (группы, текстовые аннотации, ассоциации)
- [ ] Drag-and-drop элементов из палитры на холст; snap-to-grid (настраиваемый шаг сетки); выравнивание по вспомогательным линиям
- [ ] **События**: стартовое (пустое, таймер, сообщение, сигнал, условие, ошибка), конечное (пустое, сообщение, ошибка, эскалация, завершение), промежуточное захватывающее / генерирующее (таймер, сообщение, сигнал, ошибка, эскалация, компенсация, граничное прерывающее / непрерывающее)
- [ ] **Задачи**: пользовательская (User Task), сервисная (Service Task / HTTP-вызов), бизнес-правило (Business Rule Task), скрипт (Script Task / C#), ручная (Manual Task), отправка сообщения (Send Task), получение сообщения (Receive Task), RPA-задача (расширение)
- [ ] **Шлюзы**: эксклюзивный (XOR), параллельный (AND), включающий (OR), основанный на событии (Event-based), сложный (Complex)
- [ ] **Подпроцессы**: встроенный (inline), ссылочный (Call Activity — вызов другого определения процесса), событийный подпроцесс (Event Sub-Process), транзакционный подпроцесс
- [ ] **Дорожки (swimlanes)**: пул (Pool) и дорожка (Lane) с изменяемыми размерами; автоматическое переназначение задач при перемещении между дорожками
- [ ] Маркеры задач: цикл (Loop), множественный экземпляр последовательный/параллельный (MI), компенсация (Compensation), ад-хок (Ad-hoc)
- [ ] Соединители: поток управления (Sequence Flow) с условием на XOR/OR; поток сообщений (Message Flow) между пулами; ассоциация (Association)
- [ ] Редактирование соединителя: точки перегиба (waypoints), ортогональный/прямой/изогнутый режим линии
- [ ] Контекстное меню элемента (правая кнопка): свойства, дублировать, удалить, добавить граничное событие, изменить тип
- [ ] Быстрое добавление следующего элемента через popover-иконки вокруг выбранного узла (morph menu)
- [ ] Операции буфера обмена: копировать / вырезать / вставить (Ctrl+C/X/V); Ctrl+Z / Ctrl+Y (undo/redo с историей до 100 шагов)
- [ ] Мультиселект (рамочное выделение + Shift+клик); групповое перемещение, удаление, изменение размера
- [ ] Сохранение диаграммы в формате BPMN 2.0 XML (`PUT /api/processes/{id}/diagram`); экспорт в PNG/SVG/PDF
- [ ] Импорт BPMN XML из файла (`POST /api/processes/{id}/diagram/import`); валидация схемы при импорте

#### FR-BPM-01.2: Свойства элементов и конфигурация
- [ ] Панель свойств справа: открывается при выборе элемента, содержит вкладки — Основное, Выполнение, Форма, Переменные, Уведомления, Дополнительно
- [ ] **Пользовательская задача**: исполнитель (роль / конкретный пользователь / выражение из переменной), срок (фиксированный / относительный от даты старта / из переменной), приоритет, форма задачи (ссылка на форму из FR-BPM-01.4), делегирование (разрешено/запрещено), контроль руководителем
- [ ] **Сервисная задача / HTTP-вызов**: метод (GET/POST/PUT/DELETE/PATCH), URL (поддержка шаблонов с переменными процесса), заголовки, тело запроса (JSON-шаблон), маппинг ответа в переменные процесса, таймаут, количество повторов при ошибке, стратегия retry (linear / exponential backoff)
- [ ] **Скрипт (C#)**: встроенный редактор кода с подсветкой синтаксиса (Monaco Editor), автодополнение переменных процесса, кнопка «Проверить синтаксис», кнопка «Эмулировать» (запуск скрипта с тестовым контекстом без сохранения в БД)
- [ ] **Бизнес-правило**: выбор опубликованной таблицы правил из FR-BPM-01.3, маппинг входных переменных процесса → входные поля правила, маппинг выходных полей правила → переменные процесса
- [ ] **RPA-задача**: выбор робота (из реестра RPA-агентов), передача входных параметров (маппинг переменных), получение результата, таймаут ожидания, поведение при недоступности агента
- [ ] **Шлюз XOR / OR**: настройка условий переходов в виде EL-выражений (`${переменная == "значение"}`), выбор перехода «по умолчанию» (default flow)
- [ ] **Таймерное событие**: настройка типа — дата (ISO 8601), длительность (`PT1H30M`), цикл (cron или ISO 8601 R-cycle); поддержка переменных процесса в значении
- [ ] **Сигнальное / сообщение-событие**: выбор сигнала / имени сообщения из глобального реестра; для throw-событий — маппинг данных в полезную нагрузку
- [ ] **Граничное событие**: привязка к задаче, тип (таймер / ошибка / сигнал / сообщение), прерывающее/непрерывающее; задание маршрута эскалации
- [ ] **Уведомления на задаче**: настройка триггеров (создана, просрочена, выполнена, переназначена), шаблон сообщения (переменные процесса через mustache `{{var}}`), каналы (email / push / in-app), получатели (исполнитель / инициатор / роль / выражение)
- [ ] **Переменные процесса**: список переменных с именем, типом (string, int, decimal, bool, date, datetime, json, file, user, list), значением по умолчанию, признаком «ключевая переменная» (отображается на карточке экземпляра), признаком «внешняя» (передаётся при запуске)
- [ ] **Матрица ответственности**: таблица «стадия процесса × роль», ячейки: ответственный (R), исполнитель (A), консультант (C), информируется (I) — RACI

#### FR-BPM-01.3: Таблицы бизнес-правил (DMN)

- [ ] Редактор таблицы DMN-решений: колонки — входные условия (тип, оператор, значение) и выходные результаты; хит-политики (FIRST, UNIQUE, COLLECT, ANY, RULE ORDER, OUTPUT ORDER)

#### FR-BPM-01.4: Конструктор форм задач

- [ ] **Компоненты ввода**: текстовое поле (однострочное / многострочное), число, дата/время, чекбокс, переключатель (radio), выпадающий список (справочник / статичные варианты / из переменной), поле выбора пользователя / подразделения / должности, загрузка файла(ов), подпись (draw/type), таблица-грид (ввод нескольких строк)

- [ ] **Сложные компоненты**: вложенная форма (subform), карточка связанного объекта (документ, задача), компонент «согласование» (кнопки Согласовать / Отклонить + поле комментария)

#### FR-BPM-01.5: Версионирование, валидация и публикация процесса
- [ ] Каждое сохранение диаграммы создаёт новый черновик-снапшот с инкрементальным номером (draft); сохранение вручную по кнопке «Сохранить черновик» или автосохранение каждые 30 с
- [ ] `GET /api/processes/{id}/versions` — список всех версий (черновики + опубликованные); просмотр диаграммы любой версии
- [ ] Валидация перед публикацией (`POST /api/processes/{id}/validate`): проверки — каждый стартовый элемент соединён с потоком, каждый конечный элемент достижим, все шлюзы имеют корректные условия на исходящих потоках, все User Task имеют исполнителя и форму, все Service Task имеют URL, все Receive Task имеют имя сообщения; вывод списка ошибок и предупреждений с навигацией к проблемному элементу
- [ ] Публикация (`POST /api/processes/{id}/publish`): версия помечается как «Активная»; предыдущая активная версия переходит в статус «Устаревшая»; у процесса всегда ровно одна активная версия
- [ ] Сравнение двух версий (diff): визуальное выделение добавленных / удалённых / изменённых элементов на холсте; текстовое diff-окно изменений свойств
- [ ] Откат к ранней версии: создание новой версии-копии из выбранной исторической версии (не удаляет историю)
- [ ] **Debug-режим**: запуск тестового экземпляра с пошаговым выполнением; на диаграмме подсвечивается текущий активный узел; доступен контекст переменных на каждом шаге; возможность вручную завершить или пропустить текущую задачу; трассировка — лог событий с временными метками
- [ ] Документирование процесса: автогенерация PDF-регламента (`GET /api/processes/{id}/document`); содержит: диаграмму (SVG), таблицу задач с исполнителями, RACI-матрицу, описание переменных, описание переходов; шаблон регламента настраивается администратором

### FR-BPM-02: Выполнение процессов

#### FR-BPM-02.1: Запуск экземпляра
- [ ] Запуск вручную из UI: кнопка «Запустить процесс» на странице определения процесса → диалог заполнения стартовой формы (поля, помеченные как «внешние» переменные) → `POST /api/processes/{id}/instances`
- [ ] Запуск по расписанию: настройка cron-выражения или ISO 8601 R-цикла в стартовом таймерном событии; `POST /api/scheduler/jobs` — регистрация задания в планировщике; поддержка часовых поясов
- [ ] Запуск по событию (сообщение / сигнал): `POST /api/events/messages/{messageName}` — отправка сообщения с коррелирующими ключами и полезной нагрузкой; `POST /api/events/signals/{signalName}` — широковещательный сигнал всем подписанным экземплярам
- [ ] Запуск из другого процесса (Call Activity): передача входных переменных, получение выходных переменных по завершении дочернего экземпляра
- [ ] Запуск через REST-webhook: внешняя система вызывает `POST /api/webhooks/{webhookKey}` — создаётся экземпляр процесса, тело запроса маппируется в переменные
- [ ] Пакетный запуск: администратор загружает CSV/Excel-файл со значениями переменных → создаётся N экземпляров (`POST /api/processes/{id}/instances/batch`); прогресс запуска отображается в реальном времени

#### FR-BPM-02.2: Страница и управление экземпляром
- [ ] `GET /api/instances/{instanceId}` — получение полного состояния экземпляра: статус, версия процесса, дата старта/завершения, инициатор, ключевые переменные
- [ ] Карта процесса на странице экземпляра: та же диаграмма BPMN, на которой выделены завершённые узлы (зелёный), активные (синий / анимация), узлы с ошибкой (красный); отображение количества выполненных итераций на узлах с маркером Loop/MI
- [ ] Панель «Контекст»: таблица всех переменных экземпляра с текущими значениями; редактирование значений переменных (для администратора / роли «оператор процессов»); история изменений каждой переменной
- [ ] Панель «История»: хронологический лог событий экземпляра — запуск, выполнение задачи, переход по потоку, срабатывание таймера, ошибка, изменение переменной, действие пользователя; каждая запись содержит: временную метку, тип события, актора, детали; пагинация
- [ ] Панель «Задачи»: список всех пользовательских задач экземпляра (активные + завершённые) с исполнителями, сроками, статусами; переход к задаче
- [ ] Переключение версии работающего экземпляра: `PUT /api/instances/{instanceId}/version` — миграция на другую версию процесса; алгоритм миграции: сохранение текущих переменных, перенос активных токенов на совместимые узлы новой версии; предупреждение о несовместимых изменениях
- [ ] Ручное управление токеном: администратор / оператор может переместить активный токен на другой узел (`PUT /api/instances/{instanceId}/token`); добавить / удалить токен (для параллельных шлюзов)
- [ ] Смена статуса экземпляра: suspend (приостановка — все активные задачи замораживаются) / resume / terminate (немедленное завершение с событием Terminate End) / cancel (отмена с компенсацией)
- [ ] Комментарии к экземпляру: пользователи могут оставлять текстовые комментарии; `POST /api/instances/{instanceId}/comments`, `GET /api/instances/{instanceId}/comments`
- [ ] Вложения к экземпляру: загрузка файлов; `POST /api/instances/{instanceId}/attachments`

#### FR-BPM-02.3: Монитор процессов
- [ ] Страница «Монитор»: таблица всех экземпляров процессов с колонками — процесс, версия, статус, инициатор, дата старта, дата завершения, текущий узел, ключевая переменная
- [ ] Фильтры: по определению процесса, статусу (активный / завершён / ошибка / приостановлен / отменён), инициатору, дате старта (диапазон), дате завершения (диапазон), ключевой переменной (текстовый поиск)
- [ ] Быстрые фильтры-кнопки: «Активные», «С ошибкой», «Просроченные» (экземпляры, у которых активная User Task просрочена)
- [ ] Массовые действия: suspend / resume / terminate для выбранных экземпляров
- [ ] Экспорт списка экземпляров в Excel / CSV: `GET /api/instances/export`
- [ ] Дашборд мониторинга: виджеты — количество активных экземпляров по процессам (столбчатая), динамика запуска/завершения по дням (линейная), топ-5 процессов по количеству экземпляров, процент выполненных в срок
- [ ] Оповещения на ошибки: при переходе экземпляра в статус «Ошибка» — уведомление администратору / ответственному за процесс через in-app + email; `PUT /api/processes/{id}/alert-settings`
- [ ] «Мои процессы»: личная страница пользователя — экземпляры, в которых он является инициатором или активным исполнителем; фильтры аналогичны монитору

#### FR-BPM-02.4: Очередь исполнения и обработка ошибок
- [ ] Асинхронные узлы: Service Task, Script Task, RPA-задача, таймерные события выполняются в фоновой очереди (Hangfire / MassTransit); узел помечается как «асинхронный» в свойствах
- [ ] `GET /api/admin/queue` — просмотр очереди: ожидающие, выполняющиеся, завершённые, упавшие задания с деталями (процесс, экземпляр, узел, попытка, последняя ошибка, время следующей попытки)
- [ ] Ручной retry упавшего задания: `POST /api/admin/queue/{jobId}/retry`
- [ ] Dead-letter queue: задания, исчерпавшие все попытки retry, помещаются в DLQ; просмотр и ручная обработка
- [ ] Граничное событие «Ошибка» на задаче: при падении задачи (исключение в скрипте / HTTP 5xx / таймаут) — активируется граничный поток ошибки; тип ошибки (errorCode) можно проверить в условии перехода
- [ ] Обработка ошибок сервисной задачи: настройка числа повторов (retry count), задержки перед повтором, типов исключений для retry vs немедленного фейла
- [ ] Мониторинг производительности узлов: `GET /api/analytics/nodes` — среднее и перцентильное время выполнения каждого узла по всем экземпляром за период; выявление узких мест

## 5. Задачи

### FR-TASK-01: Управление задачами

#### FR-TASK-01.1: Создание и карточка задачи
- [ ] Навигация в раздел «Задачи»: через левое меню системы; через главное меню «Мои задачи»; через портлет «Задачи» на главной странице
- [ ] Создание задачи тремя способами: кнопка «Создать» → «Задачу» на главной странице; «Создать задачу» в главном меню; кнопка «Создать» или выпадающее меню в разделе «Задачи»; `POST /api/tasks`
- [ ] Создание задачи из процесса: User Task в BPMN автоматически создаёт запись при достижении узла, заполняя поля из переменных процесса; связь с экземпляром процесса и узлом сохраняется
- [ ] **Роли пользователей в задаче**:
  - *Автор* — создатель задачи; имеет права просмотра и редактирования; руководители автора также могут редактировать
  - *Исполнитель* — ответственный за выполнение; может быть переназначен; имеет права просмотра
  - *Соисполнитель* — участвует в выполнении вместе с исполнителем, не является ответственным; получает все оповещения; имеет право выполнения задачи, если администратор разрешил (`настройка «Соисполнитель имеет право выполнения»`)
  - *Наблюдатель* — получает оповещения о действиях с задачей; является подписчиком
  - *Согласующий* — ответственный за согласование задачи
  - *Контролёр* — контролирует процесс выполнения задачи; у задачи может быть только один контролёр
  - Права просмотра: автор, исполнитель, соисполнитель, наблюдатель, контролёр, руководители автора/исполнителя/соисполнителя (согласно оргструктуре)
  - Права редактирования: автор задачи, руководители автора (согласно оргструктуре)
- [ ] **Поля карточки задачи**:
  - Тема (обязательно) — краткая формулировка; отображается в списке задач и портлете
  - Исполнитель (обязательно) — один пользователь или сохранённая группа
  - Сроки задачи (обязательно) — дата и время начала и завершения; задача может длиться несколько дней; по умолчанию нельзя установить на прошедшую дату (разрешается отдельным правом)
  - Корректировка сроков — автоматическое смещение даты начала, если она выпала на праздничный/выходной день: *Без смещения* (по умолчанию, значение берётся из настроек администратора), *Первый рабочий день* (смещает дату начала; дата завершения не меняется), *Сохранять длительность* (смещаются обе даты с учётом длительности)
  - Приоритет — степень важности задачи
  - Категория — из справочника «Категория задач»; участвует в фильтрации и сортировке
  - Описание — подробное объяснение задачи
  - Вложения — файлы с компьютера / внешних носителей и электронные документы; `POST /api/tasks/{id}/attachments`
  - Согласующий (+Согласующий) — один пользователь; опционально при создании
  - Соисполнители (+Соисполнитель) — участвуют в выполнении; получают все оповещения по задаче
  - Наблюдатели (+Соисполнитель) — следят за ходом выполнения; являются подписчиками
  - Контроль выполнения (+Контроль) — тип контроля и контролёр; если не заполнено, применяется тип по умолчанию из настроек администратора
  - Плановые трудозатраты (через «Еще») — плановое время на задачу; прогресс-бар «факт / план» на карточке
  - Мероприятие (через «Еще») — привязка задачи к маркетинговому мероприятию
  - Проект (через «Еще») — привязка задачи к проекту (при наличии приложения Проекты+)
- [ ] Страница задачи — вкладки: *Описание* (срок, описание, вложения, автор, исполнители, контролёр, статус, приоритет, история действий), *Родительская задача* (если текущая является подзадачей и есть права доступа к родителю), *Подзадачи* (если созданы подзадачи), *Трудозатраты* (после добавления трудозатрат)
- [ ] Подзадачи: создание подзадач внутри карточки; дерево подзадач с раскрытием; `POST /api/tasks/{id}/subtasks`; при переходе на родительскую задачу после выполнения дочерней — настраивается администратором
- [ ] Комментарии к задаче: добавление комментария, вложений, ответов на вопросы; история (комментарии, вложения, вопросы, действия) отображается на вкладке «Описание»; `GET/POST /api/tasks/{id}/comments`

#### FR-TASK-01.2: Статусы задачи
- [ ] Полный жизненный цикл простой задачи со следующими статусами:
  - *Новая* — задача создана, отредактирована или переназначена на нового исполнителя
  - *Прочитана* — исполнитель первый раз открыл страницу задачи
  - *Выполняется* — исполнитель выбрал действие «Работа начата»
  - *На предварительном согласовании* — автор задал согласующего при создании и сохранил задачу
  - *На согласовании* — исполнитель выбрал «Отправить на согласование»
  - *Согласовано* — согласующий нажал «Согласовать»; задача возвращается к исполнителю со статусом «Новая»
  - *Отказано в предварительном согласовании* — согласующий нажал «Отказать» при предварительном согласовании
  - *Отказано в согласовании* — согласующий нажал «Отказать»; можно повторно отправить на согласование
  - *Выполнено* — исполнитель нажал «Сделано» (контроль не назначен)
  - *Выполнено, нужен контроль* — исполнитель нажал «Сделано», контролёр ещё не проверил
  - *Выполнено и проконтролировано* — исполнитель и контролёр оба нажали «Сделано»
  - *Невозможно выполнить* — исполнитель выбрал «Невозможно выполнить» (контроль не назначен)
  - *Невозможно выполнить, нужен контроль* — исполнитель выбрал «Невозможно выполнить», контролёр ещё не проверил
  - *Невозможно выполнить и проконтролировано* — исполнитель выбрал «Невозможно выполнить» и контролёр нажал «Сделано»
  - *Закрыто* — автор задачи или пользователь с правами управления задачей выбрал «Отменить задачу»; также присваивается незавершённой задаче переназначения заблокированного пользователя при его разблокировке
- [ ] Автоматический флаг просрочки: фоновый job проверяет задачи с истёкшим сроком и нефинальным статусом; флаг хранится отдельно от статуса; в списке задачи выделяются красным
- [ ] При переходе задачи в финальный статус («Выполнено…» / «Закрыто»): уведомление наблюдателям; если задача из процесса — отправка результата в движок процессов (complete User Task token)
- [ ] SLA-правила: администратор задаёт правила — для задач категории X с приоритетом Y срок по умолчанию Z рабочих часов; при создании задачи срок заполняется автоматически по SLA; `GET/POST/PUT/DELETE /api/admin/task-sla`

#### FR-TASK-01.3: Согласование задачи
- [ ] Два способа отправки на согласование:
  - *Предварительное согласование* — автор добавляет согласующего (+Согласующий) при создании задачи; задача получает статус «На предварительном согласовании» сразу после сохранения
  - *Согласование от исполнителя* — исполнитель выбирает «Отправить на согласование» (меню «Еще»); указывает согласующего, сроки и комментарий
- [ ] При создании согласования формируется отдельная задача согласования для согласующего; родительская задача (обозначение ![approve]) и задача согласования (обозначение ![approval-sub]) различаются иконками
- [ ] До завершения задачи согласования исполнитель не может выполнить родительскую задачу
- [ ] Действия согласующего: *Согласовать* / *Отказать* + опциональный комментарий и вложение; результат и комментарий отображаются на странице родительской задачи
- [ ] `GET /api/tasks/{id}/approval` — текущее состояние согласования; `POST /api/tasks/{id}/approval/decision` — решение согласующего
- [ ] При согласовании родительская задача возвращается к исполнителю со статусом «Новая»; при отказе — статус «Отказано в согласовании»; можно повторно отправить
- [ ] Уведомление согласующему при поступлении задачи на его согласование; напоминание при приближении/истечении срока
- [ ] Шаблоны маршрутов согласования: `GET/POST/PUT/DELETE /api/admin/approval-templates`; возможность сохранить текущий маршрут как шаблон

#### FR-TASK-01.4: Контроль исполнения и трудозатраты
- [ ] У задачи может быть только один контролёр; назначается при создании через «+Контроль» или изменяется после создания
- [ ] Типы контроля (настраиваются при создании задачи, выбор дефолтного — у администратора):
  - *Контроль выполнения* — после нажатия исполнителем «Сделано» задача направляется контролёру; контролёр может принять («Сделано» → статус «Выполнено и проконтролировано») или вернуть («Открыть заново» → задача получает статус «Новая» у исполнителя)
  - *Текущий контроль* — задача отображается у контролёра с момента создания; контролёр отслеживает весь процесс; после выполнения исполнителем переходит в фильтр «Контроль выполнения» у контролёра; может вернуть задачу на доработку
  - *Оповещать при выполнении* — пользователь получает оповещение в ленте сообщений после выполнения задачи; не может принять или отклонить результат
  - *Без контроля* — контролёр не назначается
- [ ] Настройки контроля в системе (администратор): автоматически устанавливать контроль при создании (Да/Нет), тип контроля по умолчанию, разрешить пользователям изменять контроль после создания, разрешить персональную настройку контроля в профиле пользователя
- [ ] Изменение параметров контроля после создания задачи: автор или контролёр меняет контролёра, тип контроля, или снимает контроль; `PUT /api/tasks/{id}/control`
- [ ] Взять задачу на текущий контроль: любая роль может выбрать «Взять на контроль» в меню «Еще»; задача появляется в фильтре «Текущий контроль»; снять через «Снять с контроля»
- [ ] Массовый контроль задач: выбрать несколько задач флажками → «Сделано» — подтвердить; `POST /api/tasks/bulk-verify`
- [ ] `POST /api/tasks/{id}/verify` — отметка контроля; `GET /api/tasks/{id}/verification` — статус и имя контролёра
- [ ] Трудозатраты при выполнении задачи: в диалоге «Сделано» — поля «Затрачено» (чч:мм) и «Дата начала»; `POST /api/tasks/{id}/timelogs`
- [ ] Трудозатраты в процессе работы: «Еще» → «Добавить трудозатраты» — указать время, дату начала, опциональный комментарий; `POST /api/tasks/{id}/timelogs`
- [ ] Настройка обязательного добавления трудозатрат администратором: пользователь не может выполнить задачу без ввода трудозатрат
- [ ] Плановые трудозатраты на задаче (поле «Плановые трудозатраты»); прогресс-бар «факт / план» на карточке
- [ ] Суммирование трудозатрат по подзадачам на родительской задаче
- [ ] Журнал трудозатрат на вкладке «Трудозатраты» страницы задачи: две панели — *Отчёт о поданных трудозатратах* (если трудозатраты отправлены на согласование; сводная таблица с группировкой по виду деятельности или исполнителям) и *Журнал трудозатрат* (трудозатраты, добавленные через «Еще»); `GET /api/tasks/{id}/timelogs`
- [ ] Отчёт по трудозатратам: `GET /api/reports/timelogs` — фильтры по периоду, пользователю, задаче, проекту; экспорт в Excel; группировка по дням / неделям / месяцам / исполнителю / задаче

#### FR-TASK-01.5: Типы задач

##### FR-TASK-01.5.1: Периодическая задача
- [ ] Периодическая задача — задача, которая повторяется с заданной частотой; при создании указывается периодичность, условие завершения и количество создаваемых вперёд экземпляров; `POST /api/tasks` с полем `type: "periodic"`
- [ ] Периодичность: *По рабочим дням* (с учётом рабочего календаря), *Ежедневно* (без учёта рабочего календаря), *Еженедельно*, *Ежемесячно*, *Ежеквартально*, *Ежегодно*
- [ ] Расчёт сроков: для всех типов, кроме «Ежедневно без учёта календаря», длительность и сроки рассчитываются в рабочих днях по производственному календарю; если сроки попадают на выходные/праздники, сдвигаются на ближайшие рабочие дни
- [ ] Завершение серии: *Не завершать* — повторяется до принудительной остановки; *Завершить* — прекратить повторение в указанную дату
- [ ] Количество вперёд создаваемых задач: на следующий день после начала очередного экземпляра создаётся новый, чтобы их количество оставалось постоянным; при значении 0 экземпляры создаются только в день начала
- [ ] Периодические задачи не назначаются на заблокированных пользователей; при разблокировке пользователя назначение возобновляется
- [ ] Просмотр: подраздел «Периодические задачи» → «Текущие» и «Завершённые»; раскрытие списка экземпляров по клику на название
- [ ] Редактирование периодической задачи: доступно автору и его руководителям; `PUT /api/tasks/{id}`
- [ ] Удаление: доступно автору и его руководителям; при удалении выбирается действие для активных экземпляров — «Принудительно завершить» или «Удалить»

##### FR-TASK-01.5.2: Задача по процессу
- [ ] Задача по процессу — создаётся автоматически при достижении узла User Task в BPMN; обозначается специальной иконкой в списке задач; `GET /api/tasks/{id}` содержит ссылку на экземпляр процесса
- [ ] На странице задачи отображается: название процесса с номером версии, название экземпляра (ссылка), дата запуска, инициатор, ответственный; вкладки «Главная страница» (данные для выполнения) и «История»
- [ ] Если исполнитель задачи удалён или заблокирован из оргструктуры, задача перенаправляется ответственному за экземпляр процесса
- [ ] Дополнительные действия для задачи по процессу: *Улучшить* — создать предложение по улучшению процесса (доступно инициатору, участнику или ответственному экземпляра, или пользователям с правом создания запросов на улучшение); *Карта процесса* — просмотр карты (при наличии права на мониторинг); *Скачать вложения* — скачать все файлы архивом
- [ ] При выполнении задачи по процессу — отправка результата в движок процессов (complete User Task token)

### FR-TASK-02: Действия с задачами

#### FR-TASK-02.1: Действия с задачей
- [ ] **Сделано** — завершить работу над задачей; диалог: Трудозатраты (Затрачено, Дата начала), Комментарий, «Скопировать вложения из подзадач» (флажок), Вложения, «Оповестить соисполнителей» (флажок); `PUT /api/tasks/{id}/complete`
- [ ] **Невозможно выполнить** — отметить, если задача не может быть выполнена; аналогичный диалог; задача получает статус «Невозможно выполнить» (с учётом наличия/отсутствия контроля); `PUT /api/tasks/{id}/impossible`
- [ ] **Работа начата** — необязательное действие; фиксирует начало работы; диалог: комментарий, вложения, оповещение соисполнителей; задача получает статус «Выполняется»; `PUT /api/tasks/{id}/start`
- [ ] **Изменить** — редактировать задачу; доступно автору задачи и руководителям автора
- [ ] **Создать подзадачу** — создать подзадачу; текущая задача становится родительской; `POST /api/tasks/{id}/subtasks`
- [ ] **Копия** — создать копию задачи для быстрого создания схожих задач; доступно всем участникам, кроме согласующего; запрещено копировать завершённые задачи и задачи по резолюции документа; `POST /api/tasks/{id}/copy`
- [ ] **Переназначить** — выбрать другого исполнителя; диалог: новый исполнитель, комментарий; доступно автору, исполнителю и контролёру задачи; для задачи по процессу — также согласующему при наличии прав к процессу; `PUT /api/tasks/{id}/reassign`; в активности фиксируется кто, кому и когда переназначил
- [ ] **Перенести срок** — перенести сроки задачи с комментарием; доступно автору и контролёру; `PUT /api/tasks/{id}/reschedule`
- [ ] **Отменить задачу** — нельзя удалить задачу, но можно отменить; диалог: комментарий, вложения, оповещение соисполнителей; статус задачи становится «Закрыто»; `PUT /api/tasks/{id}/cancel`
- [ ] **Отправить на согласование** — отправить задачу на согласование (доступно исполнителю); диалог: согласующий, сроки, комментарий; `POST /api/tasks/{id}/approval`
- [ ] **Добавить комментарий** — добавить комментарий к задаче; `POST /api/tasks/{id}/comments`
- [ ] **Задать вопрос** — задать вопрос по задаче с возможностью прикрепить вложения; получатель вопроса видит задачу с отдельным обозначением; `POST /api/tasks/{id}/questions`
- [ ] **Добавить трудозатраты** — добавить трудозатраты в процессе работы; доступно всем ролям, кроме наблюдателя
- [ ] **Список наблюдателей** — добавить наблюдателей; `POST /api/tasks/{id}/watchers`
- [ ] **Взять на контроль** / **Снять с контроля** — доступно всем ролям; `POST /api/tasks/{id}/control`, `DELETE /api/tasks/{id}/control`
- [ ] **Открыть заново** — возвращает задачу со статусом «Новая» к исполнителю; доступно контролёру; `PUT /api/tasks/{id}/reopen`
- [ ] **Скачать вложения** — скачать все прикреплённые файлы (включая добавленные в комментариях) одним архивом; доступно всем ролям; `GET /api/tasks/{id}/attachments/download`
- [ ] Принятие задачи из очереди роли: если исполнитель — роль, любой участник роли может «взять задачу себе»; `POST /api/tasks/{id}/claim`; после принятия задача пропадает из очереди роли у остальных
- [ ] Упоминания `@user` в описании и комментариях: упомянутый пользователь получает уведомление и автоматически добавляется в наблюдатели

#### FR-TASK-02.2: Списки, фильтры и представления
- [ ] Раздел «Задачи» — подраздел «Список задач» содержит группы:
  - *Входящие* — задачи пользователя, задачи на соисполнении, задачи подчинённых
  - *Исходящие* — задачи от пользователя, от подчинённых
  - *Контроль выполнения* — задачи на контроле (проверка после завершения исполнителем)

  - *Соисполнение* — задачи, в которых пользователь является соисполнителем (отображается при наличии таких задач)
  - *Мой отдел* — текущие, просроченные и выполненные задачи подчинённых; задачи, назначенные подчинёнными другим; архив; сворачивание/раскрытие блока пользователя

- [ ] **Периодические задачи** — отдельный подраздел: страницы «Текущие» и «Завершённые»; отображаются задачи текущего пользователя и (для руководителя) подчинённых
- [ ] **Режимы отображения** списка задач: *Таблица* (строки) и *Таблица-карточка* (строки с расширенной карточкой); каждый режим настраивается независимо; `GET /api/tasks`
- [ ] **Цветовое кодирование** задач в списке: непрочитанные — синий шрифт + синяя полоса слева (у высокоприоритетных после прочтения полоса становится красной); в работе («Работа начата») — зелёный; невозможно выполнить — оранжевый; просроченные — красный; выполненные — зелёный зачёркнутый

- [ ] Сортировка: по столбцу (в режиме «Таблица» клик по заголовку); по сохранённым фильтрам
- [ ] **Расширенный фильтр**: по исполнителю, статусу, приоритету, типу, категории, дате создания (диапазон, в т.ч. относительные даты), сроку завершения (диапазон), дате выполнения, авторам, соисполнителям, родительской задаче, признаку «Задача по процессу» (с дополнительными параметрами: процесс, состояние, название экземпляра, инициатор, ответственный, дата запуска, роль в процессе); фильтры сохраняются как именованные фильтры
- [ ] EQL-поиск: поиск задач с помощью EQL-запроса в форме расширенного поиска; `GET /api/tasks?eql=...`
- [ ] `GET /api/tasks` — список задач с параметрами фильтрации, сортировки и пагинации (cursor-based); поле `fields` для выбора возвращаемых полей
- [ ] Группировка в таблице: по исполнителю / приоритету / статусу / категории / сроку (сегодня / эта неделя / позже / просрочено)
- [ ] Экспорт списка задач в Excel / CSV: `GET /api/tasks/export`
- [ ] Счётчики задач в боковом меню: «Мои задачи (N)», «Просроченные (N)», «На согласовании (N)» — обновляются в реальном времени (SSE / WebSocket)

#### FR-TASK-02.3: Поиск, подписка, уведомления и календарь
- [ ] **Быстрый поиск**: строка поиска в разделе «Задачи» — ввод названия или его части + фильтр по автору; поиск в пределах выбранной группы задач; `GET /api/tasks?q=...`

- [ ] Наблюдатель получает оповещения обо всех действиях с задачей; задача может не отображаться в его списке задач, но открывается по ссылке из оповещения
- [ ] Настройки уведомлений пользователя: выбор событий и каналов (in-app / email / push) для каждого типа события задачи; `PUT /api/users/me/notification-settings`
- [ ] Напоминания о сроке: за N дней/часов (настраивается пользователем) — уведомление исполнителю и наблюдателям; `POST /api/tasks/{id}/reminders`
- [ ] **Планирование задачи в календаре** — два способа:
  - Из списка задач: нажать иконку «незапланировано» у задачи → открывается окно календаря → выбрать день и время → подтвердить; иконка меняется на «запланировано»
  - Из раздела «Календарь»: кнопка открывает панель планирования → перетащить задачу из списка «Незапланированные» на нужный слот → задача перемещается в список «Запланированные»; поиск по названию на панели; отмена — удалить задачу из календаря; `POST /api/tasks/{id}/schedule`
- [ ] Интеграция с внешними календарями: экспорт задач пользователя в формате iCalendar (`GET /api/users/me/tasks.ics`); подписка внешнего клиента (Google Calendar, Outlook) на этот URL
- [ ] **Дашборд задач**: виджеты — мои открытые задачи по приоритету, динамика создания/закрытия за 30 дней, задачи по статусам; `GET /api/dashboard/tasks`

- [ ] **Корзина**: удалённые документы помещаются в корзину; восстановление `POST /api/documents/{id}/restore`; автоочистка через N дней (настройка администратора); `GET /api/trash`, `DELETE /api/trash` (очистить корзину)

#### FR-DOC-01.2: Права доступа к папкам и документам
- [ ] Модель прав: Просмотр / Загрузка (добавление файлов) / Редактирование метаданных / Управление (создание подпапок, изменение прав) / Полный доступ; права задаются для пользователя, группы, роли или подразделения; `GET/PUT /api/folders/{id}/permissions`
- [ ] Наследование прав: вложенная папка наследует права родителя по умолчанию; возможность переопределить права на любом уровне (флаг «не наследовать»)
- [ ] Права на документ могут отличаться от прав папки: `GET/PUT /api/documents/{id}/permissions`
- [ ] Проверка прав на уровне API (401/403) и на уровне UI (скрытие/блокировка элементов)
- [ ] Аудит доступа: каждый факт просмотра, скачивания, редактирования документа записывается в лог; `GET /api/documents/{id}/access-log`

#### FR-DOC-01.3: Групповые операции с документами
- [ ] Выбор нескольких документов в папке (чекбоксы + Shift/Ctrl-клик); отображение панели групповых действий
- [ ] Групповые действия: переместить в папку, копировать в папку, скачать архивом (ZIP), удалить, изменить права, добавить теги, сменить статус, запустить процесс
- [ ] `POST /api/documents/bulk` — тело запроса: массив `ids` + `action` + дополнительные параметры действия
- [ ] Прогресс-бар при длительных операциях (массовое скачивание, конвертация) через SSE: `GET /api/bulk-operations/{operationId}/progress`

### FR-DOC-02: Электронные документы

#### FR-DOC-02.1: Типы документов и карточка
- [ ] Справочник типов документов (администратор): имя, иконка, набор метаданных-полей (текст, число, дата, справочник, пользователь, логический флаг), обязательность, порядок отображения; `GET/POST/PUT/DELETE /api/admin/document-types`
- [ ] Создание документа: выбор типа → заполнение метаданных → загрузка файла (опционально — для веб-документа файл не нужен); `POST /api/documents`
- [ ] **Карточка документа** — вкладки:
  - *Свойства*: все поля метаданных типа + системные (автор, дата создания, дата изменения, папка, статус, размер)
  - *Файлы*: список загруженных файлов со статусами (актуальный / заархивированный), кнопки «Скачать», «Просмотр», «Чекин», «Чекаут»
  - *Версии*: история версий каждого файла — номер, автор, дата, размер, комментарий к версии; сравнение двух версий (diff для текстовых и Office); `GET /api/documents/{id}/files/{fileId}/versions`
  - *Связи*: список связанных объектов (документ, задача, процесс) с типом связи (ссылается на / версия / приложение к / заменяет)
  - *История*: лог всех изменений — метаданные, права, загрузка файлов, смена статуса; `GET /api/documents/{id}/history`
  - *Комментарии*: rich-text + вложения + `@упоминания`; `GET/POST /api/documents/{id}/comments`
- [ ] Статусы документа: Черновик → На согласовании → Действующий → Архив → Аннулирован; переходы статусов настраиваются для каждого типа документа; `PUT /api/documents/{id}/status`
- [ ] Маршрут согласования документа (аналогичен согласованию задачи): последовательный / параллельный / смешанный; действия: Согласовать / Отклонить / Подписать (с ЭЦП); `GET/POST /api/documents/{id}/approval`

#### FR-DOC-02.2: Версионирование файлов
- [ ] **Чекаут** (Check Out): блокировка файла для редактирования; другие пользователи видят «заблокировано пользователем X»; скачивание актуальной версии для редактирования; `POST /api/documents/{id}/files/{fileId}/checkout`
- [ ] **Чекин** (Check In): загрузка отредактированной версии; поле «Комментарий к версии» (обязательно/опционально); автоматическое снятие блокировки; инкремент номера версии (major: 1.0 → 2.0, minor: 1.0 → 1.1 — выбор при чекине); `POST /api/documents/{id}/files/{fileId}/checkin`
- [ ] Отмена чекаута без сохранения изменений; `DELETE /api/documents/{id}/files/{fileId}/checkout`
- [ ] Принудительное снятие блокировки администратором
- [ ] Восстановление предыдущей версии как актуальной; `POST /api/documents/{id}/files/{fileId}/versions/{versionId}/restore`
- [ ] Сравнение версий: для Office-документов (через LibreOffice diff) и текстовых файлов — inline diff; `GET /api/documents/{id}/files/{fileId}/versions/diff?v1=...&v2=...`
- [ ] Автоматическое версионирование при загрузке нового файла через «Заменить файл» без явного чекаута (light versioning)

#### FR-DOC-02.3: Предпросмотр, конвертация и веб-документ
- [ ] **Предпросмотр в браузере**: PDF — нативный просмотрщик; Office (DOCX, XLSX, PPTX) — конвертация на лету в PDF через LibreOffice, результат кешируется; изображения (JPG/PNG/GIF/WebP/SVG) — прямой показ; видео — HTML5-плеер; иные форматы — кнопка «Скачать»; `GET /api/documents/{id}/files/{fileId}/preview`
- [ ] **Конвертация форматов**: фоновая задача конвертации в PDF / PDF/A (архивный); DOCX → PDF, XLS → PDF, PPTX → PDF; результат сохраняется как дополнительный файл карточки; `POST /api/documents/{id}/files/{fileId}/convert`; статус конвертации через SSE
- [ ] **Веб-документ** (тип документа без файла): встроенный rich-text редактор (на базе TipTap / ProseMirror); поддержка таблиц, вложенных изображений, кодовых блоков, `@упоминаний`; автосохранение каждые 30 с; история версий веб-содержимого (аналогично файловому версионированию); экспорт в DOCX / PDF по запросу; `GET/PUT /api/documents/{id}/web-content`
- [ ] **Генерация документа из шаблона**: шаблон DOCX с плейсхолдерами `{{переменная}}`; при запуске генерации передаётся набор переменных (вручную или из переменных процесса); результат — новый файл версии документа; `POST /api/documents/{id}/generate`

#### FR-DOC-02.4: Подписка, архив и интеграция с процессами
- [ ] Подписка на изменения документа: уведомления при смене статуса, новой версии файла, изменении метаданных, новом комментарии; `POST /api/documents/{id}/watch`; отписка `DELETE /api/documents/{id}/watch`
- [ ] **Архив документов**: документ переводится в статус «Архив» вручную или автоматически (по правилу — через N дней после смены статуса на «Действующий» или по дате в поле «Срок хранения»); архивные документы отображаются отдельно; `PUT /api/documents/{id}/status` + фоновый job архивации
- [ ] **Аннулирование**: финальный статус; аннулированный документ доступен только для чтения; причина аннулирования — обязательный комментарий
- [ ] Срок хранения документа: поле «Хранить до» на карточке; фоновый job уведомляет ответственного за N дней до истечения; `GET /api/documents/expiring?days=30`
- [ ] **Интеграция с процессами**:
  - Service Task в BPMN может создавать документ, менять его статус, менять права, генерировать файл из шаблона — через соответствующие API вызовы движка
  - Изменение прав доступа к документу из процесса: `PUT /api/documents/{id}/permissions` вызывается из Service Task с параметрами из переменных процесса
  - Результаты согласования документа могут быть переданы в процесс как переменные (approved/rejected + комментарий)

### FR-DOC-03: Поиск документов

#### FR-DOC-03.1: Полнотекстовый и метаданных поиск
- [ ] **Глобальный поиск**: полнотекстовый поиск по названию документа, полям метаданных и содержимому файлов (PDF, Office — через Apache Tika / Elasticsearch); тип объекта «Документ» в глобальной строке поиска; `GET /api/search?q=...&type=document`
- [ ] Поиск по содержимому файла: при загрузке файла запускается фоновая задача индексации текста; статус индексации видим на карточке (не проиндексирован / индексируется / проиндексирован)
- [ ] **Расширенный поиск документов**: `GET /api/documents/search` — параметры:
  - `q` — полнотекстовый запрос
  - `type` — тип документа (из справочника)
  - `status` — статус (один или список)

- [ ] Сохранённые поисковые запросы: пользователь может сохранить набор фильтров как «Сохранённый поиск» и вернуться к нему; `GET/POST/DELETE /api/users/me/saved-searches`
- [ ] Результаты поиска: список с релевантностью; подсветка совпадений в названии и сниппете содержимого; сортировка по релевантности / дате изменения / дате создания / названию
- [ ] Экспорт результатов поиска в Excel / CSV; `GET /api/documents/search/export`
- [ ] **Теги документов**: free-form теги с автодополнением; массовое добавление/удаление тегов; фильтрация по тегам в поиске и в папке; `GET /api/tags?type=document`

#### FR-TIME-01.3: Отчёты и аналитика по трудозатратам

## 8. Календарь

### FR-CAL-01: Управление событиями

#### FR-CAL-01.1: Создание и редактирование событий
- [ ] Создание события: название (обязательно), описание (rich-text), дата/время начала и окончания, флаг «весь день», место (свободный текст + опциональные координаты), онлайн-ссылка (Zoom/Teams/Meet URL); `POST /api/calendar/events`; `PUT /api/calendar/events/{id}`; `DELETE /api/calendar/events/{id}`
- [ ] **Участники события**: приглашение конкретных пользователей и групп; статус ответа: Принято / Отклонено / Под вопросом / Не ответил; `POST /api/calendar/events/{id}/invites`; `PUT /api/calendar/events/{id}/rsvp` (пользователь меняет свой статус); организатор видит сводку ответов
- [ ] Уведомление участникам при создании/изменении/отмене события (in-app + email); при отмене — уведомление с пометкой «Событие отменено»
- [ ] **Вложения** к событию: файлы, ссылки; `POST /api/calendar/events/{id}/attachments`
- [ ] **Повестка события** (agenda): список пунктов повестки с докладчиком и продолжительностью; `GET/PUT /api/calendar/events/{id}/agenda`
- [ ] **Протокол события**: rich-text документ, создаётся прямо в карточке события или генерируется из шаблона; привязка задач, возникших на встрече; `GET/PUT /api/calendar/events/{id}/minutes`
- [ ] Переносить событие перетаскиванием на другой слот в сетке календаря (drag-and-drop); `PATCH /api/calendar/events/{id}` с новыми датами

#### FR-CAL-01.2: Периодические события
- [ ] Правила повторения (RRULE-совместимые): ежедневно / каждые N дней / еженедельно (дни недели) / ежемесячно (N-е число / Nth weekday) / ежегодно; конец серии: по дате / после N повторений / бесконечно; `POST /api/calendar/events` с полем `recurrence`
- [ ] Редактирование одного экземпляра / всей серии / этого и последующих (instance / all / following); при удалении — те же три варианта
- [ ] Исключения в серии (exdates): конкретные даты, в которые событие не происходит
- [ ] Отображение периодических событий в сетке: все экземпляры разворачиваются в заданном диапазоне отображения

#### FR-CAL-01.3: Виды отображения и общие календари
- [ ] **Виды**: День / Рабочая неделя (пн–пт) / Неделя (пн–вс) / Месяц / Повестка (список событий); переключение кнопками в шапке; `GET /api/calendar/events?from=...&to=...&calendarIds[]=...`
- [ ] **Мои календари**: личный календарь пользователя; календарь задач (события из задач с датами); календарь процессов (дедлайны экземпляров процессов); каждый отключаемый/включаемый через чекбокс в боковой панели
- [ ] **Общие / командные календари**: создаются администратором или пользователем; можно подписаться (`POST /api/calendar/subscriptions`); права: только просмотр / просмотр + добавление событий / управление; `GET/POST/PUT/DELETE /api/admin/shared-calendars`
- [ ] **Просмотр чужого календаря**: пользователь может открыть сетку коллеги (если есть права) для подбора времени встречи; режим «Совместное расписание» (overlay нескольких пользователей/ресурсов)
- [ ] **Ресурсные календари**: переговорные комнаты, оборудование; бронирование ресурса при создании события; `GET /api/resources/availability?from=...&to=...`; `POST /api/calendar/events/{id}/resources`
- [ ] **Напоминания**: настраивается на каждом событии — за N минут / часов / дней; несколько напоминаний на одном событии; каналы: in-app / email / push; `PUT /api/calendar/events/{id}/reminders`
- [ ] **Мини-календарь** (пикер дат) в боковой панели для быстрой навигации; точки-индикаторы на датах с событиями

### FR-CAL-02: Интеграция с внешними календарями

#### FR-CAL-02.1: Экспорт и двусторонняя синхронизация
- [ ] **iCalendar-фид**: `GET /api/users/me/calendar.ics?token=<personalToken>` — постоянный URL, совместимый с RFC 5545; содержит события пользователя (выбранные календари); внешний клиент (Google Calendar, Outlook, Apple Calendar) может подписаться по этому URL и получать обновления; токен генерируется в настройках профиля, может быть отозван; `POST /api/users/me/calendar-token` (генерация), `DELETE /api/users/me/calendar-token` (отзыв)
- [ ] **Экспорт диапазона в .ics файл**: `GET /api/calendar/export.ics?from=...&to=...&calendarIds[]=...` — скачивание статичного файла для однократного импорта
- [ ] **Двусторонняя синхронизация с Exchange / Microsoft 365**: подключение учётной записи пользователя через OAuth2 / EWS; синхронизация событий Exchange → BPM и BPM → Exchange; интервал синхронизации: 5 мин (настраивается); конфликты: приоритет последнего изменения (last-write-wins); `POST /api/integrations/exchange/connect`; статус синхронизации `GET /api/integrations/exchange/status`
- [ ] **Двусторонняя синхронизация с Google Calendar**: подключение через Google OAuth2 + Google Calendar API v3; аналогичная логика синхронизации; `POST /api/integrations/google-calendar/connect`; `GET /api/integrations/google-calendar/status`

- [ ] Журнал отправленных уведомлений (успех / ошибка / пропущено из-за DND): `GET /api/admin/notification-logs?userId=...&type=...`

#### FR-REF-01.2: Иерархические справочники и записи
- [ ] **Иерархия (дерево)**: тип объекта можно пометить как иерархический — у каждой записи появляется родительская ссылка `parentId`; `GET /api/objects/{typeCode}?tree=true` возвращает дерево; операции перемещения узлов `PATCH /api/objects/{typeCode}/{id}/parent`
- [ ] **CRUD записей**: `GET /api/objects/{typeCode}`; `GET /api/objects/{typeCode}/{id}`; `POST /api/objects/{typeCode}`; `PUT /api/objects/{typeCode}/{id}`; `DELETE /api/objects/{typeCode}/{id}`; soft-delete с флагом `isArchived`
- [ ] **Фильтры и представления**: пользователь сохраняет набор фильтров как «Представление»; представления бывают личные и публичные; `GET/POST/PUT/DELETE /api/objects/{typeCode}/views`; поддерживаемые операторы фильтра: равно / не равно / содержит / начинается с / больше / меньше / входит в список / пусто
- [ ] **Массовые операции**: `POST /api/objects/{typeCode}/bulk` — смена значения поля, архивирование, удаление по выборке
- [ ] **Поиск по справочнику**: полнотекстовый поиск по строковым полям; `GET /api/objects/{typeCode}?search=...`
- [ ] **Версионирование записей**: опционально для типа; при каждом изменении сохраняется снимок значений; `GET /api/objects/{typeCode}/{id}/history`

#### FR-REF-01.3: Связи между объектами
- [ ] **Типы связей**: администратор создаёт именованные типы связей между двумя типами объектов (Поставщик → Договор, Сотрудник → Проект и т.д.); кратность: один-к-одному / один-ко-многим / многие-ко-многим; `GET/POST/PUT/DELETE /api/admin/relation-types`
- [ ] **Добавление/удаление связей** на карточке объекта: `POST /api/objects/{typeCode}/{id}/relations`; `DELETE /api/objects/{typeCode}/{id}/relations/{relationId}`; на карточке — раздел «Связанные объекты» с фильтрацией по типу связи
- [ ] **Наследование типов**: дочерний тип объекта наследует все свойства родительского; дополнительные свойства добавляются только у дочернего; `PUT /api/admin/object-types/{id}/parent`

#### FR-REF-01.4: Запуск процесса из записи справочника
- [ ] На карточке записи объекта — кнопка «Запустить процесс»; список доступных процессов (тех, у которых настроена привязка к данному типу объекта); `GET /api/processes?objectType={typeCode}`; при запуске контекст экземпляра процесса автоматически получает ссылку на запись объекта
- [ ] **Журнал процессов** записи: список всех запущенных и завершённых экземпляров процессов, связанных с данной записью; `GET /api/objects/{typeCode}/{id}/process-instances`

### FR-REF-02: Перечисления

#### FR-REF-02.1: Управление перечислениями
- [ ] **Создание перечисления**: название, код, описание; `POST /api/admin/enumerations`; `PUT /api/admin/enumerations/{id}`; `DELETE /api/admin/enumerations/{id}` (запрещено если используется в свойствах объектов)
- [ ] **Значения перечисления**: текстовое название, код (уникальный в рамках перечисления), цвет (hex), иконка, порядок; признак «активно» (деактивированные значения не предлагаются при выборе, но старые записи с ними отображаются); `GET/POST/PUT/DELETE /api/admin/enumerations/{id}/values`
- [ ] **Группы объектов и перечислений**: логическая группировка типов объектов и перечислений по модулям (Сотрудники / CRM / Финансы / …); `GET/POST/PUT/DELETE /api/admin/object-groups`; в сайдбаре администрирования справочники отображаются сгруппированно
- [ ] **Использование** перечисления: поле типа «Список» у объекта ссылается на перечисление; при выборе значения отображается цвет/иконка; в фильтрах поддерживается «входит в список»
- [ ] **Импорт значений** перечисления из CSV; `POST /api/admin/enumerations/{id}/import`

## 12. Бизнес-правила

### FR-BR-01: Управление бизнес-правилами

#### FR-BR-01.1: Конструктор правил
- [ ] **Создание правила**: название, код, описание, категория; `POST /api/business-rules`; `PUT /api/business-rules/{id}`; `DELETE /api/business-rules/{id}` (только если не используется в активных процессах)
- [ ] **Условия (if-блок)**: визуальный конструктор условий — набор условных выражений с операторами AND / OR; каждое условие: поле (переменная контекста / глобальная переменная / атрибут объекта) + оператор (равно / не равно / больше / меньше / содержит / входит в список / пусто / не пусто / начинается с / заканчивается на) + значение (константа / переменная / выражение); вложенные группы условий (AND внутри OR и наоборот); `GET/PUT /api/business-rules/{id}/conditions`
- [ ] **Действия (then-блок)**: список действий, выполняемых при истинности условий; типы действий: установить значение переменной, вызвать сервис/интеграцию, отправить уведомление, запустить дочерний процесс, выбросить ошибку, записать в лог; `GET/PUT /api/business-rules/{id}/actions`
- [ ] **Else-блок** (опционально): альтернативные действия, выполняемые если условие ложно
- [ ] **Таблицы решений (Decision Table)**: матричный вид правила — строки = наборы условий, колонки = результирующие действия; режимы: «первое совпадение» / «все совпадения»; `GET/PUT /api/business-rules/{id}/decision-table`
- [ ] **Формульный редактор**: в условиях и действиях доступен ввод выражений (аналог Excel-формул) с функциями: строковые (concat, trim, substring), числовые (round, abs, min, max), даты (addDays, diffDays, now()), логические (if, and, or, not), коллекции (count, sum, filter, map)

#### FR-BR-01.2: Жизненный цикл правила
- [ ] **Версионирование**: при каждом сохранении создаётся новая версия (major.minor); активная версия помечается явно; `GET /api/business-rules/{id}/versions`; `POST /api/business-rules/{id}/versions/{v}/activate`
- [ ] **Согласование версии**: перевод версии в статус «На согласовании» → «Согласовано» / «Отклонено»; настраиваемая цепочка согласующих; `POST /api/business-rules/{id}/versions/{v}/submit`; `POST /api/business-rules/{id}/versions/{v}/approve`; `POST /api/business-rules/{id}/versions/{v}/reject`
- [ ] **Карточка правила**: отображает метаданные, текущую активную версию, список версий с историей изменений (кто изменил, когда, что изменилось — diff); `GET /api/business-rules/{id}`
- [ ] **История изменений** (audit log): каждое изменение правила логируется (пользователь, дата, тип изменения); `GET /api/business-rules/{id}/history`
- [ ] **Тестирование правила**: ввод тестовых входных данных (JSON) → получение результата выполнения (выходные значения + трассировка какие условия сработали); `POST /api/business-rules/{id}/test`

#### FR-BR-01.3: Исполнение правил
- [ ] **Задача исполнения правила** в BPMN-процессе: Service Task с типом «Бизнес-правило»; привязка к конкретному правилу и версии (или «активная»); входные/выходные маппинги переменных процесса
- [ ] **Ручной запуск** (ad-hoc): запуск правила с произвольными входными данными через UI-форму; `POST /api/business-rules/{id}/execute`
- [ ] **Авто-запуск по событию**: триггеры — создание/изменение объекта справочника, наступление даты, webhook; конфигурируется в настройках правила; `GET/POST/PUT/DELETE /api/business-rules/{id}/triggers`
- [ ] **Журнал исполнений**: лог каждого вызова правила (дата, инициатор, входные данные, результат, время выполнения, ошибки); `GET /api/business-rules/{id}/executions`

#### FR-BR-01.4: Глобальные переменные
- [ ] **Создание глобальной переменной**: имя, код, тип данных (строка / число / дата / Да/Нет / JSON), начальное значение, описание; `GET/POST/PUT/DELETE /api/global-variables`
- [ ] **Использование** в правилах и BPMN-процессах: переменная доступна в формулах и условиях через синтаксис `$varCode`; изменение значения из действия правила или сервисной задачи
- [ ] **Права доступа к переменным**: какие роли/правила могут читать / изменять конкретную переменную; `PUT /api/global-variables/{id}/permissions`
- [ ] **Аудит изменений**: лог каждого изменения значения переменной (кто, когда, старое/новое значение); `GET /api/global-variables/{id}/history`

- [ ] **Таблица данных (Data Table)**: переиспользуемый компонент таблицы с пагинацией (cursor-based), сортировкой по колонкам, поиском, inline-фильтрами по колонкам, настройкой видимых колонок, выбором строк (чекбоксы), экспортом в Excel/CSV; применяется в списках задач, документов, пользователей, записей справочников

- [ ] **Панель истории действий (History Panel)**: унифицированный компонент хронологического лога изменений объекта — аватар актора, тип действия (создан / изменён / статус изменён / комментарий / вложение добавлено), изменённые поля с указанием старого и нового значения, временна́я метка; подгрузка по «Показать ещё» (cursor-based пагинация); применяется на всех карточках объектов

## 14. Интеграции

### FR-INT-01: Модули интеграции

#### FR-INT-01.1: Конструктор модулей
- [ ] **Создание модуля интеграции**: название, тип (REST / SOAP / gRPC / GraphQL), базовый URL, описание; `POST /api/integrations/modules`; `PUT /api/integrations/modules/{id}`; `DELETE /api/integrations/modules/{id}`
- [ ] **Аутентификация модуля**: без аутентификации / Basic (login+password) / API Key (header/query-param) / Bearer Token (статичный) / OAuth2 Client Credentials (автоматическое обновление токена) / OAuth2 Authorization Code / mTLS (клиентский сертификат); `PUT /api/integrations/modules/{id}/auth`
- [ ] **Методы модуля (Operations)**: HTTP-метод + путь + параметры (path / query / header / body); маппинг входных и выходных параметров на DTO; `GET/POST/PUT/DELETE /api/integrations/modules/{id}/operations`
- [ ] **SOAP-модуль**: импорт WSDL по URL → автоматическая генерация операций и DTO; `POST /api/integrations/modules/import-wsdl`
- [ ] **Тестирование операции**: ввод входных параметров → вызов → просмотр ответа (статус, заголовки, тело); `POST /api/integrations/modules/{id}/operations/{opId}/test`
- [ ] **Версионирование модуля**: каждый модуль имеет версии; процессы/правила ссылаются на конкретную версию или «стабильную»; `POST /api/integrations/modules/{id}/versions/{v}/publish`
- [ ] **Публикация и использование в процессах**: опубликованный модуль доступен как Service Task в BPMN-редакторе; поле выбора модуля + операции + маппинг переменных

#### FR-INT-01.2: Структуры обмена данными (DTO)
- [ ] **Глобальные DTO**: описание структуры данных (имя, поля с типами) доступно для повторного использования в разных модулях и правилах; `GET/POST/PUT/DELETE /api/integrations/dto`
- [ ] **Внутренние DTO модуля**: структуры, видимые только внутри конкретного модуля
- [ ] **Генерация DTO из JSON-схемы** (JSON Schema / OpenAPI 3.0 `#/components/schemas`): `POST /api/integrations/dto/import-schema`
- [ ] **Маппинг полей**: визуальный маппер (дерево источника → дерево назначения), поддержка трансформаций (выражения на каждом маппинге)

### FR-INT-02: Очереди сообщений

#### FR-INT-02.1: Управление очередями
- [ ] **Создание типа сообщения**: имя, код, DTO (схема payload); `POST /api/messaging/message-types`; `PUT /api/messaging/message-types/{id}`
- [ ] **Встроенная очередь**: каждый тип сообщения имеет очередь FIFO; сообщения хранятся в БД; `POST /api/messaging/queues/{typeCode}/publish` — отправить сообщение в очередь; `GET /api/messaging/queues/{typeCode}/messages` — просмотр накопившихся сообщений
- [ ] **Отправка из процесса**: BPMN Send Task с типом «Сообщение» → выбор типа сообщения + заполнение payload из переменных процесса
- [ ] **Ожидание сообщения**: BPMN Receive Task — процесс приостанавливается до поступления сообщения заданного типа (с опциональной корреляцией по ключу — например `correlationKey = orderId`); при получении — продолжение процесса с payload сообщения как переменными
- [ ] **Корреляция сообщений**: при публикации сообщения указывается корреляционный ключ; брокер сопоставляет с ожидающими экземплярами процессов; `POST /api/messaging/queues/{typeCode}/publish` с полем `correlationKey`
- [ ] **Dead Letter Queue**: сообщения, которые не удалось доставить после N попыток, перемещаются в DLQ; `GET /api/messaging/dlq`; ручная повторная отправка: `POST /api/messaging/dlq/{id}/retry`
- [ ] **Мониторинг очередей**: глубина очереди, среднее время ожидания, количество ошибок; `GET /api/messaging/stats`

### FR-INT-03: Внешние источники данных

#### FR-INT-03.1: Подключение к внешним БД
- [ ] **Источник данных (Data Source)**: тип (PostgreSQL / MS SQL / Oracle / MySQL / MongoDB), хост, порт, имя базы, логин, пароль (хранится зашифрованным); проверка соединения; `POST /api/integrations/datasources`; `PUT /api/integrations/datasources/{id}`; `POST /api/integrations/datasources/{id}/test`
- [ ] **Запросы (Queries)**: именованный SQL/MQL-запрос с параметрами; `GET/POST/PUT/DELETE /api/integrations/datasources/{id}/queries`; ввод тестовых параметров → выполнение → просмотр результата в виде таблицы
- [ ] **Использование в процессах**: Service Task «Внешний источник» → выбор источника + запроса + маппинг параметров из переменных процесса + маппинг результата в переменные процесса
- [ ] **Использование в формах**: виджет Lookup/Table с источником = внешний запрос; загрузка данных при открытии формы или по событию смены значения другого поля
- [ ] **Права доступа**: конфигурация, какие роли/процессы могут использовать конкретный источник; `PUT /api/integrations/datasources/{id}/permissions`

#### FR-INT-03.2: Входящие вебхуки
- [ ] **Создание вебхука**: название, уникальный URL (`POST /api/webhooks/{token}`), ожидаемый формат payload (схема), секрет для проверки подписи HMAC; `POST /api/integrations/webhooks`; `PUT /api/integrations/webhooks/{id}`
- [ ] При поступлении HTTP-запроса на URL вебхука — публикация сообщения в очередь или запуск процесса (настраивается); `GET /api/integrations/webhooks/{id}/logs` — лог входящих запросов

### FR-INT-04: RPA

#### FR-INT-04.1: Интеграция с RPA-платформой
- [ ] **Настройка RPA-агента**: подключение к платформе (UiPath / Automation Anywhere / BluePrism / произвольный REST API); адрес оркестратора, токен, список доступных роботов; `POST /api/integrations/rpa/agents`; `PUT /api/integrations/rpa/agents/{id}`
- [ ] **Запуск робота из BPMN-узла**: Service Task типа «RPA» → выбор агента + имени процесса робота + входные параметры из переменных процесса; ожидание завершения (синхронный режим) или продолжение без ожидания (асинхронный)
- [ ] **Получение результата**: маппинг выходных данных завершённого задания робота в переменные процесса
- [ ] **Мониторинг заданий RPA**: `GET /api/integrations/rpa/jobs` — статус заданий (Очередь / Выполняется / Завершено / Ошибка); детали задания: лог, выходные данные, время выполнения
- [ ] **Обработка ошибок**: при сбое задания RPA — настраиваемая стратегия: повтор N раз / граница ошибки BPMN / уведомление ответственному

### FR-RPT-02: Скоринг

#### FR-RPT-02.1: Настройка скоринговых правил
- [ ] **Скоринговая модель**: название, тип объекта, к которому применяется (контрагент / сделка / заявка / произвольный тип из FR-REF); шкала (мин. и макс. балл); `POST /api/scoring/models`; `PUT /api/scoring/models/{id}`; `DELETE /api/scoring/models/{id}`
- [ ] **Критерии (факторы)**: каждый критерий = поле объекта + шкала значений → балл; типы шкал: числовой диапазон (от X до Y → N баллов), точное значение, перечисление (каждое значение enum → свой балл), формула (вычисляемый балл); `GET/POST/PUT/DELETE /api/scoring/models/{id}/criteria`
- [ ] **Веса критериев**: числовой коэффициент (0–100); итоговый балл = сумма (балл × вес) / сумма весов × масштаб шкалы; `PUT /api/scoring/models/{id}/criteria/{criterionId}/weight`
- [ ] **Группировка критериев**: критерии объединяются в разделы (финансовые / репутационные / …); раздел имеет общий вес; `GET/POST/PUT/DELETE /api/scoring/models/{id}/groups`
- [ ] **Версионирование модели**: история версий; активация версии; `GET /api/scoring/models/{id}/versions`; `POST /api/scoring/models/{id}/versions/{v}/activate`
- [ ] **Тестовый расчёт**: ввод значений полей объекта → расчёт итогового балла с разбивкой по критериям; `POST /api/scoring/models/{id}/test`

#### FR-RPT-02.2: Панель скоринга в карточках объектов
- [ ] **Виджет скоринга**: на карточке объекта (FR-REF-01) отображается итоговый балл по всем применимым скоринговым моделям; индикатор цвета (зелёный / жёлтый / красный) по порогам, настроенным в модели; `GET /api/objects/{typeCode}/{id}/scoring`
- [ ] **Детализация**: раскрываемая панель с разбивкой балла по разделам и критериям; значение каждого поля на момент расчёта
- [ ] **История скоринга**: график изменения итогового балла во времени; `GET /api/objects/{typeCode}/{id}/scoring/history`
- [ ] **Пересчёт по запросу**: кнопка «Пересчитать» — принудительный пересчёт баллов; `POST /api/objects/{typeCode}/{id}/scoring/recalculate`
- [ ] **Авто-пересчёт**: при изменении значения любого поля, участвующего в критериях, балл пересчитывается автоматически (асинхронно)

#### FR-RPT-02.3: Глобальные настройки скоринга
- [ ] **Привязка моделей к типам объектов**: администратор назначает, какие модели активны для каждого типа объекта; `PUT /api/admin/scoring/object-type-mappings`
- [ ] **Пороги сигнализации**: настройка диапазонов «высокий / средний / низкий риск» с цветами для каждой модели; `PUT /api/scoring/models/{id}/thresholds`
- [ ] **Сводный скоринг-дашборд**: виджет для главной страницы — топ-N объектов с наихудшим баллом; распределение объектов по зонам риска (круговая диаграмма); `GET /api/scoring/dashboard?modelId=&typeCode=`
- [ ] **Права на модели**: роли, которым разрешено просматривать / редактировать конкретную модель; `PUT /api/scoring/models/{id}/permissions`

## 17. Нефункциональные требования

- [ ] **Производительность**: отклик API ≤ 300 мс для 95% запросов при нагрузке 500 одновременных пользователей
- [ ] **Масштабируемость**: горизонтальное масштабирование backend (stateless-сервисы + внешний session store)
- [ ] **Безопасность**: HTTPS, хранение паролей (bcrypt/PBKDF2), JWT с коротким TTL + refresh, CSRF-защита, rate limiting
- [ ] **Аудит**: лог всех значимых действий пользователей (кто, что, когда, объект)
- [ ] **Интернационализация**: поддержка i18n (минимум: русский, английский)
- [ ] **Доступность**: SPA на React, поддержка современных браузеров (Chrome, Edge, Firefox)

| **PostgreSQL** | Мощный, JSON/JSONB, полнотекстовый поиск, open source, отличная поддержка в .NET (EF Core + Npgsql) | Требует настройки для больших нагрузок | **Рекомендуется как основной вариант** — универсален, покрывает 95% задач |
| **MS SQL Server** | Зрелый корпоративный продукт, первоклассная интеграция с .NET/EF Core, отличные инструменты мониторинга | Платный (Enterprise), привязка к экосистеме MS | Если заказчик уже в MS-стеке или есть лицензии |
| **Oracle Database** | Корпоративные возможности, высокая надёжность | Дорогой, сложный в управлении | Крупные госкорпорации с существующей лицензией Oracle |

### Дополнительные хранилища (рекомендуется независимо от основной СУБД)

- [ ] **Redis** — кеш сессий, очереди задач (Redis Streams / MassTransit), distributed lock
- [ ] **Elasticsearch / OpenSearch** — полнотекстовый поиск по документам и объектам (EQL-поиск)
- [ ] **MinIO / Azure Blob / S3** — хранение файлов и вложений (вне БД)

> **Итоговая рекомендация**: PostgreSQL + Redis + Elasticsearch + S3-совместимое хранилище.  
> Полностью open source, хорошо интегрируется с .NET Core, покрывает все сценарии системы.
