# Функциональные требования: BPM-система

**Стек**: Backend — C# ASP.NET Core, Frontend — React  
**Архитектура**: REST API + SignalR для real-time, модульная монолитная архитектура (возможен переход на микросервисы)

---

## 1. Аутентификация и авторизация

### FR-AUTH-01: Вход в систему

#### FR-AUTH-01.1: Вход по логину и паролю (JWT-токены, refresh token)
- [x] Форма входа: поля «логин» (email или username) и «пароль», кнопка «Войти»
- [ ] Клиентская валидация полей (не пусто, минимальная длина, формат email)
- [x] `POST /api/auth/login` — возвращает access token (JWT) + refresh token
- [x] Access token: короткое время жизни (настраивается, по умолчанию 15–60 мин), хранится в памяти / sessionStorage на фронтенде
- [x] Refresh token: длинное время жизни (настраивается, по умолчанию 7–30 дней), передаётся в HttpOnly-cookie
- [x] `POST /api/auth/refresh` — обновление access token по действующему refresh token
- [x] `POST /api/auth/logout` — инвалидация refresh token (занесение в чёрный список / удаление из БД)
- [ ] Политика паролей: минимальная длина, требования к сложности (буквы, цифры, спецсимволы), срок действия пароля — настраивается администратором
- [x] Блокировка учётной записи после N неудачных попыток входа (N и время блокировки — настраиваются)
- [ ] Смена пароля пользователем: форма «старый пароль / новый пароль / подтверждение»
- [ ] Сброс пароля: отправка ссылки / одноразового кода на email, форма установки нового пароля, ограниченное время жизни ссылки
- [ ] Страница входа: отображение ошибок (неверный логин/пароль, аккаунт заблокирован, необходима смена пароля)
- [ ] Опция «Запомнить меня» (продление жизни refresh token до максимального значения)

#### FR-AUTH-01.2: Вход по сертификату (X.509)
- [ ] Административный интерфейс: загрузка и управление доверенными корневыми и промежуточными CA
- [ ] Настройка политики: обязательный или опциональный вход по сертификату для указанных ролей / всех пользователей
- [ ] Страница входа: кнопка «Войти по сертификату», выбор сертификата из хранилища браузера или загрузка PEM/PFX-файла
- [ ] Backend: проверка цепочки доверия, срока действия и статуса отзыва сертификата (OCSP / CRL)
- [ ] Сопоставление сертификата с учётной записью по Subject DN или Subject Alternative Name (email / UPN)
- [ ] Выдача JWT-пары (access + refresh) при успешной проверке сертификата
- [ ] Управление доверенными сертификатами в профиле пользователя: привязка нескольких сертификатов, просмотр, удаление

#### FR-AUTH-01.3: Вход по hardware-токену (OTP / FIDO2)
- [ ] Поддержка TOTP (RFC 6238 / Google Authenticator): генерация QR-кода для регистрации устройства, поле ввода одноразового кода на странице входа
- [ ] Поддержка FIDO2 / WebAuthn: регистрация аппаратного ключа (YubiKey и аналоги) через браузерный API, беспарольная аутентификация или в качестве второго фактора
- [ ] Двухфакторная аутентификация (2FA): вход в два шага — сначала логин/пароль, затем OTP или FIDO2-подтверждение
- [ ] Административные настройки: включение/отключение 2FA глобально или для отдельных ролей, обязательная 2FA
- [ ] Резервные коды (backup codes): генерация и выдача пользователю при регистрации устройства, однократное использование
- [ ] Управление токенами в профиле пользователя: список зарегистрированных устройств, добавление нового, удаление/отзыв

#### FR-AUTH-01.4: SSO / сквозная авторизация (SAML 2.0, OpenID Connect)
- [ ] Поддержка SAML 2.0: SP-Initiated и IdP-Initiated потоки, импорт метаданных IdP (URL или XML-файл), настройка сертификата подписи утверждений, маппинг атрибутов SAML → поля пользователя
- [ ] Поддержка OpenID Connect / OAuth 2.0: настройка клиента (client_id, client_secret, authorization/token/userinfo endpoints, scopes), маппинг claims → поля пользователя
- [ ] Поддержка нескольких провайдеров одновременно (мультитенантность / несколько IdP)
- [ ] Автоматическое создание учётной записи при первом SSO-входе (Just-In-Time provisioning) с применением дефолтной роли
- [ ] Синхронизация данных профиля при каждом SSO-входе (ФИО, email, роли из claims)
- [ ] Страница входа: отдельная кнопка / ссылка для каждого настроенного SSO-провайдера
- [ ] Административный интерфейс: CRUD-управление SSO-провайдерами, тестирование подключения
- [ ] Выход из системы с уведомлением IdP (Single Logout, SLO) — для SAML 2.0

#### FR-AUTH-01.5: Интеграция с LDAP / Active Directory
- [ ] Административный интерфейс: настройка подключения (хост, порт, протокол LDAP/LDAPS, Base DN, сервисная учётная запись и пароль), проверка соединения
- [ ] Режим аутентификации через LDAP: проверка логина и пароля напрямую в каталоге (bind-аутентификация)
- [ ] Режим импорта: синхронизация пользователей и групп/подразделений из каталога без аутентификации через LDAP
- [ ] Маппинг атрибутов LDAP → поля профиля системы (ФИО, email, телефон, должность, подразделение, фото)
- [ ] Маппинг групп LDAP → роли системы
- [ ] Ручной запуск синхронизации из административного интерфейса
- [ ] Автоматическая синхронизация по расписанию (настраиваемый cron-интервал)
- [ ] Журнал синхронизации: дата/время, количество обработанных/созданных/обновлённых/ошибочных записей
- [ ] Обработка удалённых / заблокированных пользователей в AD: автоматическая деактивация учётной записи в системе
- [ ] Поддержка вложенных групп (nested groups) Active Directory

### FR-AUTH-02: Управление сессиями

#### FR-AUTH-02.1: Активные сессии пользователя
- [ ] Страница «Мои активные сессии» в профиле пользователя: таблица со столбцами — устройство/браузер (User-Agent), IP-адрес, геолокация (город, страна по IP), дата и время последней активности, статус (текущая / другая)
- [ ] `GET /api/auth/sessions` — получение списка активных сессий текущего пользователя
- [ ] Выделение текущей сессии (пометка «эта сессия»)
- [ ] `DELETE /api/auth/sessions/{sessionId}` — завершение конкретной сессии: инвалидация соответствующего refresh token
- [ ] `DELETE /api/auth/sessions` — завершение всех остальных сессий, кроме текущей («Выйти со всех устройств»)
- [ ] Автоматическое завершение сессии по истечении времени неактивности (настраиваемый idle timeout в системных настройках)
- [ ] Административный интерфейс: просмотр активных сессий любого пользователя, принудительное завершение сессии администратором
- [ ] `GET /api/admin/users/{userId}/sessions` + `DELETE /api/admin/users/{userId}/sessions/{sessionId}` — административные эндпоинты
- [ ] Уведомление пользователя (in-app / email) при входе с нового устройства или неизвестного IP

#### FR-AUTH-02.2: Доверенные устройства (Remember Device)
- [ ] При входе в систему — опция «Запомнить это устройство» (чекбокс или отдельный экран после успешного 2FA)
- [ ] При отметке устройства как доверенного: выпуск уникального device token (UUID), сохранение его в БД (привязка к пользователю, User-Agent, fingerprint, дата выдачи, срок действия), передача в HttpOnly-cookie с длинным сроком жизни
- [ ] При следующем входе с того же устройства — пропуск шага 2FA (TOTP / FIDO2), если device token действителен
- [ ] `GET /api/auth/trusted-devices` — список доверенных устройств пользователя: название/UA, дата добавления, последнее использование
- [ ] `DELETE /api/auth/trusted-devices/{deviceId}` — отзыв (удаление) доверенного устройства
- [ ] `DELETE /api/auth/trusted-devices` — отзыв всех доверенных устройств сразу
- [ ] Административные настройки: максимальный срок действия device token (по умолчанию 30 дней), запрет на сохранение доверенных устройств (принудительный 2FA при каждом входе)
- [ ] Страница в профиле пользователя: список доверенных устройств с кнопкой удаления каждого

#### FR-AUTH-02.3: Доверенные сертификаты для подписания документов
- [ ] Страница «Мои сертификаты для подписания» в профиле пользователя
- [ ] Загрузка сертификата: форма с полем для PEM/DER/PFX-файла и необязательным паролем к закрытому ключу
- [ ] `POST /api/auth/signing-certificates` — загрузка и привязка сертификата к профилю пользователя
- [ ] Серверная проверка сертификата при загрузке: формат, срок действия, цепочка доверия до корневого CA (настроенного в системе)
- [ ] Хранение только публичной части сертификата на сервере; закрытый ключ хранится на стороне пользователя (смарт-карта, токен, локальное хранилище браузера)
- [ ] Список сохранённых сертификатов: субъект (CN), эмитент, серийный номер, срок действия, дата добавления
- [ ] `DELETE /api/auth/signing-certificates/{certId}` — отвязка и удаление сертификата
- [ ] Отображение статуса сертификата: действителен, истёк, отозван (проверка OCSP при просмотре списка)
- [ ] Использование сертификата при подписании документа в рамках процесса: выбор сертификата из списка доверенных, формирование электронной подписи (CMS/CAdES) на стороне клиента, передача подписи на сервер для верификации и сохранения
- [ ] Административные настройки: список разрешённых корневых CA для подписания, политика обязательного наличия сертификата для определённых ролей

### FR-AUTH-03: Управление доступом (RBAC + ABAC)

#### FR-AUTH-03.1: Роли и права на уровне модулей (RBAC)
- [ ] Административный интерфейс: CRUD-управление ролями (название, описание, цвет/иконка, статус активна/неактивна)
- [ ] Иерархия ролей: наследование прав от родительской роли с возможностью расширения или ограничения
- [ ] Предустановленные системные роли (Администратор, Пользователь, Гость) — защищены от удаления, частично редактируемы
- [ ] Матрица прав: список модулей системы (Процессы, Задачи, Документы, Справочники, Отчёты, Настройки) × типы действий (просмотр, создание, редактирование, удаление, публикация, администрирование)
- [ ] `GET /api/admin/roles` — список ролей с пагинацией и фильтрацией
- [ ] `POST /api/admin/roles` — создание роли
- [ ] `PUT /api/admin/roles/{roleId}` — обновление роли
- [ ] `DELETE /api/admin/roles/{roleId}` — удаление роли (с проверкой: роль не назначена ни одному пользователю)
- [ ] `GET /api/admin/roles/{roleId}/permissions` — матрица прав роли
- [ ] `PUT /api/admin/roles/{roleId}/permissions` — массовое обновление матрицы прав
- [ ] Назначение ролей пользователям: несколько ролей на одного пользователя; назначение через карточку пользователя или страницу роли
- [ ] `POST /api/admin/users/{userId}/roles` / `DELETE /api/admin/users/{userId}/roles/{roleId}` — добавление и снятие роли
- [ ] `GET /api/admin/roles/{roleId}/users` — список пользователей с данной ролью
- [ ] Назначение роли группе пользователей из оргструктуры или LDAP-группы
- [ ] Страница «Роли» в административном интерфейсе: таблица с поиском, фильтрами, счётчиком пользователей на роль
- [ ] Копирование роли (клонирование с именем «Копия <название>») для быстрого создания похожих ролей
- [ ] Экспорт/импорт матрицы прав в JSON/CSV для переноса между окружениями

#### FR-AUTH-03.2: Права на уровне объектов (ABAC — ACL)
- [ ] Политики доступа к объектам (документ, папка, справочник, экземпляр процесса): набор ACE-записей «субъект → операция → разрешено / запрещено»
- [ ] Субъекты политики: конкретный пользователь, роль, подразделение, все авторизованные пользователи
- [ ] Операции над объектами: чтение, запись, удаление, согласование, архивирование, управление правами (share)
- [ ] Явный запрет (Deny) с приоритетом над разрешением (Deny overrides Allow)
- [ ] Наследование прав: дочерние папки/объекты наследуют ACL родительской папки с возможностью переопределения на уровне дочернего
- [ ] Режим «только владелец»: автор объекта всегда сохраняет полный доступ и не может быть лишён его через ACL
- [ ] Интерфейс управления правами (контекстное меню объекта → «Управление доступом»): таблица ACE с добавлением, редактированием и удалением записей
- [ ] `GET /api/permissions/{objectType}/{objectId}` — получение текущего ACL объекта
- [ ] `POST /api/permissions/{objectType}/{objectId}` — добавление новой ACE
- [ ] `PUT /api/permissions/{objectType}/{objectId}/{aceId}` — обновление ACE
- [ ] `DELETE /api/permissions/{objectType}/{objectId}/{aceId}` — удаление ACE
- [ ] Проверка прав на уровне middleware / ASP.NET Core authorization handler перед каждой операцией с объектом
- [ ] Делегирование прав: пользователь с правом «управление правами» может передать часть своих прав на объект другому пользователю
- [ ] Инструмент «Проверить доступ»: администратор вводит пользователя и объект — система отображает итоговый набор прав с объяснением, какое правило сработало

#### FR-AUTH-03.3: Права на уровне полей/свойств объектов
- [ ] Матрица «поле × роль» для каждой формы: состояния — скрыто, только для чтения, редактируемо
- [ ] Настройка матрицы в дизайнере форм (для каждого поля — кнопка/панель прав) и в административном интерфейсе (отдельная вкладка формы)
- [ ] `GET /api/admin/forms/{formId}/field-permissions` — получение матрицы прав полей формы
- [ ] `PUT /api/admin/forms/{formId}/field-permissions` — обновление матрицы
- [ ] Динамические ABAC-правила видимости и редактируемости на основе атрибутов контекста: выражения вида «поле "Сумма" → только чтение, если роль = "Исполнитель" И статус ≠ "Утверждён"»
- [ ] Применение ограничений на уровне API: сервер не включает скрытые поля в ответ, игнорирует значения readonly-полей из тела запроса
- [ ] Маскирование значений по роли (например, номер счёта — только последние 4 цифры для всех ролей, кроме «Финансист»)
- [ ] Отображение в UI: скрытые поля не рендерятся, readonly-поля блокируются средствами фреймворка (React controlled components + серверная валидация)

#### FR-AUTH-03.4: Глобальные настройки доступа
- [ ] Страница глобальных настроек безопасности в административном интерфейсе
- [ ] Режим «закрытая система»: доступ к объекту запрещён по умолчанию, если явно не разрешён ACL
- [ ] Режим «открытая система»: доступ разрешён по умолчанию для всех авторизованных пользователей, если не указан явный запрет
- [ ] Настройка прав по умолчанию для новых объектов: автор — владелец с полным доступом; дополнительные роли и их права (например, роль «Пользователь» — только чтение)
- [ ] `GET /api/admin/access-settings` / `PUT /api/admin/access-settings` — чтение и сохранение глобальных настроек
- [ ] Матрица совместимости ролей (Separation of Duties): настройка пар несовместимых ролей (система запрещает одновременное назначение обеих ролей одному пользователю)
- [ ] Настройка максимального числа ролей у одного пользователя (опционально)
- [ ] Политика доступа для API-ключей и сервисных аккаунтов: отдельный набор разрешений, ограниченная область действия (scopes)
- [ ] Аудит операций: запись в журнал событий всех READ/WRITE/DELETE по защищённым объектам (включается в глобальных настройках, уровень детализации настраивается)
- [ ] Журнал изменений прав: кто, когда, что изменил в ролях, ACL и политиках — с возможностью просмотра diff и отката на предыдущее состояние

#### FR-AUTH-03.5: Замещение пользователей
- [ ] Страница «Замещения» в профиле пользователя (мои замещения) и в административном интерфейсе (все замещения)
- [ ] Создание замещения: выбор замещаемого пользователя, заместителя, периода (дата/время начала и окончания), области действия (все права / только задачи / выбранные роли)
- [ ] `POST /api/substitutions` — создание замещения
- [ ] `GET /api/substitutions` — список активных и запланированных замещений текущего пользователя
- [ ] `GET /api/admin/substitutions` — административный список всех замещений с фильтрами (по пользователям, статусу, периоду)
- [ ] `PUT /api/substitutions/{subId}` — изменение периода или области действия замещения
- [ ] `DELETE /api/substitutions/{subId}` — досрочное завершение замещения
- [ ] Автоматическая активация и деактивация замещения по расписанию: фоновый сервис (Hangfire / .NET hosted service) отслеживает плановые даты
- [ ] Переключатель контекста в интерфейсе: заместитель видит индикатор «Работаю за <ФИО>» и может переключаться между своим рабочим пространством и пространством замещаемого без повторной аутентификации
- [ ] Задачи и процессы замещаемого пользователя отображаются в рабочем пространстве заместителя в режиме замещения
- [ ] Уведомления: заместителю — о начале замещения (email / in-app); замещаемому — о передаче замещения и о его завершении
- [ ] Аудит действий заместителя: в журнале отображается «Иванов И.И. (замещает Петрова П.П.)» для полной прозрачности
- [ ] Ограничения: заместитель не может делегировать полученные полномочия третьим лицам; цепочки замещений (A → B → C) запрещены
- [ ] Административное принудительное замещение: администратор может создать замещение от имени любого пользователя (например, при экстренной ситуации)

## 2. Профиль и оргструктура

### FR-ORG-01: Организационная структура

#### FR-ORG-01.5: Моделирование оргструктуры в Дизайнере
- [ ] Раздел «Оргструктура» в Дизайнере — визуальная схема с иерархией элементов; работа доступна только пользователям с доступом к Дизайнеру; добавление нового элемента наведением курсора → иконки действий; после создания элемента указывается название и описание
- [ ] **Элементы оргструктуры**: *Должность* (![position-icon]) — один сотрудник; *Отдел* (![orgstructure-4]) — несколько сотрудников; содержит начальника (один), сотрудников (неограниченно) и вложенную иерархию; *Группа пользователей* (![user-group-icon]) — несколько сотрудников без начальника; *Вложенная оргструктура* (![nested-structure-icon]) — обособленная подструктура (напр., филиал)
- [ ] **Должность**: только один пользователь; может быть помечена как «руководитель» внутри отдела; добавление должностей и групп внутри отдела на одном уровне с начальником
- [ ] **Вложенная оргструктура**: создаётся внутри существующей схемы; открывается на отдельной вкладке; один головной элемент; компания может иметь неограниченное количество вложенных оргструктур
- [ ] Перетаскивание элементов для изменения расположения схемы; удаление элемента с выбором: «Удалить вместе с дочерними» или «Оставить дочерние»; поиск использования элемента в процессах, документах, РКК, матрицах ответственности и пользователях
- [ ] **Режим дерева**: переключатель «Отображать в виде дерева» — список с иерархией вместо визуальной схемы; действия по наведению на узел: добавить дочерний элемент, перейти во вложенную оргструктуру
- [ ] **Публикация оргструктуры**: изменения в оргструктуре сохраняются как черновик; публикация через кнопку на вкладке «Оргструктура» Дизайнера или с общей страницы раздела «Публикация»; `POST /api/designer/orgstructure/publish`; до публикации изменения не отображаются на портале

### FR-PORTAL-02: Поиск

#### FR-PORTAL-02.1: Полнотекстовый поиск
- [ ] Единое поле глобального поиска в хедере портала: поиск по всем проиндексированным типам объектов одновременно
- [ ] `GET /api/search?q={query}&types={}&filters={}` — унифицированный поисковый endpoint; возвращает результаты, сгруппированные по типу объекта
- [ ] Поиск по следующим типам объектов: задачи, экземпляры процессов, документы (метаданные + содержимое), пользователи, подразделения, должности, страницы портала, новости
- [ ] Ранжирование результатов: релевантность (TF-IDF / BM25), дата изменения, тип объекта; настройка весов в конфигурации
- [ ] Подсветка вхождений поискового запроса в заголовке и тексте анонса результата (highlight)
- [ ] Поиск с учётом морфологии русского и английского языков (стемминг / лемматизация через встроенные анализаторы поискового движка)
- [ ] Поиск с опечатками (fuzzy search): допуск до 1–2 замен символов; порог нечёткости настраивается
- [ ] Автодополнение (autocomplete): при вводе от 2 символов в поле поиска — выпадающий список подсказок (название задачи, имя пользователя, номер документа); `GET /api/search/suggest?q=`
- [ ] Результаты живого поиска (instant search): предпросмотр топ-5 результатов прямо в выпадающем меню хедера (без перехода на страницу поиска)
- [ ] Страница полных результатов: открывается при нажатии Enter или «Показать все результаты»; пагинация (page / page_size)
- [ ] Права доступа в поиске: результаты фильтруются по правам текущего пользователя (не возвращаются объекты, к которым у него нет доступа)
- [ ] Индексирование в реальном времени: при создании/изменении/удалении объекта событие публикуется во внутреннюю очередь, индекс обновляется асинхронно с задержкой не более 5–10 с

#### FR-PORTAL-02.2: Расширенный поиск и EQL
- [ ] Страница «Расширенный поиск»: форма с отдельными полями-фильтрами для каждого типа объекта; переключатель «Простой / Расширенный»
- [ ] Фильтры для задач: статус, приоритет, исполнитель, назначивший, процесс, дата создания (диапазон), срок (диапазон), теги
- [ ] Фильтры для документов: тип документа, автор, папка, дата изменения (диапазон), статус, версия
- [ ] Фильтры для процессов: определение процесса, стадия, исполнитель, дата запуска (диапазон), статус
- [ ] Фильтры для пользователей: подразделение, должность, роль, статус (активный/архивный)
- [ ] EQL (Enterprise Query Language): текстовый синтаксис запросов вида `status:active AND assignee:me AND due:<2026-05-01 AND tag:"важно"`; поддержка операторов AND, OR, NOT, скобок, wildcards (`*`, `?`), диапазонов (`due:[2026-01-01 TO 2026-06-01]`), специальных значений (`me`, `today`, `this_week`)
- [ ] Поле EQL в расширенном поиске: подсветка синтаксиса, автодополнение полей и значений, сообщения об ошибках парсинга
- [ ] `GET /api/search/eql?q={eql_query}` — поиск через EQL; возвращает единый список с type-дискриминатором
- [ ] Сохранённые поисковые запросы: пользователь сохраняет поиск под именем; `POST /api/search/saved`, `GET /api/search/saved`, `DELETE /api/search/saved/{id}`
- [ ] Запуск сохранённого поиска одним кликом из меню «Сохранённые поиски»
- [ ] Поиск-подписка: пользователь подписывается на сохранённый поиск — получает уведомление при появлении новых результатов; `POST /api/search/saved/{id}/subscribe`
- [ ] Экспорт результатов поиска в Excel / CSV (до 10 000 записей): `GET /api/search/export?q=...&format=xlsx`

#### FR-PORTAL-02.3: Поисковые результаты и UX
- [ ] Страница результатов: вкладки по типам объектов с счётчиками («Все (142) / Задачи (47) / Документы (31) / ...»)
- [ ] Карточка результата: иконка типа, название (ссылка), анонс с подсветкой, метаданные (автор, дата, статус)

### FR-BPM-01: Моделирование процессов (Дизайнер)

#### FR-BPM-01.1: Графический редактор BPMN 2.0
- [x] Холст (canvas) с бесконечной прокруткой, масштабирование (Ctrl+scroll / кнопки ±), режим «По размеру» (Fit to screen)
- [x] **Миникарта (minimap)** *(приоритет: средний)*: накладная миниатюра диаграммы (правый нижний угол), отображает весь граф и текущий вьюпорт; кнопка скрыть/показать; реализуется через плагин `diagram-js-minimap`
- [x] **Авто-раскладка (auto-layout)** *(приоритет: средний)*: кнопка «Упорядочить схему» — автоматическое расположение элементов по алгоритму направленного графа; реализуется через `bpmn-auto-layout`
- [x] Палитра элементов BPMN 2.0 слева: события, задачи, шлюзы, соединители, подпроцессы, swimlane, аннотации, артефакты (группы, текстовые аннотации, ассоциации); кнопка свернуть/развернуть палитру
- [x] Drag-and-drop элементов из палитры на холст; snap-to-grid (настраиваемый шаг сетки); выравнивание по вспомогательным линиям
- [x] **События**: стартовое (пустое, таймер, сообщение, сигнал, условие, ошибка), конечное (пустое, сообщение, ошибка, эскалация, завершение), промежуточное захватывающее / генерирующее (таймер, сообщение, сигнал, ошибка, эскалация, компенсация, граничное прерывающее / непрерывающее)
- [x] **Задачи**: пользовательская (User Task), сервисная (Service Task / HTTP-вызов), бизнес-правило (Business Rule Task), скрипт (Script Task / C#), ручная (Manual Task), отправка сообщения (Send Task), получение сообщения (Receive Task), RPA-задача (расширение)
- [x] **Шлюзы**: эксклюзивный (XOR), параллельный (AND), включающий (OR), основанный на событии (Event-based), сложный (Complex)
- [x] **Подпроцессы**: встроенный (inline), ссылочный (Call Activity — вызов другого определения процесса), событийный подпроцесс (Event Sub-Process), транзакционный подпроцесс

### FR-BPM-02: Выполнение процессов

#### FR-BPM-02.0: Роли в процессе
- [x] **Инициатор** — пользователь, запускающий бизнес-процесс; является участником экземпляра и по умолчанию совпадает с ответственным; определяется зоной ответственности стартового события в дизайнере
- [x] **Владелец** — пользователь (или группа), ответственный за результат выполнения процесса на всех этапах; может быть задан как должность или группа сотрудников; один владелец на процесс; настраивается на вкладке «Роли» в дизайнере (BpmProcessRoleConfig, тип Owner)
- [x] **Куратор** — пользователь, не участвующий в выполнении, но имеющий права просмотра всей информации по экземплярам; может быть несколько кураторов; настраивается на вкладке «Роли» (BpmProcessRoleConfig, тип Curator)
- [x] **Ответственный за экземпляр** — пользователь, отвечающий за достижение результата конкретного экземпляра; по умолчанию совпадает с инициатором; может быть изменён владельцем, инициатором или пользователем с соответствующими правами; только один на экземпляр (UI-описание в дизайнере, runtime — FR-BPM-02.2)
- [x] **Участник** — пользователь, видящий экземпляр процесса в разделе «Мои процессы»; список участников управляем (добавление / удаление) при наличии прав; по умолчанию — владелец и ответственный за экземпляр (UI-описание в дизайнере, runtime — FR-BPM-02.3)
- [x] **Информируемый** — сотрудник, получающий сведения о ходе процесса организационными мерами; не взаимодействует с системой, но упоминается в документации и регламенте процесса (UI-описание в дизайнере)

#### FR-BPM-02.1: Запуск экземпляра
- [x] Запуск вручную из UI: (1) кнопка «▷» в карточке процесса (кнопка «Запустить процесс» на панели инструментов добавлена); (2) через главное меню — отложено (портальный раздел); (3) через портлет — отложено; открывается стартовое окно → `POST /api/bpm/processes/{id}/instances`
- [x] Стартовое окно: варианты отображения — (a) стандартное (только поле названия экземпляра); (b) стандартное с описанием процесса и контекстными переменными (`IsInput`-переменные); (d) название формируется по шаблону автоматически; (c) форма из конструктора форм — отложено до реализации рендерера форм; просмотр карты перед запуском — отложено

#### FR-BPM-02.8: Движок выполнения BPMN (Execution Engine)
- [x] **Доменная модель**: `BpmToken` (таблица `bpm_tokens`) — `InstanceId`, `ElementId`, `ElementType`, `ElementName`, `Status`, `SignalCode`, `MessageCode`, `CorrelationKey`, `CreatedAt`, `CompletedAt`; `BpmTokenStatus` enum (Active/WaitingUserAction/WaitingSignal/WaitingMessage/Completed/WaitingJoin/WaitingTimer); миграция `AddBpmTokens`
- [x] **Интерфейс движка** `IBpmExecutionEngine`: `StartAsync`, `AdvanceFromAsync`, `CompleteUserTaskAsync`, `SendSignalAsync`, `SendMessageAsync`, `ExecuteJobAsync`, `GetTokensAsync`
- [x] **Реализация `BpmExecutionEngine`**:
  - `startEvent` → немедленное продвижение через `AdvanceFromAsync`; `endEvent` → завершение экземпляра (`State = Completed`)
  - `userTask` / `receiveTask` → токен `WaitingUserAction` + SignalR-уведомление участникам через `IBpmNotificationService.NotifyUserTaskActivatedAsync`
  - `serviceTask` / `scriptTask` → создание `BpmExecutionJob` + токен `Active`
  - `exclusiveGateway` → выбор первого потока с выполненным условием; `inclusiveGateway` → активация всех подходящих потоков; `parallelGateway` → все выходы
  - **AND-Join семантика**: `BpmJoinCounter` (таблица `bpm_join_counters`), при входе токена в `parallelGateway`/`inclusiveGateway` с несколькими входящими дугами — счётчик инкрементируется; продвижение вперёд только когда `ArrivedCount >= ExpectedCount`; миграция `AddBpmJoinCounters`
  - `intermediateCatchEvent` + `timerEventDefinition`: токен `WaitingTimer`, создание `BpmExecutionJob` с `IsTimer=true` и `NextRunAt = now + duration` (ISO 8601); срабатывание — через `BpmEngineWorker`; `ResolveTimerFireAt` поддерживает `timeDuration`, `timeDate`, `timeCycle`
  - `intermediateCatchEvent` → токен `WaitingSignal` / `WaitingMessage`; `intermediateThrowEvent` → рассылка сигнала + немедленное продвижение
  - **Граничные события (Boundary Events)**: `ParseXml` парсит `boundaryEvent` с `attachedToRef`, `cancelActivity`, `errorEventDefinition`, `timerEventDefinition`, `signalEventDefinition`; при ошибке задания после исчерпания попыток — `TryActivateBoundaryErrorEventAsync` сопоставляет код ошибки с `BoundaryErrorCode`, при совпадении активирует граничный путь вместо перехода в `Faulted`
  - `subProcess`: рекурсивный разбор вложенных элементов через `ParseContainerElements`
  - **ServiceTask op=HttpCall**: HTTP-вызов к внешнему URL с подстановкой переменных `{{var}}`; **op=ChangeInstanceStatus**: обновление переменной статуса
  - **ScriptTask**: реальное выполнение C#-сценариев через Roslyn; контекст `ScriptContext`; таймаут; inline-скрипт или `BpmScriptModule`
  - **BusinessRuleTask**: заглушка (DMN-интерпретатор отложен)
  - При успехе задания: `NodeExecuted` + `AdvanceFromAsync`; при исчерпании попыток: `NodeFailed` + проверка граничного события ошибки → `State = Faulted` если нет обработчика
  - Оценка условий `ConditionExpression`: поддержка `==`, `!=`, `>=`, `<=`, `>`, `<`; JUEL/FEEL-обёртки
- [x] **`BpmEngineWorker`** (`BackgroundService`): polling каждые 5 секунд; атомарный захват до 20 заданий; параллельное выполнение; обработка таймерных заданий (`IsTimer=true`)
- [x] **Интеграция с `BpmInstanceService`**: after `CreateInstanceAsync` — `_engine.StartAsync`
- [x] **API токенов**: `GET /api/bpm/instances/{id}/tokens`; `POST .../tokens/{elementId}/complete`
- [x] **API событий**: `POST /api/events/signals/{signalCode}`; `POST /api/events/messages/{messageCode}`
- [x] **Frontend**: `BpmTokenStatus` расширен (`WaitingJoin`, `WaitingTimer`); `InstancePage` — вкладка «Карта процесса» с bpmn-js NavigatedViewer и подсветкой активных токенов по `elementId`; уведомления `UserTaskActivated` обрабатываются через SignalR
- [x] **`IBpmNotificationService.NotifyUserTaskActivatedAsync`**: рассылка уведомления группам `instance:{id}` и Admin при активации UserTask
- Отложено: Call Activity с передачей переменных (создание дочернего экземпляра), BusinessRuleTask (DMN-движок), граничный таймер для задач (TimerDeadline в BpmExecutionJob)

## 5. Задачи

### FR-TASK-01: Управление задачами

#### FR-TASK-01.1: Создание и карточка задачи
- [ ] Навигация в раздел «Задачи»: через левое меню системы; через главное меню «Мои задачи»; через портлет «Задачи» на главной странице
- [ ] Создание задачи тремя способами: кнопка «Создать» → «Задачу» на главной странице; «Создать задачу» в главном меню; кнопка «Создать» или выпадающее меню в разделе «Задачи»; `POST /api/tasks`
- [ ] Создание задачи из процесса: User Task в BPMN автоматически создаёт запись при достижении узла, заполняя поля из переменных процесса; связь с экземпляром процесса и узлом сохраняется
- [ ] **Роли пользователей в задаче**:
  - *Автор* — создатель задачи; имеет права просмотра и редактирования; руководители автора также могут редактировать
  - *Исполнитель* — ответственный за выполнение; может быть переназначен; имеет права просмотра
  - *Соисполнитель* — участвует в выполнении вместе с исполнителем, не является ответственным; получает все оповещения; имеет право выполнения задачи, если администратор разрешил (`настройка «Соисполнитель имеет право выполнения»`)
  - *Наблюдатель* — получает оповещения о действиях с задачей; является подписчиком
  - *Согласующий* — ответственный за согласование задачи
  - *Контролёр* — контролирует процесс выполнения задачи; у задачи может быть только один контролёр
  - Права просмотра: автор, исполнитель, соисполнитель, наблюдатель, контролёр, руководители автора/исполнителя/соисполнителя (согласно оргструктуре)
  - Права редактирования: автор задачи, руководители автора (согласно оргструктуре)
- [ ] **Поля карточки задачи**:
  - Тема (обязательно) — краткая формулировка; отображается в списке задач и портлете
  - Исполнитель (обязательно) — один пользователь или сохранённая группа
  - Сроки задачи (обязательно) — дата и время начала и завершения; задача может длиться несколько дней; по умолчанию нельзя установить на прошедшую дату (разрешается отдельным правом)
  - Корректировка сроков — автоматическое смещение даты начала, если она выпала на праздничный/выходной день: *Без смещения* (по умолчанию, значение берётся из настроек администратора), *Первый рабочий день* (смещает дату начала; дата завершения не меняется), *Сохранять длительность* (смещаются обе даты с учётом длительности)
  - Приоритет — степень важности задачи
  - Категория — из справочника «Категория задач»; участвует в фильтрации и сортировке
  - Описание — подробное объяснение задачи
  - Вложения — файлы с компьютера / внешних носителей и электронные документы; `POST /api/tasks/{id}/attachments`
  - Согласующий (+Согласующий) — один пользователь; опционально при создании
  - Соисполнители (+Соисполнитель) — участвуют в выполнении; получают все оповещения по задаче
  - Наблюдатели (+Соисполнитель) — следят за ходом выполнения; являются подписчиками
  - Контроль выполнения (+Контроль) — тип контроля и контролёр; если не заполнено, применяется тип по умолчанию из настроек администратора
  - Плановые трудозатраты (через «Еще») — плановое время на задачу; прогресс-бар «факт / план» на карточке
  - Мероприятие (через «Еще») — привязка задачи к маркетинговому мероприятию
  - Проект (через «Еще») — привязка задачи к проекту (при наличии приложения Проекты+)
- [ ] Страница задачи — вкладки: *Описание* (срок, описание, вложения, автор, исполнители, контролёр, статус, приоритет, история действий), *Родительская задача* (если текущая является подзадачей и есть права доступа к родителю), *Подзадачи* (если созданы подзадачи), *Трудозатраты* (после добавления трудозатрат)
- [ ] Подзадачи: создание подзадач внутри карточки; дерево подзадач с раскрытием; `POST /api/tasks/{id}/subtasks`; при переходе на родительскую задачу после выполнения дочерней — настраивается администратором
- [ ] Комментарии к задаче: добавление комментария, вложений, ответов на вопросы; история (комментарии, вложения, вопросы, действия) отображается на вкладке «Описание»; `GET/POST /api/tasks/{id}/comments`

#### FR-TASK-01.2: Статусы задачи
- [ ] Полный жизненный цикл простой задачи со следующими статусами:
  - *Новая* — задача создана, отредактирована или переназначена на нового исполнителя
  - *Прочитана* — исполнитель первый раз открыл страницу задачи
  - *Выполняется* — исполнитель выбрал действие «Работа начата»
  - *На предварительном согласовании* — автор задал согласующего при создании и сохранил задачу
  - *На согласовании* — исполнитель выбрал «Отправить на согласование»
  - *Согласовано* — согласующий нажал «Согласовать»; задача возвращается к исполнителю со статусом «Новая»
  - *Отказано в предварительном согласовании* — согласующий нажал «Отказать» при предварительном согласовании
  - *Отказано в согласовании* — согласующий нажал «Отказать»; можно повторно отправить на согласование
  - *Выполнено* — исполнитель нажал «Сделано» (контроль не назначен)
  - *Выполнено, нужен контроль* — исполнитель нажал «Сделано», контролёр ещё не проверил
  - *Выполнено и проконтролировано* — исполнитель и контролёр оба нажали «Сделано»
  - *Невозможно выполнить* — исполнитель выбрал «Невозможно выполнить» (контроль не назначен)
  - *Невозможно выполнить, нужен контроль* — исполнитель выбрал «Невозможно выполнить», контролёр ещё не проверил
  - *Невозможно выполнить и проконтролировано* — исполнитель выбрал «Невозможно выполнить» и контролёр нажал «Сделано»
  - *Закрыто* — автор задачи или пользователь с правами управления задачей выбрал «Отменить задачу»; также присваивается незавершённой задаче переназначения заблокированного пользователя при его разблокировке
- [ ] Автоматический флаг просрочки: фоновый job проверяет задачи с истёкшим сроком и нефинальным статусом; флаг хранится отдельно от статуса; в списке задачи выделяются красным
- [ ] При переходе задачи в финальный статус («Выполнено…» / «Закрыто»): уведомление наблюдателям; если задача из процесса — отправка результата в движок процессов (complete User Task token)
- [ ] SLA-правила: администратор задаёт правила — для задач категории X с приоритетом Y срок по умолчанию Z рабочих часов; при создании задачи срок заполняется автоматически по SLA; `GET/POST/PUT/DELETE /api/admin/task-sla`

#### FR-TASK-01.3: Согласование задачи
- [ ] Два способа отправки на согласование:
  - *Предварительное согласование* — автор добавляет согласующего (+Согласующий) при создании задачи; задача получает статус «На предварительном согласовании» сразу после сохранения
  - *Согласование от исполнителя* — исполнитель выбирает «Отправить на согласование» (меню «Еще»); указывает согласующего, сроки и комментарий
- [ ] При создании согласования формируется отдельная задача согласования для согласующего; родительская задача (обозначение ![approve]) и задача согласования (обозначение ![approval-sub]) различаются иконками
- [ ] До завершения задачи согласования исполнитель не может выполнить родительскую задачу
- [ ] Действия согласующего: *Согласовать* / *Отказать* + опциональный комментарий и вложение; результат и комментарий отображаются на странице родительской задачи
- [ ] `GET /api/tasks/{id}/approval` — текущее состояние согласования; `POST /api/tasks/{id}/approval/decision` — решение согласующего
- [ ] При согласовании родительская задача возвращается к исполнителю со статусом «Новая»; при отказе — статус «Отказано в согласовании»; можно повторно отправить
- [ ] Уведомление согласующему при поступлении задачи на его согласование; напоминание при приближении/истечении срока
- [ ] Шаблоны маршрутов согласования: `GET/POST/PUT/DELETE /api/admin/approval-templates`; возможность сохранить текущий маршрут как шаблон

#### FR-TASK-01.4: Контроль исполнения и трудозатраты
- [ ] У задачи может быть только один контролёр; назначается при создании через «+Контроль» или изменяется после создания
- [ ] Типы контроля (настраиваются при создании задачи, выбор дефолтного — у администратора):
  - *Контроль выполнения* — после нажатия исполнителем «Сделано» задача направляется контролёру; контролёр может принять («Сделано» → статус «Выполнено и проконтролировано») или вернуть («Открыть заново» → задача получает статус «Новая» у исполнителя)
  - *Текущий контроль* — задача отображается у контролёра с момента создания; контролёр отслеживает весь процесс; после выполнения исполнителем переходит в фильтр «Контроль выполнения» у контролёра; может вернуть задачу на доработку
  - *Оповещать при выполнении* — пользователь получает оповещение в ленте сообщений после выполнения задачи; не может принять или отклонить результат
  - *Без контроля* — контролёр не назначается
- [ ] Настройки контроля в системе (администратор): автоматически устанавливать контроль при создании (Да/Нет), тип контроля по умолчанию, разрешить пользователям изменять контроль после создания, разрешить персональную настройку контроля в профиле пользователя
- [ ] Изменение параметров контроля после создания задачи: автор или контролёр меняет контролёра, тип контроля, или снимает контроль; `PUT /api/tasks/{id}/control`
- [ ] Взять задачу на текущий контроль: любая роль может выбрать «Взять на контроль» в меню «Еще»; задача появляется в фильтре «Текущий контроль»; снять через «Снять с контроля»
- [ ] Массовый контроль задач: выбрать несколько задач флажками → «Сделано» — подтвердить; `POST /api/tasks/bulk-verify`
- [ ] `POST /api/tasks/{id}/verify` — отметка контроля; `GET /api/tasks/{id}/verification` — статус и имя контролёра
- [ ] Трудозатраты при выполнении задачи: в диалоге «Сделано» — поля «Затрачено» (чч:мм) и «Дата начала»; `POST /api/tasks/{id}/timelogs`
- [ ] Трудозатраты в процессе работы: «Еще» → «Добавить трудозатраты» — указать время, дату начала, опциональный комментарий; `POST /api/tasks/{id}/timelogs`
- [ ] Настройка обязательного добавления трудозатрат администратором: пользователь не может выполнить задачу без ввода трудозатрат
- [ ] Плановые трудозатраты на задаче (поле «Плановые трудозатраты»); прогресс-бар «факт / план» на карточке
- [ ] Суммирование трудозатрат по подзадачам на родительской задаче
- [ ] Журнал трудозатрат на вкладке «Трудозатраты» страницы задачи: две панели — *Отчёт о поданных трудозатратах* (если трудозатраты отправлены на согласование; сводная таблица с группировкой по виду деятельности или исполнителям) и *Журнал трудозатрат* (трудозатраты, добавленные через «Еще»); `GET /api/tasks/{id}/timelogs`
- [ ] Отчёт по трудозатратам: `GET /api/reports/timelogs` — фильтры по периоду, пользователю, задаче, проекту; экспорт в Excel; группировка по дням / неделям / месяцам / исполнителю / задаче

#### FR-TASK-01.5: Типы задач

##### FR-TASK-01.5.1: Периодическая задача
- [ ] Периодическая задача — задача, которая повторяется с заданной частотой; при создании указывается периодичность, условие завершения и количество создаваемых вперёд экземпляров; `POST /api/tasks` с полем `type: "periodic"`
- [ ] Периодичность: *По рабочим дням* (с учётом рабочего календаря), *Ежедневно* (без учёта рабочего календаря), *Еженедельно*, *Ежемесячно*, *Ежеквартально*, *Ежегодно*
- [ ] Расчёт сроков: для всех типов, кроме «Ежедневно без учёта календаря», длительность и сроки рассчитываются в рабочих днях по производственному календарю; если сроки попадают на выходные/праздники, сдвигаются на ближайшие рабочие дни
- [ ] Завершение серии: *Не завершать* — повторяется до принудительной остановки; *Завершить* — прекратить повторение в указанную дату
- [ ] Количество вперёд создаваемых задач: на следующий день после начала очередного экземпляра создаётся новый, чтобы их количество оставалось постоянным; при значении 0 экземпляры создаются только в день начала
- [ ] Периодические задачи не назначаются на заблокированных пользователей; при разблокировке пользователя назначение возобновляется
- [ ] Просмотр: подраздел «Периодические задачи» → «Текущие» и «Завершённые»; раскрытие списка экземпляров по клику на название
- [ ] Редактирование периодической задачи: доступно автору и его руководителям; `PUT /api/tasks/{id}`
- [ ] Удаление: доступно автору и его руководителям; при удалении выбирается действие для активных экземпляров — «Принудительно завершить» или «Удалить»

##### FR-TASK-01.5.2: Задача по процессу
- [ ] Задача по процессу — создаётся автоматически при достижении узла User Task в BPMN; обозначается специальной иконкой в списке задач; `GET /api/tasks/{id}` содержит ссылку на экземпляр процесса
- [ ] На странице задачи отображается: название процесса с номером версии, название экземпляра (ссылка), дата запуска, инициатор, ответственный; вкладки «Главная страница» (данные для выполнения) и «История»
- [ ] Если исполнитель задачи удалён или заблокирован из оргструктуры, задача перенаправляется ответственному за экземпляр процесса
- [ ] Дополнительные действия для задачи по процессу: *Улучшить* — создать предложение по улучшению процесса (доступно инициатору, участнику или ответственному экземпляра, или пользователям с правом создания запросов на улучшение); *Карта процесса* — просмотр карты (при наличии права на мониторинг); *Скачать вложения* — скачать все файлы архивом
- [ ] При выполнении задачи по процессу — отправка результата в движок процессов (complete User Task token)

### FR-TASK-02: Действия с задачами

#### FR-TASK-02.1: Действия с задачей
- [ ] **Сделано** — завершить работу над задачей; диалог: Трудозатраты (Затрачено, Дата начала), Комментарий, «Скопировать вложения из подзадач» (флажок), Вложения, «Оповестить соисполнителей» (флажок); `PUT /api/tasks/{id}/complete`
- [ ] **Невозможно выполнить** — отметить, если задача не может быть выполнена; аналогичный диалог; задача получает статус «Невозможно выполнить» (с учётом наличия/отсутствия контроля); `PUT /api/tasks/{id}/impossible`
- [ ] **Работа начата** — необязательное действие; фиксирует начало работы; диалог: комментарий, вложения, оповещение соисполнителей; задача получает статус «Выполняется»; `PUT /api/tasks/{id}/start`
- [ ] **Изменить** — редактировать задачу; доступно автору задачи и руководителям автора
- [ ] **Создать подзадачу** — создать подзадачу; текущая задача становится родительской; `POST /api/tasks/{id}/subtasks`
- [ ] **Копия** — создать копию задачи для быстрого создания схожих задач; доступно всем участникам, кроме согласующего; запрещено копировать завершённые задачи и задачи по резолюции документа; `POST /api/tasks/{id}/copy`
- [ ] **Переназначить** — выбрать другого исполнителя; диалог: новый исполнитель, комментарий; доступно автору, исполнителю и контролёру задачи; для задачи по процессу — также согласующему при наличии прав к процессу; `PUT /api/tasks/{id}/reassign`; в активности фиксируется кто, кому и когда переназначил
- [ ] **Перенести срок** — перенести сроки задачи с комментарием; доступно автору и контролёру; `PUT /api/tasks/{id}/reschedule`
- [ ] **Отменить задачу** — нельзя удалить задачу, но можно отменить; диалог: комментарий, вложения, оповещение соисполнителей; статус задачи становится «Закрыто»; `PUT /api/tasks/{id}/cancel`
- [ ] **Отправить на согласование** — отправить задачу на согласование (доступно исполнителю); диалог: согласующий, сроки, комментарий; `POST /api/tasks/{id}/approval`
- [ ] **Добавить комментарий** — добавить комментарий к задаче; `POST /api/tasks/{id}/comments`
- [ ] **Задать вопрос** — задать вопрос по задаче с возможностью прикрепить вложения; получатель вопроса видит задачу с отдельным обозначением; `POST /api/tasks/{id}/questions`
- [ ] **Добавить трудозатраты** — добавить трудозатраты в процессе работы; доступно всем ролям, кроме наблюдателя
- [ ] **Список наблюдателей** — добавить наблюдателей; `POST /api/tasks/{id}/watchers`
- [ ] **Взять на контроль** / **Снять с контроля** — доступно всем ролям; `POST /api/tasks/{id}/control`, `DELETE /api/tasks/{id}/control`
- [ ] **Открыть заново** — возвращает задачу со статусом «Новая» к исполнителю; доступно контролёру; `PUT /api/tasks/{id}/reopen`
- [ ] **Скачать вложения** — скачать все прикреплённые файлы (включая добавленные в комментариях) одним архивом; доступно всем ролям; `GET /api/tasks/{id}/attachments/download`
- [ ] Принятие задачи из очереди роли: если исполнитель — роль, любой участник роли может «взять задачу себе»; `POST /api/tasks/{id}/claim`; после принятия задача пропадает из очереди роли у остальных
- [ ] Упоминания `@user` в описании и комментариях: упомянутый пользователь получает уведомление и автоматически добавляется в наблюдатели

#### FR-TASK-02.2: Списки, фильтры и представления
- [ ] Раздел «Задачи» — подраздел «Список задач» содержит группы:
  - *Входящие* — задачи пользователя, задачи на соисполнении, задачи подчинённых
  - *Исходящие* — задачи от пользователя, от подчинённых
  - *Контроль выполнения* — задачи на контроле (проверка после завершения исполнителем)
  - *Текущий контроль* — задачи на контроле (отслеживание с момента создания)
  - *Соисполнение* — задачи, в которых пользователь является соисполнителем (отображается при наличии таких задач)
  - *Мой отдел* — текущие, просроченные и выполненные задачи подчинённых; задачи, назначенные подчинёнными другим; архив; сворачивание/раскрытие блока пользователя
  - Быстрые фильтры в каждой группе: *Текущие*, *Просроченные*, *Все*
- [ ] **Периодические задачи** — отдельный подраздел: страницы «Текущие» и «Завершённые»; отображаются задачи текущего пользователя и (для руководителя) подчинённых
- [ ] **Режимы отображения** списка задач: *Таблица* (строки) и *Таблица-карточка* (строки с расширенной карточкой); каждый режим настраивается независимо; `GET /api/tasks`
- [ ] **Цветовое кодирование** задач в списке: непрочитанные — синий шрифт + синяя полоса слева (у высокоприоритетных после прочтения полоса становится красной); в работе («Работа начата») — зелёный; невозможно выполнить — оранжевый; просроченные — красный; выполненные — зелёный зачёркнутый

- [ ] Сортировка: по столбцу (в режиме «Таблица» клик по заголовку); по сохранённым фильтрам
- [ ] **Расширенный фильтр**: по исполнителю, статусу, приоритету, типу, категории, дате создания (диапазон, в т.ч. относительные даты), сроку завершения (диапазон), дате выполнения, авторам, соисполнителям, родительской задаче, признаку «Задача по процессу» (с дополнительными параметрами: процесс, состояние, название экземпляра, инициатор, ответственный, дата запуска, роль в процессе); фильтры сохраняются как именованные фильтры
- [ ] EQL-поиск: поиск задач с помощью EQL-запроса в форме расширенного поиска; `GET /api/tasks?eql=...`
- [ ] `GET /api/tasks` — список задач с параметрами фильтрации, сортировки и пагинации (cursor-based); поле `fields` для выбора возвращаемых полей
- [ ] Группировка в таблице: по исполнителю / приоритету / статусу / категории / сроку (сегодня / эта неделя / позже / просрочено)
- [ ] Экспорт списка задач в Excel / CSV: `GET /api/tasks/export`
- [ ] Счётчики задач в боковом меню: «Мои задачи (N)», «Просроченные (N)», «На согласовании (N)» — обновляются в реальном времени (SSE / WebSocket)

#### FR-TASK-02.3: Поиск, подписка, уведомления и календарь
- [ ] **Быстрый поиск**: строка поиска в разделе «Задачи» — ввод названия или его части + фильтр по автору; поиск в пределах выбранной группы задач; `GET /api/tasks?q=...`

- [ ] **Подписка на задачу**: любой пользователь с доступом к задаче может подписаться; `POST /api/tasks/{id}/watch`; отписка `DELETE /api/tasks/{id}/watch`
- [ ] Наблюдатели задачи — права на добавление: все пользователи, имеющие доступ к задаче; права на удаление: автор, исполнитель, согласующий, контролёр, администратор, пользователь с правом «Доступ ко всем задачам», руководитель (только своих подчинённых), наблюдатель (только себя и тех, кого добавил)
- [ ] Наблюдатель получает оповещения обо всех действиях с задачей; задача может не отображаться в его списке задач, но открывается по ссылке из оповещения
- [ ] Настройки уведомлений пользователя: выбор событий и каналов (in-app / email / push) для каждого типа события задачи; `PUT /api/users/me/notification-settings`
- [ ] Напоминания о сроке: за N дней/часов (настраивается пользователем) — уведомление исполнителю и наблюдателям; `POST /api/tasks/{id}/reminders`

- [ ] Интеграция с внешними календарями: экспорт задач пользователя в формате iCalendar (`GET /api/users/me/tasks.ics`); подписка внешнего клиента (Google Calendar, Outlook) на этот URL
- [ ] **Дашборд задач**: виджеты — мои открытые задачи по приоритету, динамика создания/закрытия за 30 дней, задачи по статусам; `GET /api/dashboard/tasks`

- [ ] Дерево папок с неограниченной глубиной вложенности; отображение в виде боковой панели с раскрытием/свёртыванием узлов; `GET /api/folders` (tree), `POST /api/folders`, `PUT /api/folders/{id}`, `DELETE /api/folders/{id}`
- [ ] Встроенные папки по умолчанию: **Мои документы** (документы текущего пользователя; доступ — пользователь, администратор, руководители по оргструктуре), **Избранное** (ссылки на часто используемые документы и папки; вложена в «Мои документы»), **Общие папки** (документы доступны всем; добавлять документы — только администратор или пользователи с делегированными правами), **Архив** (неактуальные документы; доступ по умолчанию только администратору)
- [ ] Главная страница раздела «Документы»: список последних просмотренных документов; быстрые переходы в «Мои документы», «Избранное», «Подписки на документы»
- [ ] Создание, переименование, перемещение (drag-and-drop или диалог) и удаление папок; перед удалением папки необходимо переместить её содержимое; права на создание папки — не ниже «Редактирование» на родительскую папку
- [ ] Иконка папки — загружается пользователем или выбирается из набора; отображается в таблице данных
- [ ] Хлебные крошки в шапке страницы папки; кликабельные переходы по уровням
- [ ] Действия панели инструментов папки: **Создать** (Документ / Папку / Папку-фильтр / Папку-действие / Ссылку на папку); **Выбранные** — групповые операции (см. FR-DOC-01.3); **Ещё** (Редактировать, В избранное, Привязать веб-документ, Отвязать документ, Вид — список/документ, Настроить доступ, Скачать вложения)
- [ ] Контекстное меню папки в таблице данных: В избранное, Создать ссылку, Переместить, Удалить папку, Права доступа, Редактировать
- [ ] Скачать вложения: скачивание текущих версий выбранных документов папки одним ZIP-архивом; общий объём не более 4 ГБ; `GET /api/folders/{id}/attachments/download`
- [ ] **Папка-фильтр** (умная папка): не является хранилищем — документы создавать в ней нельзя, физически они хранятся в других папках; отображает динамический список документов, соответствующих условиям фильтра; обозначается отдельной иконкой в таблице данных
  - Создание: из кнопки «Создать — Папку-фильтр» на панели инструментов родительской папки (права не ниже «Редактирование»), или кнопкой «Сохранить как фильтр» на форме расширенного поиска
  - Настройки (вкладка «Фильтр»): Наименование, Иконка, Родительская папка, «Отключить права доступа» (Да — показывать все подходящие документы независимо от прав; Нет — только доступные пользователю), «Выполнять поиск при открытии» (Да/Нет), «Развернуть расширенный поиск при открытии», EQL-поиск, параметры «Значения для поиска»
  - Вкладка «Регистрационная карточка»: параметры поиска по атрибутам РКК (при наличии расширения ECM+)
  - Вкладка «Поля фильтра»: настройка видимости и порядка полей расширенного поиска (доступные / выбранные элементы)
  - Вкладка «Настройка колонок таблицы»: переопределение отображаемых колонок (по умолчанию — системные)
- [ ] **Папка-действие**: ссылка-ярлык на действие из реестра действий системы — открывает диалог или выполняет активность (запуск процесса, отправка сообщения, создание задачи, открытие страницы системы и т.д.); при нажатии на папку-действие запускается выбранное действие из реестра; обозначается отдельной иконкой; создаётся через «Создать — Папку-действие»; настройки: Название, Действие из реестра (выбор через кнопку «Выбрать», в т.ч. вкладка «Дополнительно» с полным списком); `POST /api/folders` с `type: "action"` и `actionRef`
- [ ] Привязка веб-документа к папке: при открытии папки отображается содержимое привязанного веб-документа; переключение между видом «Список» (содержимое папки) и видом «Документ» (веб-документ)

#### FR-DOC-01.2: Права доступа к папкам и документам
- [ ] Уровни прав доступа (в порядке возрастания): **Просмотр** — просмотр папки/документа; **Загрузка** — скачивание документа (включает «Просмотр»); **Редактирование** — создание версий, редактирование, перемещение документов/папок, удаление папок (включает «Просмотр»); **Создание документов** — создание документов в папке и перемещение документов в неё; **Выдача прав** — изменение прав доступа пользователей с уровнем не выше своего (включает «Просмотр»); **Полный доступ** — все права; назначается автоматически автору, администратору, руководителям автора по оргструктуре
- [ ] Права задаются для пользователя, группы пользователей, элемента оргструктуры, роли, подразделения; `GET/PUT /api/folders/{id}/permissions`
- [ ] Наследование прав: вложенная папка наследует права родителя по умолчанию; флаг «Наследовать права родительской папки» снимается для переопределения; блоки «Права родительской папки» (только просмотр) и «Дополнительные права» (редактирование); при перемещении папки/документа дополнительные права не меняются, наследуемые — пересчитываются от нового родителя
- [ ] Руководитель по умолчанию имеет полный доступ к папкам и документам своих подчинённых
- [ ] Права на документ могут отличаться от прав папки: `GET/PUT /api/documents/{id}/permissions`; если для типа документа задан набор прав по умолчанию, новый документ создаётся с этим набором
- [ ] Шифрование документа: настраивается администратором в типе документа; при включении на вкладке «Доступ» в карточке документа появляется возможность зашифровать документ
- [ ] **Запрос прав доступа**: при попытке открыть недоступный объект появляется кнопка «Запросить права»; форма запроса содержит: тему, согласующего (пользователи с правом выдачи доступа), срок, запрашиваемый уровень прав, описание, вложения; назначается задача выдачи прав; исполнитель нажимает «Предоставить доступ» или «Отказать» с опциональным комментарием; результат отображается в ленте сообщений заявителя; `POST /api/access-requests`, `GET /api/access-requests/{id}`, `PUT /api/access-requests/{id}/decision`
- [ ] Проверка прав на уровне API (401/403) и на уровне UI (скрытие/блокировка элементов)
- [ ] Аудит доступа: каждый факт просмотра, скачивания, редактирования документа записывается в лог; `GET /api/documents/{id}/access-log`
- [ ] Доступ к документу, вложенному в задачу: при создании задачи с вложенным документом, если у исполнителей нет прав на документ, открывается диалог назначения прав для каждого исполнителя; аналогично при отправке на согласование/ознакомление/рассмотрение

#### FR-DOC-01.3: Групповые операции с документами
- [ ] Выбор нескольких документов в папке: чекбокс в строке (один объект), чекбокс в заголовке таблицы (все документы), кнопка раскрытия (выделить все документы / все папки / все ссылки); `Shift`/`Ctrl`-клик
- [ ] Групповые действия кнопки «Выбранные»:
  - **Отправить**: На согласование / На ознакомление / Вложить в задачу
  - **Подписать**: подписание текущих версий выбранных документов (доступно при наличии прав на подписание)
  - **Конвертировать**: преобразование выбранных документов в другой тип (по умолчанию — только администратор)
  - **Создать ссылки**: создать ссылки на выбранные документы/папки в указанной папке
  - **Удалить ссылки**: удалить выбранные ссылки на документы/папки
  - **Переместить**: переместить выбранные документы/папки/ссылки в другую доступную папку
  - **Отправить в архив**: отправить выбранные документы в папку «Архив»
- [ ] `POST /api/documents/bulk` — тело запроса: массив `ids` + `action` + дополнительные параметры действия
- [ ] Прогресс-бар при длительных операциях (массовое скачивание, конвертация) через SSE: `GET /api/bulk-operations/{operationId}/progress`

#### FR-DOC-01.4: Персональные настройки хранилища
- [ ] Страница настроек хранилища: открывается из любой папки через иконку настроек в правом верхнем углу; настройки разделены по блокам
- [ ] **Создание документов через Drag-n-drop**: тип документа (по умолчанию «Файл»); шаблон названия (из типа документа или ручной шаблон с атрибутами вида `{$FileName}`); действие при дублировании названий — «Создать с названием файла» / «Запрашивать название» / «Добавить новую версию»; `PUT /api/users/me/document-storage-settings`
- [ ] **Штрихкоды**: флаг «Печатать штрихкод после регистрации» (Да/Нет); флаг «Показывать окно печати штрихкода после регистрации» (Да — с подтверждением; Нет — автоматически); при включении в карточке документа появляется кнопка «Напечатать штрихкод» (печать через Агент)
- [ ] **Канцелярия**: флаг «Предлагать выполнить действие после регистрации документа» (Да/Нет) — при регистрации открывается окно выбора следующего действия
- [ ] **Настройки создания документа**: папка по умолчанию для созданных документов (переопределяет папку из типа документа, если в типе не задана)

### FR-DOC-02: Электронные документы

#### FR-DOC-02.1: Типы документов и карточка
- [ ] Справочник типов документов (администратор): имя, иконка, набор полей метаданных (текст, число, дата, справочник, пользователь, логический флаг), обязательность, порядок; настройка жизненного цикла, маршрутов, шаблонов названий, шаблонов версий, папки по умолчанию; `GET/POST/PUT/DELETE /api/admin/document-types`
- [ ] Создание документа несколькими способами: кнопка «Создать» в главном меню, кнопка быстрых действий, из раздела «Документы» или из папки; форма включает: выбор типа, Название (автоматическое из шаблона или ручное), Родительская папка, блок Версия (Прикрепить файл с компьютера / со сканера / Создать из шаблона, флаг «Сделать текущей»), Описание; `POST /api/documents`
- [ ] **Карточка документа** — вкладки:
  - *Общая информация*: атрибуты документа; управление статусом жизненного цикла (при включённой настройке типа); статус согласования с резолюциями; текущая версия (скачать/просмотреть/открыть в Агент); текущие задачи по документу; панель истории действий (комментарии/вложения, вопросы, действия — переключаемые секции)
  - *Предпросмотр*: содержимое текущей версии документа без скачивания (см. FR-DOC-02.3)
  - *Версии*: таблица всех версий (очерёдность, дата создания/изменения, автор, статус); добавление новой версии; действия с версией через контекстное меню; `GET /api/documents/{id}/versions`
  - *Связи*: список связанных документов с именем связи; добавление/удаление связей; `GET/POST/DELETE /api/documents/{id}/links`
  - *Доступ*: блоки «Права родительской папки» и «Дополнительные права»; управление наследованием
  - *Задачи*: список задач, в которые вложен документ; переход к задаче; действия из контекстного меню; экспорт в Excel; `GET /api/documents/{id}/tasks`
  - *История*: лог всех действий с документом (создание, редактирование, перемещение, версии, согласования, подписание, архив, конвертация, права); настройки отображения — Комментарии и вложения / Вопросы / Действия; `GET /api/documents/{id}/history`
- [ ] Действия панели инструментов карточки:
  - **Редактировать** — изменение атрибутов документа
  - **Отправить**: По маршруту (при наличии маршрутов в типе документа) / На ознакомление / На согласование / На рассмотрение (только для зарегистрированных) / На регистрацию (для незарегистрированных или при множественной регистрации) / На обсуждение (отправить сообщение с вложенным документом) / Вложить в задачу
  - **Ещё**: Добавить версию / Изменить статус (жизненного цикла) / Подписать / Зарегистрировать (создать РКК) / Редактировать РКК / Внести резолюцию / Указать факт отправки / Переименовать / Переместить / Сформировать лист согласования / Сформировать лист ознакомления / Добавить комментарий / Задать вопрос / Копировать (создать документ того же типа с предзаполненными полями) / Создать ссылку / Отправить в архив / Вернуть из архива / Удалить (только из архива) / Конвертировать / Просмотр подписок / Скачать вложения (все версии и файлы из комментариев одним архивом)
- [ ] Статусы жизненного цикла документа: настраиваются для каждого типа документа; переходы управляются через «Изменить статус»; `PUT /api/documents/{id}/status`
- [ ] Маршрут согласования документа: последовательный / параллельный / смешанный; действия: Согласовать / Отклонить / Подписать (с ЭЦП); `GET/POST /api/documents/{id}/approval`
- [ ] Обсуждение документа: «Отправить — На обсуждение» открывает форму сообщения; сообщения и ответы хранятся на вкладке «История» и панели истории; `POST /api/documents/{id}/discussions`

#### FR-DOC-02.2: Версионирование файлов
- [ ] Статусы версии: **Текущая** (актуальная, одна на документ; зелёный цвет) / **Черновик** (в стадии доработки; синий) / **Устаревшая** (серый) / **Заблокирована** (закрыта для работы; красный; указывается дата, время, пользователь)
- [ ] Добавление версии: через «Ещё — Добавить версию» или кнопку «+ Добавить» на вкладке «Версии»; варианты — файл с компьютера, со сканера, из шаблона; флаг «Сделать текущей» (без него — статус Черновик); флаг «Подписать» (устанавливает статус Текущая автоматически); `POST /api/documents/{id}/versions`
- [ ] Добавление версии через drag-and-drop: перетащить файл в окно браузера на карточке документа; несколько файлов загружаются сразу; для одного файла можно установить статус «Текущая»; иначе — Черновик
- [ ] **Изменение версии** (замена файла): заменяет файл версии на новый; предыдущий файл версии удаляется из системы (в отличие от добавления новой версии); через «Изменить версию» в контекстном меню версии
- [ ] Действия с версией (контекстное меню): **Подписать** (при наличии прав); **Выгрузить** (заблокировать версию и скачать на компьютер, статус → «Заблокирована»); **Блокировать** / **Разблокировать**; **Изменить версию**; **Сменить статус** (кроме подписанных и заблокированных); **Агент**
- [ ] Принудительное снятие блокировки администратором
- [ ] Восстановление предыдущей версии как актуальной; `POST /api/documents/{id}/versions/{versionId}/restore`
- [ ] **Агент** — работа с версиями через десктопное приложение:
  - **Блокировать и изменить**: версия блокируется, документ открывается в локальном приложении; после сохранения — выбор: «Сохранить изменения» / «Продолжить работу» / «Отменить изменения»
  - **Изменить без блокировки**: версия доступна для одновременного редактирования несколькими пользователями; сохраняется последнее изменение
  - **Новая версия (из текущей)**: открывает текущую версию без блокировки; сохранение создаёт новую версию со статусом «Черновик»
  - **Новая версия (из выбранной)**: аналогично из любой версии
  - **Блокировать и создать**: блокирует исходную версию, создаёт на её основе новую
  - **Сравнить**: сравнение двух версий .doc/.docx через Microsoft Word (открывает «Результаты сравнения»)
- [ ] Сравнение версий в браузере: для текстовых файлов — inline diff; `GET /api/documents/{id}/versions/diff?v1=...&v2=...`

#### FR-DOC-02.3: Предпросмотр, конвертация и веб-документ
- [ ] **Предпросмотр в браузере** (вкладка «Предпросмотр» в карточке и отдельное окно): поддерживаемые форматы — Word (.doc, .docx, .rtf), Visio (.vsd, .vsdx, .vss, .vst, .vsx, .vtx, .vdw, .vdx), TIFF (.tiff, .tif), текстовые (.txt, .xml), открытые форматы (.ods, .ooxml, .odt, .xps), MS Project (.mpp, .mpt), PowerPoint (.ppt, .pptx, .potx, .pot, .pps, .ppsx), PDF (.pdf), графика (.bmp, .jpg, .jpeg, .png, .gif), Excel (.xls, .xlsx, .xlsb, .ods), HTML (.html, .htm), аудио (.mp3, .aac, .ogg, .wav), видео (.mp4, .webm); форматы не из списка — кнопка «Скачать»; `GET /api/documents/{id}/versions/{versionId}/preview`
- [ ] PDF-просмотрщик: панель навигации (эскизы страниц, структура, закладки); кнопки переходов между страницами; масштаб; полноэкранный режим; закладка на текущую страницу (запоминается при следующем открытии); режимы прокрутки (вертикальная / горизонтальная / масштабируемая / одиночные / разворот); поворот страниц; инструменты «Выделение текста» и «Рука»; гиперссылки открываются в той же вкладке
- [ ] Просмотрщик изображений: масштабирование (+/-/100%), поворот ±90°; захват и перемещение мышью
- [ ] Медиаплеер: воспроизведение, регулировка громкости, перемотка, полноэкранный режим для видео
- [ ] Зашифрованные файлы (MS Word / Excel с паролем): поле ввода пароля и кнопка «Расшифровать файл» на вкладке «Предпросмотр»
- [ ] Предпросмотр в отдельном окне браузера: кнопки — Скачать файл, Распечатать (с настройками колонтитулов по браузеру), Во весь экран, Закрыть; открывается из таблицы данных и вкладки «Общая информация»; `GET /api/documents/{id}/versions/{versionId}/preview/window`
- [ ] Настройки формирования предпросмотра (административные): администратор задаёт список разрешённых форматов; при неподдерживаемом формате — уведомление
- [ ] **Конвертация типа документа** (смена типа, не формата файла): изменение типа документа в системе на другой тип; требует права на конвертацию (выдаёт администратор); для одного документа или групповая операция; процесс конвертации:
  - Шаг 1 «Атрибуты»: маппинг полей конвертируемого типа → поля конечного типа (флажок «полное соответствие» или ручной выбор)
  - Шаг 2 «РКК» (при ECM+ и наличии дела в номенклатуре): маппинг атрибутов регистрационной карточки; поле «Регистрационный номер» — флаг «Автогенерация»
  - Шаг 3 «Список документов»: просмотр конвертируемых документов и соответствий
  - Шаг 4 «Список РКК» (при ECM+): просмотр РКК
  - Перед конвертацией выполняется проверка конфликтов (ссылки на документ из других объектов); при конфликтах конвертация блокируется с перечнем ссылок
  - Результат: документ конвертирован, история фиксирует начальный и конечный тип; `POST /api/documents/{id}/convert`
- [ ] **Конвертация файла в PDF/PDF-A**: фоновая задача конвертации; DOCX/XLS/PPTX → PDF через LibreOffice; результат сохраняется как дополнительный файл карточки; `POST /api/documents/{id}/versions/{versionId}/convert-pdf`; статус конвертации через SSE
- [ ] **Веб-документ** — тип документа с HTML-содержимым вместо файла: создаётся через выбор типа «Веб-документ»; содержимое редактируется в HTML-редакторе (см. FR-DOC-02.4) или из шаблона; обозначается отдельной иконкой
  - **Простой вид**: по умолчанию после создания; панель с действиями (Редактировать, Отправить, Доступ); кнопка «Подробнее» — переход в расширенный вид
  - **Расширенный вид**: аналогичен карточкам других типов документов; дополнительное действие «Выгрузить содержимое» (скачать в формате .docx); кнопка «Простой вид» для возврата
  - Привязка к папке: при открытии папки отображается содержимое веб-документа; переключение между «Список» (содержимое папки) и «Документ» (веб-документ) через «Ещё — Вид»
  - Подписка, избранное — аналогично обычным документам
  - `GET/PUT /api/documents/{id}/web-content`
- [ ] **HTML-редактор** (для веб-документов, HTML-портлетов, карточек бизнес-правил): операции Правки (Отменить, Повторить, Вырезать, Копировать, Вставить, Выделить все, Поиск и замена); Вставить (ссылку, изображение, спецсимволы); Вид (невидимые символы, блоки, контуры, полноэкранный режим — Ctrl+Alt+F); Форматирование (полужирный, курсив, подчёркнутый, зачёркнутый, индексы); Таблицы (создание, удаление, настройка ячеек/строк/столбцов); стиль, шрифт, размер, цвет текста и фона, выравнивание, списки, отступы; просмотр и правка HTML-кода; контекстное меню (ссылка, изображение, таблица); **Добавление системных действий** → Добавить действие → выбор из реестра (вкладки «Основные» и «Дополнительно»); настройка параметров (напр., процесс и входные переменные для «Запустить процесс»); настройка внешнего вида кнопки/ссылки с предпросмотром
- [ ] **Генерация документа из шаблона**: шаблон DOCX с плейсхолдерами `{{переменная}}`; передаётся набор переменных (вручную или из переменных процесса); результат — новый файл версии документа; `POST /api/documents/{id}/generate`

#### FR-DOC-02.4: Подписки на документы
- [ ] Подписка на конкретный документ: кнопка-иконка в карточке документа; повторное нажатие — отписка; уведомления о всех изменениях приходят в ленту сообщений пользователя; `POST /api/documents/{id}/watch`, `DELETE /api/documents/{id}/watch`
- [ ] Подписка на тип документа: администратор настраивает подписку для указанных пользователей на все документы определённого типа; значок подписки в карточке такого документа отличается от значка личной подписки
- [ ] Управление подписчиками: «Ещё — Просмотр подписок» в карточке документа — список наблюдателей; кнопка «+ Добавить» для добавления; удаление через иконку; панель «Подписчики на тип документа» — только для просмотра; `GET/POST/DELETE /api/documents/{id}/subscribers`
- [ ] Уведомления: оповещения приходят в ленту сообщений (название — ссылка на карточку документа); вкладки «Общая информация» и «История» позволяют ознакомиться со всеми изменениями

#### FR-DOC-02.5: Ссылки на документы и папки
- [ ] Создание ссылки на документ: через контекстное меню в таблице, через «Ещё — Создать ссылку» в карточке, через групповую операцию «Создать ссылки»; настройки: Название ссылки (по умолчанию — имя документа), флаг «Использовать название источника» (название ссылки меняется при переименовании документа), «Поместить в папку» (по умолчанию — «Мои документы»); ссылка обозначается отдельной иконкой в таблице; `POST /api/documents/{id}/links`
- [ ] Создание ссылки на папку: через контекстное меню папки или «Создать — Ссылку на папку»; обозначается отдельной иконкой
- [ ] Групповая операция «Создать ссылки»: создать ссылки на несколько выбранных объектов в указанной папке
- [ ] Действия со ссылкой в контекстном меню: Удалить, Изменить (название), Переместить, Копировать, Агент (для ссылок на документы)

#### FR-DOC-02.6: Архив документов
- [ ] Архив хранит неактуальные документы; отображается в дереве хранилища при наличии прав доступа уровня «Просмотр» и выше
- [ ] Права для отправки в архив: **Полный доступ** на документ + права **Редактирование** на родительскую папку
- [ ] Отправка в архив: через контекстное меню в таблице, через «Ещё — Отправить в архив» в карточке, через групповую операцию; диалог подтверждения; при наличии открытых задач — выбор: «Оставить задачи как есть» / «Закрыть задачи» / «Отмена»
- [ ] Архивный документ отображается с меткой `[Архив]` в названии; в карточке — статус «В архиве» с датой/временем и автором операции; ссылки на архивный документ также содержат метку `[Архив]`; `PUT /api/documents/{id}/archive`
- [ ] Восстановление из архива: через контекстное меню, панель инструментов в карточке или групповую операцию; документ восстанавливается в исходную папку; `PUT /api/documents/{id}/restore`
- [ ] Удаление документа: возможно только из папки «Архив»; права на удаление — по умолчанию только администратор (может делегировать); через контекстное меню, карточку или групповую операцию; `DELETE /api/documents/{id}`
- [ ] Назначение прав доступа к архиву: администратор и пользователи с «Полным доступом» на архив
- [ ] Архивные документы могут отображаться в расширенном поиске при включённом параметре «Включить учёт архива»

#### FR-DOC-02.7: Интеграция с процессами
- [ ] Service Task в BPMN может создавать документ, менять его статус, менять права, генерировать файл из шаблона — через соответствующие API вызовы движка
- [ ] Изменение прав доступа к документу из процесса: `PUT /api/documents/{id}/permissions` вызывается из Service Task с параметрами из переменных процесса
- [ ] Результаты согласования документа могут быть переданы в процесс как переменные (approved/rejected + комментарий)

### FR-DOC-03: Поиск документов

#### FR-DOC-03.1: Полнотекстовый и метаданных поиск
- [ ] **Глобальный поиск**: полнотекстовый поиск по названию документа, полям метаданных и содержимому файлов (PDF, Office — через Apache Tika / Elasticsearch); тип объекта «Документ» в глобальной строке поиска; `GET /api/search?q=...&type=document`
- [ ] Поиск по содержимому файла: при загрузке файла запускается фоновая задача индексации текста; статус индексации видим на карточке (не проиндексирован / индексируется / проиндексирован)
- [ ] Форма поиска: открывается из главного меню, со страницы «Хранилище документов» (кнопка «Поиск документов»), из любой папки; быстрый поиск по названию в строке; расширенный поиск через кнопку
- [ ] **Расширенный поиск документов** — `GET /api/documents/search` — параметры:
  - `q` — полнотекстовый запрос
  - `type` — тип документа (из справочника)
  - `status` — статус (один или список)
  - `lifecycleStatus` — статус жизненного цикла документа (из справочника «Статус жизненного цикла»)
  - `author` — пользователь-автор
  - `changedBy` — пользователь, вносивший изменения
  - `folder` — конкретная папка или её поддерево (`recursive=true`; включить поиск по всем папкам)
  - `tags` — один или несколько тегов
  - `createdFrom` / `createdTo` — диапазон дат создания (точные и относительные)
  - `modifiedFrom` / `modifiedTo` — диапазон дат изменения
  - `includeArchive` — включить документы из архива в результаты (Да/Нет)
  - `archiveOnly` — поиск только в архиве (Да/Нет)
  - `includeEncrypted` — включить зашифрованные документы (Да/Нет)
  - `requiresResponse` — документы с флагом «Требует ответа» (Да/Нет)
  - `metaField[<fieldName>]` — поиск по произвольному полю метаданных типа документа
  - При наличии ECM+: атрибуты РКК, номенклатура (выбор дела)
- [ ] Действия на форме расширенного поиска: «Сохранить как фильтр» (создать папку-фильтр с текущими параметрами); «EQL-поиск» (перейти к поиску по EQL-запросам); «Скрыть/Показать поля поиска» (свернуть форму)
- [ ] **EQL-поиск документов**: синтаксис вида `type:"Договор" AND author:me AND created:<2026-01-01`; поддержка операторов AND, OR, NOT, wildcards, диапазонов; `GET /api/documents/search?eql=...`
- [ ] Сохранённые поисковые запросы: пользователь может сохранить набор фильтров как «Сохранённый поиск» (папку-фильтр) и вернуться к нему; `GET/POST/DELETE /api/users/me/saved-searches`
- [ ] Результаты поиска: список с релевантностью; подсветка совпадений в названии и сниппете содержимого; сортировка по релевантности / дате изменения / дате создания / названию
- [ ] Экспорт результатов поиска в Excel / CSV; `GET /api/documents/search/export`
- [ ] **Теги документов**: free-form теги с автодополнением; массовое добавление/удаление тегов; фильтрация по тегам в поиске и в папке; `GET /api/tags?type=document`

## 7. Трудозатраты

### FR-TIME-01: Учёт времени

#### FR-TIME-01.1: Логирование трудозатрат
- [ ] Запись трудозатраты пользователем: дата, продолжительность (чч:мм), тип работы (из справочника), описание; привязка к задаче, экземпляру процесса или документу (полиморфная связь); `POST /api/timelogs`; `GET /api/timelogs/{id}`; `PUT /api/timelogs/{id}`; `DELETE /api/timelogs/{id}`
- [ ] Таймер «в одно нажатие»: кнопка «Старт» на карточке задачи/процесса запускает отсчёт времени; «Стоп» создаёт запись трудозатраты с рассчитанной продолжительностью; одновременно может работать только один таймер у пользователя; `POST /api/timelogs/timer/start`, `POST /api/timelogs/timer/stop`
- [ ] Таблица личных трудозатрат: неделя / двухнедельный период; строки — дни, столбцы — задачи/проекты (или наоборот); ячейки редактируются инлайн; итоговые строки по дням и по задачам; `GET /api/timelogs/my?from=...&to=...`
- [ ] Справочник типов работ (администратор): название, код, признак «оплачиваемая»; `GET/POST/PUT/DELETE /api/admin/work-types`
- [ ] Нормы рабочего времени: рабочий календарь пользователя (часов в день, выходные, праздники); при заполнении таблицы подсвечивается отклонение от нормы (недоработка / переработка); `GET /api/users/{id}/work-calendar`

#### FR-TIME-01.2: Согласование трудозатрат
- [ ] Пользователь отправляет записанные трудозатраты за период (день / неделя) на согласование: `POST /api/timelogs/submit?from=...&to=...`; статусы периода: Черновик → На согласовании → Согласовано → Отклонено
- [ ] Маршрут согласования: настраивается администратором — непосредственный руководитель / менеджер проекта / произвольная цепочка; `GET/POST/PUT /api/admin/timelog-approval-routes`
- [ ] Действия согласующего: Согласовать / Отклонить + обязательный комментарий при отклонении; `POST /api/timelogs/approval/decision`
- [ ] При отклонении пользователь получает уведомление и может исправить записи и переотправить; история изменений записей хранится
- [ ] Блокировка редактирования согласованных записей; возможность разблокировки администратором (`POST /api/timelogs/unlock`)
- [ ] Уведомления: согласующему при поступлении пакета на согласование; пользователю при согласовании/отклонении; напоминание о незаполненных трудозатратах в конце рабочего дня/недели

#### FR-TIME-01.3: Отчёты и аналитика по трудозатратам
- [ ] **Сводный отчёт**: `GET /api/reports/timelogs` — параметры: `from`, `to`, `userId[]`, `departmentId[]`, `projectId`, `taskId`, `workType`, `status` (черновик/согласовано); группировка: по сотруднику / по проекту / по задаче / по подразделению / по дню / по неделе / по месяцу; агрегаты: сумма часов, количество записей, доля оплачиваемого времени
- [ ] Диаграмма загруженности: тепловая карта (пользователь × дата) с цветовым кодированием нормы; виджет на дашборде; `GET /api/reports/timelogs/heatmap`
- [ ] Отчёт «Переработки и недоработки»: сравнение факт vs. норма за период, по сотруднику; `GET /api/reports/timelogs/deviations`
- [ ] Отчёт по оплачиваемому и неоплачиваемому времени: разбивка по типам работ; `GET /api/reports/timelogs/billable`
- [ ] Экспорт отчётов в Excel (.xlsx) и CSV; `GET /api/reports/timelogs/export?format=xlsx`
- [ ] **Поиск и фильтрация**: `GET /api/timelogs` — фильтры: период, пользователь, задача, проект, тип работы, статус согласования; сортировка по дате / продолжительности; пагинация cursor-based

## 8. Календарь

### FR-CAL-01: Управление событиями

#### FR-CAL-01.1: Создание и редактирование событий
- [ ] Создание события: название (обязательно), описание (rich-text), дата/время начала и окончания, флаг «весь день», место (свободный текст + опциональные координаты), онлайн-ссылка (Zoom/Teams/Meet URL); `POST /api/calendar/events`; `PUT /api/calendar/events/{id}`; `DELETE /api/calendar/events/{id}`
- [ ] **Участники события**: приглашение конкретных пользователей и групп; статус ответа: Принято / Отклонено / Под вопросом / Не ответил; `POST /api/calendar/events/{id}/invites`; `PUT /api/calendar/events/{id}/rsvp` (пользователь меняет свой статус); организатор видит сводку ответов
- [ ] Уведомление участникам при создании/изменении/отмене события (in-app + email); при отмене — уведомление с пометкой «Событие отменено»
- [ ] **Вложения** к событию: файлы, ссылки; `POST /api/calendar/events/{id}/attachments`
- [ ] **Повестка события** (agenda): список пунктов повестки с докладчиком и продолжительностью; `GET/PUT /api/calendar/events/{id}/agenda`
- [ ] **Протокол события**: rich-text документ, создаётся прямо в карточке события или генерируется из шаблона; привязка задач, возникших на встрече; `GET/PUT /api/calendar/events/{id}/minutes`
- [ ] Переносить событие перетаскиванием на другой слот в сетке календаря (drag-and-drop); `PATCH /api/calendar/events/{id}` с новыми датами

#### FR-CAL-01.2: Периодические события
- [ ] Правила повторения (RRULE-совместимые): ежедневно / каждые N дней / еженедельно (дни недели) / ежемесячно (N-е число / Nth weekday) / ежегодно; конец серии: по дате / после N повторений / бесконечно; `POST /api/calendar/events` с полем `recurrence`
- [ ] Редактирование одного экземпляра / всей серии / этого и последующих (instance / all / following); при удалении — те же три варианта
- [ ] Исключения в серии (exdates): конкретные даты, в которые событие не происходит
- [ ] Отображение периодических событий в сетке: все экземпляры разворачиваются в заданном диапазоне отображения

#### FR-CAL-01.3: Виды отображения и общие календари
- [ ] **Виды**: День / Рабочая неделя (пн–пт) / Неделя (пн–вс) / Месяц / Повестка (список событий); переключение кнопками в шапке; `GET /api/calendar/events?from=...&to=...&calendarIds[]=...`
- [ ] **Мои календари**: личный календарь пользователя; календарь задач (события из задач с датами); календарь процессов (дедлайны экземпляров процессов); каждый отключаемый/включаемый через чекбокс в боковой панели
- [ ] **Общие / командные календари**: создаются администратором или пользователем; можно подписаться (`POST /api/calendar/subscriptions`); права: только просмотр / просмотр + добавление событий / управление; `GET/POST/PUT/DELETE /api/admin/shared-calendars`
- [ ] **Просмотр чужого календаря**: пользователь может открыть сетку коллеги (если есть права) для подбора времени встречи; режим «Совместное расписание» (overlay нескольких пользователей/ресурсов)
- [ ] **Ресурсные календари**: переговорные комнаты, оборудование; бронирование ресурса при создании события; `GET /api/resources/availability?from=...&to=...`; `POST /api/calendar/events/{id}/resources`
- [ ] **Напоминания**: настраивается на каждом событии — за N минут / часов / дней; несколько напоминаний на одном событии; каналы: in-app / email / push; `PUT /api/calendar/events/{id}/reminders`
- [ ] **Мини-календарь** (пикер дат) в боковой панели для быстрой навигации; точки-индикаторы на датах с событиями

### FR-CAL-02: Интеграция с внешними календарями

#### FR-CAL-02.1: Экспорт и двусторонняя синхронизация
- [ ] **iCalendar-фид**: `GET /api/users/me/calendar.ics?token=<personalToken>` — постоянный URL, совместимый с RFC 5545; содержит события пользователя (выбранные календари); внешний клиент (Google Calendar, Outlook, Apple Calendar) может подписаться по этому URL и получать обновления; токен генерируется в настройках профиля, может быть отозван; `POST /api/users/me/calendar-token` (генерация), `DELETE /api/users/me/calendar-token` (отзыв)
- [ ] **Экспорт диапазона в .ics файл**: `GET /api/calendar/export.ics?from=...&to=...&calendarIds[]=...` — скачивание статичного файла для однократного импорта
- [ ] **Двусторонняя синхронизация с Exchange / Microsoft 365**: подключение учётной записи пользователя через OAuth2 / EWS; синхронизация событий Exchange → BPM и BPM → Exchange; интервал синхронизации: 5 мин (настраивается); конфликты: приоритет последнего изменения (last-write-wins); `POST /api/integrations/exchange/connect`; статус синхронизации `GET /api/integrations/exchange/status`
- [ ] **Двусторонняя синхронизация с Google Calendar**: подключение через Google OAuth2 + Google Calendar API v3; аналогичная логика синхронизации; `POST /api/integrations/google-calendar/connect`; `GET /api/integrations/google-calendar/status`
- [ ] Список подключённых интеграций пользователя: `GET /api/integrations`; отключение: `DELETE /api/integrations/{id}`; логи синхронизации с отображением ошибок: `GET /api/integrations/{id}/logs`
- [ ] Маппинг статусов участников: принятые/отклонённые ответы синхронизируются между BPM и внешней системой

#### FR-REF-01.1: Конструктор типов объектов
- [ ] **Создание типа объекта-справочника**: название (единственное/множественное), код (уникальный ключ), иконка, цвет; `POST /api/admin/object-types`; `PUT /api/admin/object-types/{id}`; `DELETE /api/admin/object-types/{id}` (только если нет записей)
- [ ] **Произвольные свойства (поля)**: добавление полей к типу; имя поля, код, тип данных, обязательность, значение по умолчанию, порядок отображения; `GET/POST/PUT/DELETE /api/admin/object-types/{id}/properties`
- [ ] **Типы свойств**: Строка (одна строка), Текст (многострочный), HTML (rich-text редактор), Целое число, Дробное число, Дата, Дата и время, Интервал (временной), Да/Нет (boolean), Список (enum — ссылка на FR-REF-02), Деньги (сумма + валюта), Файл (одиночный вложенный файл), Файлы (список файлов), Ссылка на объект (связь с другим типом — полиморфная), Блок (вложенная группа свойств), URL (гиперссылка — хранится как строка, в интерфейсе отображается кликабельной ссылкой), Действие (Action — кастомная программируемая кнопка/команда, привязанная к объекту; логика кнопки задаётся в редакторе сценариев)
- [ ] **Валидация свойств**: для строк — мин./макс. длина, регулярное выражение; для чисел — мин./макс. значение; `PUT /api/admin/object-types/{id}/properties/{propId}/validation`
- [ ] **Группировка свойств на форме**: свойства объединяются в секции/вкладки; порядок настраивается drag-and-drop; `PUT /api/admin/object-types/{id}/layout`

#### FR-REF-01.2: Иерархические справочники и записи
- [ ] **Иерархия (дерево)**: тип объекта можно пометить как иерархический — у каждой записи появляется родительская ссылка `parentId`; `GET /api/objects/{typeCode}?tree=true` возвращает дерево; операции перемещения узлов `PATCH /api/objects/{typeCode}/{id}/parent`
- [ ] **CRUD записей**: `GET /api/objects/{typeCode}`; `GET /api/objects/{typeCode}/{id}`; `POST /api/objects/{typeCode}`; `PUT /api/objects/{typeCode}/{id}`; `DELETE /api/objects/{typeCode}/{id}`; soft-delete с флагом `isArchived`
- [ ] **Фильтры и представления**: пользователь сохраняет набор фильтров как «Представление»; представления бывают личные и публичные; `GET/POST/PUT/DELETE /api/objects/{typeCode}/views`; поддерживаемые операторы фильтра: равно / не равно / содержит / начинается с / больше / меньше / входит в список / пусто
- [ ] **Массовые операции**: `POST /api/objects/{typeCode}/bulk` — смена значения поля, архивирование, удаление по выборке
- [ ] **Поиск по справочнику**: полнотекстовый поиск по строковым полям; `GET /api/objects/{typeCode}?search=...`
- [ ] **Версионирование записей**: опционально для типа; при каждом изменении сохраняется снимок значений; `GET /api/objects/{typeCode}/{id}/history`

#### FR-REF-01.3: Связи между объектами
- [ ] **Типы связей**: администратор создаёт именованные типы связей между двумя типами объектов (Поставщик → Договор, Сотрудник → Проект и т.д.); кратность: один-к-одному / один-ко-многим / многие-ко-многим; `GET/POST/PUT/DELETE /api/admin/relation-types`
- [ ] **Добавление/удаление связей** на карточке объекта: `POST /api/objects/{typeCode}/{id}/relations`; `DELETE /api/objects/{typeCode}/{id}/relations/{relationId}`; на карточке — раздел «Связанные объекты» с фильтрацией по типу связи
- [ ] **Наследование типов**: дочерний тип объекта наследует все свойства родительского; дополнительные свойства добавляются только у дочернего; `PUT /api/admin/object-types/{id}/parent`; при публикации объекта-наследника требуется предварительная публикация родительского объекта

#### FR-REF-01.4: Запуск процесса из записи справочника
- [ ] На карточке записи объекта — кнопка «Запустить процесс»; список доступных процессов (тех, у которых настроена привязка к данному типу объекта); `GET /api/processes?objectType={typeCode}`; при запуске контекст экземпляра процесса автоматически получает ссылку на запись объекта
- [ ] **Журнал процессов** записи: список всех запущенных и завершённых экземпляров процессов, связанных с данной записью; `GET /api/objects/{typeCode}/{id}/process-instances`

#### FR-REF-01.5: Пространство имён (Namespace) и группировка
- [ ] **Пространство имён (Namespace)**: логическая область для изоляции объектов и перечислений одного модуля от другого; предотвращает конфликты имён при одновременной работе нескольких команд или установке нескольких приложений; каждый тип объекта и перечисление принадлежат строго одному пространству имён; `GET/POST/PUT/DELETE /api/admin/namespaces`
- [ ] **Группы объектов и перечислений**: логическая группировка типов объектов и перечислений по модулям (Сотрудники / CRM / Финансы / …); `GET/POST/PUT/DELETE /api/admin/object-groups`; в сайдбаре Дизайнера объекты отображаются сгруппированно; группы пронумерованы в соответствии с последними изменениями
- [ ] **Поиск по объектной модели**: поиск объекта или перечисления по названию или части названия в Дизайнере; очистка поля поиска возвращает полный список


### FR-REF-02: Перечисления

#### FR-REF-02.1: Управление перечислениями
- [ ] **Создание перечисления**: название, код, описание; `POST /api/admin/enumerations`; `PUT /api/admin/enumerations/{id}`; `DELETE /api/admin/enumerations/{id}` (запрещено если используется в свойствах объектов)
- [ ] **Значения перечисления**: текстовое название, код (уникальный в рамках перечисления), цвет (hex), иконка, порядок; признак «активно» (деактивированные значения не предлагаются при выборе, но старые записи с ними отображаются); `GET/POST/PUT/DELETE /api/admin/enumerations/{id}/values`
- [ ] **Использование** перечисления: поле типа «Список» у объекта ссылается на перечисление; при выборе значения отображается цвет/иконка; в фильтрах поддерживается «входит в список»
- [ ] **Импорт значений** перечисления из CSV; `POST /api/admin/enumerations/{id}/import`

## 12. Бизнес-правила

### FR-BR-01: Управление бизнес-правилами

#### FR-BR-01.1: Конструктор правил
- [ ] **Создание правила**: название, код, описание, категория; `POST /api/business-rules`; `PUT /api/business-rules/{id}`; `DELETE /api/business-rules/{id}` (только если не используется в активных процессах)
- [ ] **Условия (if-блок)**: визуальный конструктор условий — набор условных выражений с операторами AND / OR; каждое условие: поле (переменная контекста / глобальная переменная / атрибут объекта) + оператор (равно / не равно / больше / меньше / содержит / входит в список / пусто / не пусто / начинается с / заканчивается на) + значение (константа / переменная / выражение); вложенные группы условий (AND внутри OR и наоборот); `GET/PUT /api/business-rules/{id}/conditions`
- [ ] **Действия (then-блок)**: список действий, выполняемых при истинности условий; типы действий: установить значение переменной, вызвать сервис/интеграцию, отправить уведомление, запустить дочерний процесс, выбросить ошибку, записать в лог; `GET/PUT /api/business-rules/{id}/actions`
- [ ] **Else-блок** (опционально): альтернативные действия, выполняемые если условие ложно
- [ ] **Таблицы решений (Decision Table)**: матричный вид правила — строки = наборы условий, колонки = результирующие действия; режимы: «первое совпадение» / «все совпадения»; `GET/PUT /api/business-rules/{id}/decision-table`
- [ ] **Формульный редактор**: в условиях и действиях доступен ввод выражений (аналог Excel-формул) с функциями: строковые (concat, trim, substring), числовые (round, abs, min, max), даты (addDays, diffDays, now()), логические (if, and, or, not), коллекции (count, sum, filter, map)

#### FR-BR-01.2: Жизненный цикл правила
- [ ] **Версионирование**: при каждом сохранении создаётся новая версия (major.minor); активная версия помечается явно; `GET /api/business-rules/{id}/versions`; `POST /api/business-rules/{id}/versions/{v}/activate`
- [ ] **Согласование версии**: перевод версии в статус «На согласовании» → «Согласовано» / «Отклонено»; настраиваемая цепочка согласующих; `POST /api/business-rules/{id}/versions/{v}/submit`; `POST /api/business-rules/{id}/versions/{v}/approve`; `POST /api/business-rules/{id}/versions/{v}/reject`
- [ ] **Карточка правила**: отображает метаданные, текущую активную версию, список версий с историей изменений (кто изменил, когда, что изменилось — diff); `GET /api/business-rules/{id}`
- [ ] **История изменений** (audit log): каждое изменение правила логируется (пользователь, дата, тип изменения); `GET /api/business-rules/{id}/history`
- [ ] **Тестирование правила**: ввод тестовых входных данных (JSON) → получение результата выполнения (выходные значения + трассировка какие условия сработали); `POST /api/business-rules/{id}/test`

#### FR-BR-01.3: Исполнение правил
- [ ] **Задача исполнения правила** в BPMN-процессе: Service Task с типом «Бизнес-правило»; привязка к конкретному правилу и версии (или «активная»); входные/выходные маппинги переменных процесса
- [ ] **Ручной запуск** (ad-hoc): запуск правила с произвольными входными данными через UI-форму; `POST /api/business-rules/{id}/execute`
- [ ] **Авто-запуск по событию**: триггеры — создание/изменение объекта справочника, наступление даты, webhook; конфигурируется в настройках правила; `GET/POST/PUT/DELETE /api/business-rules/{id}/triggers`
- [ ] **Журнал исполнений**: лог каждого вызова правила (дата, инициатор, входные данные, результат, время выполнения, ошибки); `GET /api/business-rules/{id}/executions`

#### FR-BR-01.4: Глобальные переменные
- [ ] **Создание глобальной переменной**: имя, код, тип данных (строка / число / дата / Да/Нет / JSON), начальное значение, описание; `GET/POST/PUT/DELETE /api/global-variables`
- [ ] **Использование** в правилах и BPMN-процессах: переменная доступна в формулах и условиях через синтаксис `$varCode`; изменение значения из действия правила или сервисной задачи
- [ ] **Права доступа к переменным**: какие роли/правила могут читать / изменять конкретную переменную; `PUT /api/global-variables/{id}/permissions`
- [ ] **Аудит изменений**: лог каждого изменения значения переменной (кто, когда, старое/новое значение); `GET /api/global-variables/{id}/history`

## 13. Конструктор интерфейсов (Low-code UI)

### FR-UI-01: Конструктор форм

#### FR-UI-01.1: Редактор формы
- [ ] **Рабочая область редактора**: drag-and-drop холст; панель компонентов слева; панель свойств выбранного компонента справа; `GET/POST/PUT/DELETE /api/ui/forms`; навигация по вложенности элементов на верхней панели; клавиши PageUp/PageDown для перехода между элементами, стрелка вправо — к дочерним, стрелка влево — на уровень вверх, Escape — снять выделение; дерево всех элементов с поиском по названию
- [ ] **Системные формы**: формы, созданные разработчиками системы; нельзя удалить, но можно изменить; **Пользовательские формы**: созданы и настроены для конкретных потребностей; можно изменять и удалять; Конструктор форм позволяет: моделировать формы создания/редактирования/просмотра объектов; изменять существующие формы объектов; моделировать формы задач в процессах; изменять формы переходов между задачами; моделировать форму стартового события
- [ ] **Источники элементов для формы**: Свойства — свойства объекта (для форм объектов) или контекстные переменные процесса (для форм задач); созданные свойства формы (форм-специфические поля); Сценарии — клиентские и серверные C#-сценарии в встроенном редакторе; Компоненты — системные (основные и дополнительные) и пользовательские
- [ ] **Компоненты — разметка (контейнеры)**: Колонка (настройка ширины: 1/4, 1/3, 1/2, Авто), Строка свойства (контейнер из двух ячеек: заголовок + контент — для кастомного представления свойства), Размещение в строку (flex-row; только 2-й рантайм), Карточка (заголовок + контент + подвал; размер, тень, ярлык с цветом; только 2-й рантайм), Панель (collapsible; настройки заголовка, стиль, счётчик), Вкладки (обычные и динамические с источником данных; счётчик), Меню с формой (горизонтальные вкладки или выпадающее в выдвижной панели; только 2-й рантайм)
- [ ] **Компоненты — всплывающие (синяя граница)**: Модальное окно (открывается по событию; Скрыть при пустых условиях = отображается по умолчанию; настройка размера, удаление контента при закрытии; только 2-й рантайм), Выдвижная панель (расположение: Справа/Снизу/Слева/Сверху; затемнение фона; только 2-й рантайм), Окно с подсказкой (кнопка + область содержимого; только 2-й рантайм)
- [ ] **Компоненты — ввод данных**: Текстовое поле, Многострочный текст, Rich-text редактор (HTML), Число, Дата, Дата и время, Переключатель (Да/Нет), Выпадающий список (enum / фиксированные значения), Мультивыбор, Поиск по объекту (lookup с автодополнением), Загрузка файла(ов), Подпись (signature pad), **Группа ввода** (значение свойства + кнопки в один ряд; только 2-й рантайм)
- [ ] **Компоненты — отображение**: Текст (статичный / шаблонный; тип текста: Основной, Подзаголовок, Заголовок элемента, Служебный и др.; цвет), **Кнопка** (действие при нажатии из реестра или клиентский сценарий; стиль, размер, форма; валидация при нажатии; поддержка вложенных кнопок — выпадающее меню), Изображение (встроенная иконка или URL; размер, скругление; только 2-й рантайм), Разделитель (горизонтальная линия; только 2-й рантайм), Счётчик (badge с текстом/формулой, цвет; только 2-й рантайм), **Информационный блок** (уведомление-тип: Информация/Положительный/Предупреждение/Ошибка; закрываемый; только 2-й рантайм)
- [ ] **Компоненты — динамические (красная граница)**: Таблица (источник: Объект или DTO тип Список; столбцы + динамические колонки; выбор элементов; группировка, сортировка, пагинация; только 2-й рантайм), Список связанных объектов (связь между объектами в таблице; фильтр EQL; выбор элементов; сортировка/группировка/пагинация; экспорт Excel), Дерево (иерархия: свойство-ключ + свойство-ключ родителя; выбор элементов; Свернуть/Развернуть всё; только 2-й рантайм), Динамический список (отображение коллекции: Объект, Список простых типов или DTO; каждый элемент по шаблону; только 2-й рантайм), **Информационная консоль** (список уведомлений/ошибок/предупреждений из DTO; сортировка, закрытие, очистка; только 2-й рантайм)
- [ ] **Компоненты — визуализация**: Линейная диаграмма (ось X/Y; идентификатор и подпись линии; легенда; масштабирование; виды сглаживания; только 2-й рантайм), Круговая диаграмма (pie/donut; виды: Кольцо/Полукольцо/Круг/Полукруг; источник: Объект, DTO или Пользовательский список; только 2-й рантайм), Шаги (горизонтальные/вертикальные; активный шаг через контекстную переменную; только 2-й рантайм), Скоринг-панель (панель проверки дубликатов объекта; только 2-й рантайм)
- [ ] **Компоненты — дополнительные**: Результат валидации формы (список ошибок при проверке; требует настройки «Валидация при нажатии» на кнопке; только 2-й рантайм), Форма объекта (встраивает форму объекта в интерфейс; тип отображения: Создание/Редактирование/Просмотр/Кастомное; только 2-й рантайм), Древовидная структура (структура связей между объектами; ключ + ключ родителя; только 2-й рантайм), Перетаскивание (drag-and-drop элементов в UI; идентификатор скоупа; действие при начале/вставке; только 2-й рантайм), Контейнер для свойств по умолчанию (автодобавляет ещё не размещённые свойства объекта; опция «только пользовательские»; только формы объекта; на форме — один экземпляр), **Область для вставки компонента** (slot внутри пользовательского компонента для вложения других компонентов; флажок «Основная область»; только 2-й рантайм)
- [ ] **Пользовательские компоненты на форме**: добавляются через палитру компонентов; граница зелёного цвета; настройки: видимость, только на чтение, тип загрузки (Синхронная/Ленивая/При отображении), значения входных/выходных свойств; вкладка «Отображение» (скрывать на основной форме / скрывать во всплывающих окнах); кнопка перехода к редактированию компонента
- [ ] **Настройка элемента на форме**: вкладка «Основные» (скрыть, только на чтение, обязательность); вкладка «Дополнительные» (видимость пустых элементов: Наследовать/Показывать/Скрывать); вкладка «Системные» (имя элемента, уникальный идентификатор); вкладка «Отображение» (скрывать на основной форме / во всплывающих окнах); вкладка «События» (действия при наведении, уходе курсора, Enter, Escape — только для пользовательских и обновлённых системных форм)
- [ ] **Копирование / вставка / дублирование** компонента на холсте (в том числе на другую форму); удаление через кнопку, через контекстный breadcrumb на верхней панели, через клавишу Delete; при удалении свойства объекта/контекстной переменной — удаляется только с формы, не из системы; отмена/повтор действий (undo/redo)

#### FR-UI-01.2: Свойства формы и их настройки

- [ ] **Создание свойства формы**: контекстная переменная, используемая только для отображения (не хранит данные в БД); создаётся в разделе «Свойства» конструктора; вкладка «Основные настройки»: Отображаемое имя, Тип, Обязательность, Описание; вкладка «Дополнительно»: Только для веб-части (только клиентские сценарии), Вычисляемое значение (формула/функция; доступно при «Только для веб-части»), Загружать данные с сервера (EQL-запрос), Ширина колонки в таблице (пикселей), Расширенные настройки (уникальное имя для сценариев); **Внимание:** свойства формы можно редактировать и удалять, свойства объекта и контекстные переменные — нельзя
- [ ] **Добавление частей свойства на форму**: каждое свойство можно перенести целиком или только его часть — Название (только метка), Значение (только поле ввода/просмотра), Описание; части размещаются независимо и настраиваются отдельно
- [ ] **Настройки свойства на форме** — вкладка «Основные»: Скрыть (условие), Только на чтение (или наследовать от формы), Имя свойства (только просмотр), Обязательно, Название на форме, Заполнитель (для типов Текст/Строка), Описание на форме, Подсказка (иконка «?»), Поле ввода блокировано, Запретить создание объекта на форме, Сценарий при изменении значения, Скрыть текст валидации, Сценарий валидации; изменения применяются только на текущей форме
- [ ] **Настройки свойства на форме** — вкладка «События» (только пользовательские и обновлённые системные формы): Передавать фокус ввода (Не передается / По умолчанию / При отображении), Действие при наведении, Действие при уходе курсора, Действие при Enter, Действие при Escape, Действие при фокусе (не для типов Объект/DTO), Действие при вводе (не для типов Объект/DTO)
- [ ] **Настройки свойства на форме** — вкладка «Вложенные свойства» (только для свойств формы с типом Объект): список вложенных свойств с возможностью открыть настройки, задать обязательность/редактируемость, удалить
- [ ] **Настройки свойства на форме** — вкладка «Системные»: Имя элемента (уникальное, используется в сценариях), Уникальный идентификатор (копируется); вкладка «Дополнительные (Блок)» (только для типа Блок): клиентский сценарий при загрузке формы блока, префикс таблицы (1-й рантайм)
- [ ] **Поиск свойств**: строка поиска по названию; для вложенных — шаблон `<имя родителя><имя вложенного>`; одно и то же свойство типа «Блок» можно добавить на форму 1-го рантайма только один раз; вложенные свойства типа «Блок» нельзя выносить на форму задачи

#### FR-UI-01.3: Настройки формы

- [ ] **Настройки формы — вкладка «Основные»**: Уникальный идентификатор (копируется, изменить нельзя), Отображаемое имя, Версия рантайма, Только для чтения (Включено / Наследование / Не установлен)
- [ ] **Настройки формы — вкладка «Общие события»** (только 2-й рантайм): клиентские lifecycle-сценарии формы — Клиентский сценарий при инициализации (IOnInit), Клиентский сценарий при загрузке (IOnLoadAsync), Клиентский сценарий для валидации (IOnValidate), Клиентский сценарий при закрытии (IOnDestroyAsync); для каждого сценария автоматически добавляется реализация соответствующего интерфейса
- [ ] **Настройки формы — вкладка «События»**: Сценарий при загрузке формы (клиентский или серверный; серверный не рекомендуется на 2-м рантайме), Сценарий валидации (для отображения результата вынесите на форму компонент «Результат валидации формы»)

#### FR-UI-01.4: Формулы и динамическое поведение
- [ ] **Режимы задания настроек компонента**: значение (стандартное заполнение), свойство (передача из другого источника), функция (C#-метод из сценариев — выбор из списка, переход в редактор), формула (простое выражение на C#; нет гарантии null-safety — рекомендуется использовать `?.` и тернарный оператор)
- [ ] **Формульный редактор** с подсветкой синтаксиса и автодополнением; обращение к контексту: `Context.MyProperty`; обращение к элементу коллекции: `Item.MyProperty` (в таблице, динамическом списке, списке связанных объектов); доступны все функции из FR-BR-01.1
- [ ] **Связанные поля**: изменение значения одного поля автоматически пересчитывает другие (реактивность); сценарий `onChange` привязывается к полю
- [ ] **Условная видимость**: выражение-условие на компоненте → компонент скрывается/показывается без перезагрузки страницы
- [ ] **Результат валидации формы**: сценарий валидации + компонент «Результат валидации формы» (выводит все ошибки); кнопка с флажком «Валидация при нажатии» — действие выполняется только при прохождении проверки; суммарный статус формы `isValid`

#### FR-UI-01.5: Публичные клиентские сервисы
- [ ] **IClipboardService**: копирование текста в буфер обмена (`CopyToClipboard(string text)`) и чтение (`PasteFromClipboard()`)
- [ ] **ILoaderService**: отображение визуального загрузчика (`Show(key, name, permanent)` / `Hide(key)`); рекомендуется использовать в паре с try-catch, чтобы загрузчик не завис при ошибке
- [ ] **INotificationService**: всплывающие уведомления четырёх типов — `Error`, `Success`, `Warning`, `Info`; каждый метод принимает `title`, `message`, `autoClosed`
- [ ] **IRedirectService**: `Redirect(url)`, `Reload()`, `OpenInNewTab(url)`, `OpenInNewWindow(url)`, `GetCurrentUrl()`
- [ ] **Инъекция зависимостей**: сервисы получаются через конструктор класса-контроллера компонента (`ComponentController`); все четыре сервиса можно использовать одновременно

#### FR-UI-01.6: Стандартные элементы интерфейса
- [ ] **Вопрос-Ответ (Q&A)**: виджет «задать вопрос» на карточке объекта — пользователь задаёт вопрос, адресует конкретному пользователю или группе, получает уведомление об ответе; `POST /api/objects/{typeCode}/{id}/questions`; `GET /api/objects/{typeCode}/{id}/questions`; `POST /api/objects/{typeCode}/{id}/questions/{id}/answer`
- [ ] **Выбор элементов**: универсальный компонент выбора нескольких значений из списка (чекбоксы, строка поиска, пагинация); используется в форм-полях с мультивыбором объектов, ролей, категорий
- [ ] **Выбор пользователей**: специализированный компонент выбора пользователей с поиском по ФИО, подразделению, должности; поддерживает одиночный и множественный выбор; отображает фото и должность в выпадающем списке
- [ ] **Выбор папок**: компонент навигации по дереву папок документов для выбора целевой папки (при перемещении документа, настройке прав и т.д.)
- [ ] **Выбор объектов для назначения прав доступа**: компонент выбора субъектов ACL (конкретный пользователь / роль / подразделение / все авторизованные) — используется во всех диалогах управления правами
- [ ] **Таблица данных (Data Table)**: переиспользуемый компонент таблицы с пагинацией (cursor-based), сортировкой по колонкам, поиском, inline-фильтрами по колонкам, настройкой видимых колонок, выбором строк (чекбоксы), экспортом в Excel/CSV; применяется в списках задач, документов, пользователей, записей справочников
- [ ] **Файл (File component)**: компонент загрузки и отображения вложений — drag-and-drop загрузка, прогресс-бар, предпросмотр изображений/PDF прямо в компоненте, удаление, скачивание; поддерживает одиночный файл и список файлов; `POST /api/attachments` (multipart upload); применяется во всех формах с вложениями
- [ ] **Панель истории действий (History Panel)**: унифицированный компонент хронологического лога изменений объекта — аватар актора, тип действия (создан / изменён / статус изменён / комментарий / вложение добавлено), изменённые поля с указанием старого и нового значения, временна́я метка; подгрузка по «Показать ещё» (cursor-based пагинация); применяется на всех карточках объектов

### FR-UI-02: Конструктор страниц, портлетов и компонентов

#### FR-UI-02.1: Страницы
- [ ] **Конструктор страниц**: страница собирается drag-and-drop из компонентов и свойств; страница может принимать **входные свойства** (флажок Входное); `POST /api/ui/pages`; `PUT /api/ui/pages/{id}`; `DELETE /api/ui/pages/{id}`
- [ ] **Статус публикации в дереве**: чёрный шрифт — опубликованы; жирный чёрный — опубликованы с неопубликованными изменениями; голубой — созданы, но не настроены; жирный голубой — созданы с изменениями, не опубликованы
- [ ] **Группировка**: создание вложенных папок (Группа) с отображаемым именем и наименованием (только латиница/цифры/\_; после создания не изменяется); страницы/портлеты/компоненты создаются внутри групп
- [ ] **Панель инструментов на странице**: кнопка «+ Добавить панель инструментов» → добавляет компонент «Группа» из группы «Панель инструментов»; внутрь добавляются кнопки и разделители; поддержка выпадающего меню (кнопки внутри кнопки)
- [ ] **Публикация страницы**: «Сохранить и опубликовать» — страница становится доступна пользователям; созданные страницы конфигурации появляются в папке «Страницы конфигурации» при настройке главной страницы пользователя
- [ ] **Привязка данных страницы**: страница может принимать параметры маршрута через входные свойства; компоненты читают параметры через контекст
- [ ] **Страницы-дашборды**: страница без параметров маршрута; назначается как «главная страница» для роли или пользователя; `PUT /api/admin/home-pages`

#### FR-UI-02.2: Портлеты
- [ ] **Портлет**: подключаемый компонент интерфейса с актуальной тематической информацией; создаётся в разделе «Интерфейсы» дизайнера
- [ ] **Раздел «Настройки портлета»**: дополнительная панель (только для портлетов) — свойства портлета, которые пользователь настраивает для себя; создаются только для серверных сценариев (нельзя «Только для веб-части»); значения сохраняются индивидуально для каждого пользователя; нельзя использовать функции и формулы при создании
- [ ] **Публикация портлета**: «Сохранить и опубликовать»; опубликованные портлеты доступны пользователю при настройке главной страницы

#### FR-UI-02.3: Пользовательские компоненты
- [ ] **Создание пользовательского компонента**: комбинация встроенных компонентов, сохраняемая для повторного использования; создаётся в разделе «Интерфейсы»; доступно только во 2-м рантайме
- [ ] **Входные свойства (Input)**: создаются в разделе «Свойства» конструктора компонента с флажком «Входное»; задаются при размещении компонента — значение входного свойства задаётся снаружи
- [ ] **Выходные свойства (Output)**: флажок «Выходное»; типы Действие и Внутренняя DTO нельзя сделать выходными; при размещении компонента — выбрать свойство для хранения выходного значения или функцию
- [ ] **Область для вставки (Slot)**: компонент «Область для вставки компонента» внутри пользовательского компонента создаёт slot для вложения другого компонента при использовании; можно добавить несколько областей; флажок «Основная область»
- [ ] **Кастомное отображение (Custom Display)**: в разделе «Сценарии» → настройка кастомного отображения через C#-сценарий (включается администратором); для неопубликованных компонентов — вкладка «Отображение»
- [ ] **Публикация и использование**: опубликованные компоненты появляются в палитре «Компоненты» в конструкторе форм, страниц и портлетов; при изменении компонента необходимо опубликовать новую версию

#### FR-UI-02.4: Пользовательский список
- [ ] **Пользовательский список**: набор данных с произвольным составом свойств; не ограничен одним типом данных; создаётся в разделе «Свойства» конструктора страниц/портлетов/компонентов; используется как источник данных для диаграмм и таблиц
- [ ] **Поля при создании**: Отображаемое имя, Источник данных (тип объекта), Список вложенных свойств объекта, Список свойств для группировки, Список свойств агрегации (среднее/максимум/минимум/сумма/счётчик), Условие выборки на EQL
- [ ] **Использование**: как источник данных компонента Круговая диаграмма или Таблица; в компоненте указывается пользовательский список в поле «Источник данных» или «Контекстная переменная»

#### FR-UI-02.5: Настройки страниц, портлетов и компонентов
- [ ] **Настройки — вкладка «Основные»**: Уникальный идентификатор (копируется), Отображаемое имя, Описание, Серверный сценарий при загрузке, Только на чтение
- [ ] **Настройки — вкладка «Дополнительно»**: уникальное имя (транслитерация; можно изменить до первой публикации, после — только скопировать), Версия рантайма
- [ ] **Настройки — вкладка «События»** (только 2-й рантайм): lifecycle-сценарии — IOnInit, IOnLoadAsync, IOnValidate, IOnDestroyAsync; автоматически добавляется реализация соответствующего интерфейса
- [ ] **Настройки — вкладка «История версий»**: список опубликованных версий (номер, статус: Текущая / Устаревшая, дата публикации, комментарий); из устаревшей версии можно загрузить черновик для внесения изменений

#### FR-UI-02.6: Панель инструментов (Toolbar) в конструкторе интерфейсов
- [ ] **Панель инструментов на странице**: группа компонентов «Панель инструментов» (доступна только при моделировании страниц; только 2-й рантайм); компонент **Группа** — объединяет кнопки (внутрь добавляются Кнопки и Разделители); поддержка выпадающего меню (кнопки вложены в кнопку)
- [ ] **Компонент «Разделитель» в панели инструментов**: горизонтальная линия для разделения пунктов выпадающего меню внутри кнопки
- [ ] **Действия кнопки на странице**: «Запустить процесс» (выбор процесса, входные переменные, опция отключить перенаправление), «Открыть форму» — Форма объекта (заголовок окна, контекстная переменная, тип формы, форма) или Страница (заголовок, опубликованная страница); условие видимости/доступности кнопки (функция/формула)

### FR-UI-03: Структуры обмена данными (интерфейсные DTO)

> Интерфейсные DTO — для передачи данных между клиентом и сервером в формах/страницах/портлетах/компонентах. Экземпляры не хранятся в БД, наследование структур не поддерживается. Доступны только во 2-м рантайме.

#### FR-UI-03.1: Глобальные структуры

- [ ] **Создание глобальной структуры**: в разделе «Интерфейсы» дизайнера → папка «Структуры обмена данными»; кнопка «+ Добавить» → Структура обмена данными, или через папку; задать название, группу → Создать
- [ ] **Карточка структуры**: создание свойств (Отображаемое имя, Тип, Описание; уникальное имя для API и сценариев); добавление вложенных структур; Сохранить / Сохранить и опубликовать
- [ ] **Настройки глобальной структуры** — вкладка «Основные»: Уникальный идентификатор, Отображаемое имя, Описание, Пространство имён (автоматически, только для чтения, используется в сценариях); вкладка «Дополнительно»: уникальное имя (автоматически, только для чтения после публикации)
- [ ] **Настройки глобальной структуры** — вкладка «Таблица»: настройки отображения свойства типа DTO с вид «Блок» — Количество записей на странице, Показывать заголовок, Разрешить настройку/изменение ширины/позиции колонок, Разрешить сортировку и группировку (с умолчаниями), Разрешить добавление (всплывающее окно / таблица), Редактирование (всплывающее / таблица / оба), Удаление

#### FR-UI-03.2: Внутренние структуры

- [ ] **Создание внутренней структуры**: в конструкторе интерфейсов страницы/портлета/компонента → кнопка «Структуры» на левой панели → «+ Создать»; недоступны при работе с формами
- [ ] **Вкладки внутренней структуры**: Общие настройки (название, описание), Дополнительно (уникальное имя — можно изменить до публикации), Свойства (аналогично глобальным), Таблица (аналогично глобальным)
- [ ] **Использование DTO как тип свойства**: при создании свойства формы/страницы/компонента выбрать тип — глобальная или внутренняя DTO; Тип связи: Одиночная (один экземпляр) / Список — Стандартный (выпадающий список) или Блок (таблица); Элементы для выбора из списка (сценарий)

### FR-UI-04: Глобальные функции

- [ ] **Глобальные функции**: серверные C#-функции, доступные в настройках компонентов всех форм/страниц/портлетов; создаются в разделе «Интерфейсы» дизайнера → папка «Функции»; доступны только во 2-м рантайме
- [ ] **Создание функции**: кнопка «+ Добавить» → Функция, или через папку «Функции»; задать название, группу → Создать → откроется карточка → переход в редактор сценариев
- [ ] **Настройки функции** — вкладка «Основные»: Уникальный идентификатор, Отображаемое имя, Описание, Возвращаемый тип, Тип входного параметра (не более одного; для нескольких — использовать DTO); вкладка «Дополнительно»: уникальное имя (нельзя изменить после публикации)

## 14. Интеграции

### FR-INT-01: Модули интеграции

#### FR-INT-01.1: Конструктор модулей

- [ ] **Аутентификация модуля**: без аутентификации / Basic (login+password) / API Key (header/query-param) / Bearer Token (статичный) / OAuth2 Client Credentials (автоматическое обновление токена) / OAuth2 Authorization Code / mTLS (клиентский сертификат); `PUT /api/integrations/modules/{id}/auth`
- [ ] **Методы модуля (Operations)**: HTTP-метод + путь + параметры (path / query / header / body); маппинг входных и выходных параметров на DTO; `GET/POST/PUT/DELETE /api/integrations/modules/{id}/operations`

- [ ] **Версионирование модуля**: каждый модуль имеет версии; процессы/правила ссылаются на конкретную версию или «стабильную»; `POST /api/integrations/modules/{id}/versions/{v}/publish`
- [ ] **Публикация и использование в процессах**: опубликованный модуль доступен как Service Task в BPMN-редакторе; поле выбора модуля + операции + маппинг переменных

#### FR-INT-01.2: Структуры обмена данными (DTO)
- [ ] **Глобальные DTO**: описание структуры данных (имя, поля с типами) доступно для повторного использования в разных модулях и правилах; `GET/POST/PUT/DELETE /api/integrations/dto`
- [ ] **Внутренние DTO модуля**: структуры, видимые только внутри конкретного модуля

- [ ] **Маппинг полей**: визуальный маппер (дерево источника → дерево назначения), поддержка трансформаций (выражения на каждом маппинге)

### FR-INT-02: Очереди сообщений

#### FR-INT-02.1: Управление очередями
- [ ] **Создание типа сообщения**: имя, код, DTO (схема payload); `POST /api/messaging/message-types`; `PUT /api/messaging/message-types/{id}`
- [ ] **Встроенная очередь**: каждый тип сообщения имеет очередь FIFO; сообщения хранятся в БД; `POST /api/messaging/queues/{typeCode}/publish` — отправить сообщение в очередь; `GET /api/messaging/queues/{typeCode}/messages` — просмотр накопившихся сообщений
- [ ] **Отправка из процесса**: BPMN Send Task с типом «Сообщение» → выбор типа сообщения + заполнение payload из переменных процесса
- [ ] **Ожидание сообщения**: BPMN Receive Task — процесс приостанавливается до поступления сообщения заданного типа (с опциональной корреляцией по ключу — например `correlationKey = orderId`); при получении — продолжение процесса с payload сообщения как переменными
- [ ] **Корреляция сообщений**: при публикации сообщения указывается корреляционный ключ; брокер сопоставляет с ожидающими экземплярами процессов; `POST /api/messaging/queues/{typeCode}/publish` с полем `correlationKey`
- [ ] **Dead Letter Queue**: сообщения, которые не удалось доставить после N попыток, перемещаются в DLQ; `GET /api/messaging/dlq`; ручная повторная отправка: `POST /api/messaging/dlq/{id}/retry`
- [ ] **Мониторинг очередей**: глубина очереди, среднее время ожидания, количество ошибок; `GET /api/messaging/stats`

#### FR-INT-02.2: Типы внешних очередей (Enterprise-редакция)
- [ ] **RabbitMQ**: подключение к серверу RabbitMQ (хост, порт, виртуальный хост, имя exchange, имя очереди, логин, пароль, таймаут); настройка TLS-соединения (клиентский сертификат PKCS12, пароль сертификата, имя сервера для проверки SAN/CN, опция клиентской верификации)
- [ ] **MSMQ (Microsoft Message Queuing Services)**: строка подключения (форматы: `.\\private$\\queue_name` для локальной частной очереди; `FormatName:DIRECT=OS:server\\QueueName` для общей очереди на удалённом сервере; `FormatName:DIRECT=OS:server\\private$\\QueueName` для частной удалённой); подключение выполняется от имени учётной записи Windows, под которой запущен сервер системы
- [ ] **JMS (Java Message Service)**: подключение к серверу WebLogic JMS (адрес и порт в формате `t3://host:port`; имя очереди; ConnectionFactory — объект конфигурации соединения на стороне JMS-сервера; логин и пароль)
- [ ] Раздел «Очереди сообщений» в Дизайнере: таблица всех настроенных очередей; создание кнопкой «+Очередь» с выбором типа; редактирование по клику на название; удаление через иконку

- [ ] **Авто-пересчёт**: при изменении значения любого поля, участвующего в критериях, балл пересчитывается автоматически (асинхронно)

#### FR-RPT-02.3: Глобальные настройки скоринга

- [x] **Сотрудники**: связь пользователь ↔ организация — уникальная пара; сотрудник создаётся из карточки пользователя и обязательно привязывается к организации; один пользователь может быть сотрудником нескольких организаций; `GET /api/admin/employees`; `POST /api/admin/employees`; `PUT /api/admin/employees/{id}`; `DELETE /api/admin/employees/{id}`; `GET /api/admin/users/{id}/employees`
- [x] **Подразделения (иерархия)**: дерево подразделений; CRUD; перемещение узлов; `GET /api/admin/departments?tree=true`; `POST /api/admin/departments`; `PUT /api/admin/departments/{id}`; `PATCH /api/admin/departments/{id}/parent`
- [ ] **Должности**: список должностей организации; привязка должности к пользователю; `GET/POST/PUT/DELETE /api/admin/positions`; `PUT /api/admin/users/{id}/position`
- [ ] **Назначение в подразделение**: пользователь может состоять в одном основном и нескольких дополнительных подразделениях; `PUT /api/admin/users/{id}/departments`
- [ ] **Замещения**: назначение замещающего на период отсутствия основного исполнителя; типы: полное замещение (все задачи), частичное (по типу процесса); `GET/POST/PUT/DELETE /api/admin/substitutions`; задачи автоматически переназначаются / дублируются замещающему

#### FR-ADM-01.3: Роли и права
- [ ] **Системные роли**: Администратор системы / Разработчик конфигурации / Аналитик / Пользователь / Гость; права определены фиксированно
- [ ] **Кастомные роли**: название, описание; набор прав из матрицы функциональных разрешений; `GET/POST/PUT/DELETE /api/admin/roles`
- [ ] **Матрица разрешений**: разрешения сгруппированы по модулям (Задачи, Документы, CRM, HR, Справочники, Отчёты, Администрирование, …); каждое разрешение — операция (Просмотр / Создание / Редактирование / Удаление / Запуск / Экспорт); `GET /api/admin/permissions`; `PUT /api/admin/roles/{id}/permissions`
- [ ] **Назначение роли пользователю**: пользователь может иметь несколько ролей; права суммируются; `PUT /api/admin/users/{id}/roles`
- [ ] **Ролевые группы**: группа пользователей для быстрого назначения роли сразу всем; `GET/POST/PUT/DELETE /api/admin/groups`; `PUT /api/admin/groups/{id}/users`
- [ ] **Контекстные права на объекты**: для типов объектов справочника — возможность ограничить просмотр/редактирование по фильтру (напр., пользователь видит только «свои» записи по `ownerId = currentUser.id`); `PUT /api/admin/object-types/{id}/row-level-security`
- [ ] **Права доступа к модулю «Бизнес-процессы»**: матрица разрешений для работы с BPM — запуск процессов (по каждому определению), просмотр монитора процессов, управление экземплярами (форсированное завершение, отладка), права на моделирование и публикацию в дизайнере; `GET/PUT /api/admin/permissions/bpm`; настройка видимости конкретных определений процессов для конкретных ролей

#### FR-ADM-01.4: Настройки авторизации
- [ ] **Локальная аутентификация**: логин + пароль; политика паролей (мин. длина, требования к сложности, срок действия, история паролей); `PUT /api/admin/auth/password-policy`
- [ ] **SSO (SAML 2.0 / OIDC)**: настройка Identity Provider (метаданные / discovery URL); маппинг атрибутов SAML/OIDC → поля пользователя; `PUT /api/admin/auth/sso`; `GET /api/admin/auth/sso/metadata` (SP-метаданные для регистрации в IdP)
- [ ] **LDAP-аутентификация**: делегирование проверки пароля в LDAP-каталог; настраивается в рамках модуля LDAP-синхронизации
- [ ] **Двухфакторная аутентификация (2FA)**: TOTP (Google Authenticator / Authy); SMS OTP; администратор может обязать использование 2FA для роли; `PUT /api/admin/auth/mfa-policy`; `POST /api/users/me/mfa/enroll`; `POST /api/users/me/mfa/verify`
- [ ] **API-токены**: создание именованных токенов для сервисных интеграций без участия пользователя; ограниченные права на токен; `GET/POST/DELETE /api/admin/api-tokens`
- [ ] **Клиентские сертификаты (mTLS)**: загрузка доверенного CA; проверка сертификата при подключении; `PUT /api/admin/auth/mtls`

#### FR-ADM-01.5: Управление сессиями
- [ ] **Список активных сессий**: таблица (пользователь, IP, User-Agent, дата начала, последняя активность); `GET /api/admin/sessions`
- [ ] **Принудительное завершение сессии**: отдельной `DELETE /api/admin/sessions/{id}` или всех сессий пользователя `DELETE /api/admin/users/{id}/sessions`
- [ ] **Настройки сессий**: TTL access-токена, TTL refresh-токена, максимальное число параллельных сессий на пользователя; `PUT /api/admin/auth/session-policy`
- [ ] **Журнал входов**: лог попыток входа (успешных и неуспешных) с IP, User-Agent, результатом; `GET /api/admin/auth/login-log?userId=&from=&to=`

- [ ] **Версионный пакет**: к каждому экспортному архиву автоматически добавляется версия пакета и дата

#### FR-ADM-04.2: Импорт
- [ ] **Загрузка пакета**: загрузка ZIP-архива; предварительный анализ — список артефактов в пакете, конфликты (уже существующие с другой версией); `POST /api/admin/config/import/preview`
- [ ] **Стратегия при конфликте**: «Заменить», «Пропустить», «Создать с новым именем»; применяется глобально или на уровне каждого артефакта
- [ ] **Применение импорта**: `POST /api/admin/config/import/apply`; результат — лог применения (успех / ошибка для каждого артефакта)
- [ ] **Откат импорта**: сохранение снимка конфигурации до импорта; `POST /api/admin/config/import/{importId}/rollback`

#### FR-ADM-04.3: Публикация в Дизайнере (раздел «Публикация»)
- [ ] **Раздел «Публикация»** в Дизайнере: централизованная страница управления статусом публикации всех элементов конфигурации; вкладки — «Общие», «Черновики», «Процессы», «Объекты», «Документооборот» (при наличии ECM+), «Проекты» (при наличии Проекты+), «Пользовательские расширения»
- [ ] **Вкладка «Общие»**: сводная информация о статусе публикации — количество черновиков, опубликованных и удалённых элементов по каждому типу (оргструктура, процессы, объекты, перечисления, типы документов, типы проектов, расширения); кнопка публикации оргструктуры; счётчики элементов, требующих перезапуска сервера
- [ ] **Вкладка «Черновики»**: неопубликованные версии процессов, объекты, перечисления, типы документов, типы РКК, типы проектов; публикация одного элемента наведением курсора → кнопка «Опубликовать»; массовая публикация выбранных элементов с опциональным комментарием; удаление черновика процесса (безвозвратно); предупреждение при публикации без дефолтных свойств; при публикации объекта-наследника — запрос на одновременную публикацию родительского объекта
- [ ] **Вкладка «Процессы»**: опубликованные (активные) и удалённые версии процессов; перевод опубликованного в черновики; удаление; восстановление удалённых версий
- [ ] **Вкладка «Объекты»**: группы «Опубликованы» и «Требуется перезапуск» (объекты с изменениями, вступающими в силу только после перезапуска сервера)
- [ ] **Перезапуск сервера**: кнопка «Перезапустить сервер» — перезапуск сервера конфигурации Дизайнера; требуется при публикации новых объектов, перечислений и элементов документооборота; `POST /api/admin/server/restart`

### FR-ADM-05: Управление приложениями

#### FR-ADM-05.1: Создание и публикация приложений
- [ ] **Приложение**: именованный пакет конфигурации (процессы + объекты + формы + правила + интеграции + роли + меню), объединённый в единый установочный модуль; аналог Marketplace-приложения; `GET/POST/PUT/DELETE /api/admin/apps`
- [ ] **Сборка приложения**: выбор артефактов из существующей конфигурации → формирование манифеста; задание версии (SemVer), описания, иконки, зависимостей от других приложений; `POST /api/admin/apps/{id}/build`
- [ ] **Публикация**: публикация приложения во внутреннем реестре (на этом сервере) или экспорт файла для установки на другой инстанции; `POST /api/admin/apps/{id}/publish`
- [ ] **Установка приложения**: `POST /api/admin/apps/install` — загрузка файла или выбор из реестра; предпросмотр изменений; подтверждение; откат при ошибке
- [ ] **Обновление приложения**: сравнение текущей установленной версии с новой; diff артефактов; применение дельты; `POST /api/admin/apps/{id}/update`
- [ ] **Удаление приложения**: с опцией «Удалить все артефакты» или «Оставить данные»; `DELETE /api/admin/apps/{id}`
- [ ] **Журнал операций с приложениями**: история установок, обновлений, удалений с датой, пользователем, результатом; `GET /api/admin/apps/log`

### FR-ADM-06: Локализация и перевод интерфейса

#### FR-ADM-06.1: Управление языками
- [ ] **Список доступных языков**: администратор включает/отключает языки интерфейса для портала; поставляемые по умолчанию — русский и английский; `GET/PUT /api/admin/settings/locales`
- [ ] **Переключение языка пользователем**: выбор языка в профиле (FR-ORG-02.3); применяется немедленно — UI, уведомления, системные сообщения отображаются на выбранном языке; `PUT /api/users/me/preferences` (поле `locale`)
- [ ] **Язык по умолчанию**: администратор задаёт язык интерфейса для новых пользователей; `PUT /api/admin/settings/default-locale`

#### FR-ADM-06.2: Перевод интерфейса системы
- [ ] **Страница переводов**: таблица ключей локализации × языки; поиск по ключу и по тексту перевода; фильтрация по модулю/компоненту; `GET /api/admin/i18n/keys?lang=&module=&search=`
- [ ] **Редактирование перевода**: inline-редактирование ячейки; отметка «переведено / требует перевода»; сохранение изменённых значений: `PUT /api/admin/i18n/keys/{key}`
- [ ] **Экспорт переводов**: выгрузка выбранного языка в JSON (формат `{key: "value"}`) или XLSX; `GET /api/admin/i18n/export?lang=`
- [ ] **Импорт переводов**: загрузка JSON или XLSX с переводами; режим «заменить существующие» / «дополнить только пустые»; предпросмотр изменений перед применением; `POST /api/admin/i18n/import`
- [ ] **Сброс к системным значениям**: восстановление ключа к значению по умолчанию (из поставляемого пакета); `DELETE /api/admin/i18n/keys/{key}/override`

#### FR-ADM-06.3: Перевод конфигурационных объектов
- [ ] **Мультиязычные названия**: имена процессов, типов задач, типов документов, свойств объектов, ролей, статусов и других конфигурационных объектов хранятся в виде `{ru: "...", en: "..."}` и возвращаются на языке текущего пользователя
- [ ] В конструкторе форм, дизайнере процессов и других инструментах Low-code — поле ввода названия содержит вкладки по языкам; `PUT /api/admin/...` передаёт объект `{"label": {"ru": "...", "en": "..."}}`
- [ ] **Форматирование по локали**: даты, числа и валюты форматируются согласно локали пользователя (dayjs locale + Intl.NumberFormat); временны́е метки хранятся в UTC, отображаются в часовом поясе из профиля пользователя

## 17. Нефункциональные требования

- [ ] **Производительность**: отклик API ≤ 300 мс для 95% запросов при нагрузке 500 одновременных пользователей
- [ ] **Масштабируемость**: горизонтальное масштабирование backend (stateless-сервисы + внешний session store)
- [ ] **Безопасность**: HTTPS, хранение паролей (bcrypt/PBKDF2), JWT с коротким TTL + refresh, CSRF-защита, rate limiting
- [ ] **Аудит**: лог всех значимых действий пользователей (кто, что, когда, объект)
- [ ] **Интернационализация**: поддержка i18n (минимум: русский, английский)
- [ ] **Доступность**: SPA на React, поддержка современных браузеров (Chrome, Edge, Firefox)

## 18. Рекомендации по СУБД

### Основная СУБД (выбор предстоит)

| СУБД | Плюсы | Минусы | Рекомендуется для |
|------|-------|--------|-------------------|
| **PostgreSQL** | Мощный, JSON/JSONB, полнотекстовый поиск, open source, отличная поддержка в .NET (EF Core + Npgsql) | Требует настройки для больших нагрузок | **Рекомендуется как основной вариант** — универсален, покрывает 95% задач |
| **MS SQL Server** | Зрелый корпоративный продукт, первоклассная интеграция с .NET/EF Core, отличные инструменты мониторинга | Платный (Enterprise), привязка к экосистеме MS | Если заказчик уже в MS-стеке или есть лицензии |
| **Oracle Database** | Корпоративные возможности, высокая надёжность | Дорогой, сложный в управлении | Крупные госкорпорации с существующей лицензией Oracle |

### Дополнительные хранилища (рекомендуется независимо от основной СУБД)

- [ ] **Redis** — кеш сессий, очереди задач (Redis Streams / MassTransit), distributed lock
- [ ] **Elasticsearch / OpenSearch** — полнотекстовый поиск по документам и объектам (EQL-поиск)
- [ ] **MinIO / Azure Blob / S3** — хранение файлов и вложений (вне БД)

> **Итоговая рекомендация**: PostgreSQL + Redis + Elasticsearch + S3-совместимое хранилище.  
> Полностью open source, хорошо интегрируется с .NET Core, покрывает все сценарии системы.
